# 微内核 (Microkernel)

---
微内核架构模式适用于必须能够适应不断变化的系统需求的软件系统。
它将最小功能核心与扩展功能和客户特定部分分离开来。
微内核还充当一个插槽 (socket)，用于插接这些扩展组件并协调它们之间的协作。

---

## 示例 (Example)

假设我们打算为桌面计算机开发一款名为 Hydra 的全新操作系统。
我们的开发团队为此制定了一系列设计目标。
其中一项需求是，这款创新操作系统必须能轻松移植到相关硬件平台，并且能够便捷地适配未来的发展。
它还必须能够运行为其他主流操作系统编写的应用程序，例如 NeXTSTEP、微软 Windows 以及 UNIX System V。
用户应能在启动应用程序前，通过弹出菜单选择想要使用的操作系统。
Hydra 会在其主窗口中显示当前正在运行的所有应用程序。

<img src="../../img/fig-2511.png" width="90%"/></br>

为模拟所有这些操作系统，Hydra 将集成专用服务器，这些服务器会实现 Hydra 功能核心的特定视图。
视图指的是建立在核心功能之上的一层抽象接口。
由一个服务器进程模拟微软 Windows 就是这类视图的一个实例。
由于多媒体、笔式计算以及万维网等多项新技术的重要性很可能不断提升，Hydra 系统在设计上应便于这些技术的集成，同时也支持整体功能的适配、演进与增强。

## 上下文 (Context)
开发使用构建于同一核心功能之上的相似编程接口的若干应用程序。

## 问题 (Problem)

为某个需要应对广泛同类标准与技术的应用领域开发软件，是一项艰巨的任务。
典型例子包括操作系统、图形用户界面等应用平台 <sup>[20](#20)</sup> 。
这类系统通常生命周期较长，有时可达十年甚至更久。
在如此长的周期内，新技术会不断涌现，旧技术则会发生变化。
因此在设计此类系统时，需要特别考虑以下影响因素：

- 应用平台必须能够应对硬件与软件的持续演进。

- 应用平台应具备可移植性、可扩展性和适应性，以便轻松集成新兴技术。

这类应用平台的成功还取决于它们能否运行为现有标准编写的应用程序。
为支持广泛的应用，需要为底层应用平台的功能提供多种不同视图。
换句话说，像操作系统或数据库这样的应用平台，还应当能够模拟属于同一应用领域的其他应用平台。

↘️例如，Hydra 操作系统旨在运行为微软 Windows、OS/2 Warp 等主流操作系统原生开发的应用程序。

这便产生了以下影响因素：

- 你所在领域的应用程序需要支持不同但相似的应用平台。

- 应用程序可划分为多个类别，它们以不同方式使用同一功能核心，这就要求底层应用平台能够模拟现有标准。

提供某一领域功能核心的应用平台，对其使用者而言是一种独占性资源。
为避免性能问题并保证可扩展性，你的解决方案必须考虑另一项约束：

- 应用平台的功能核心应拆分为一个内存占用最小化的组件，以及尽可能少消耗处理能力的服务。

## 解决方案 (Solution)
将应用平台的基础服务封装在一个微内核组件中。
微内核包含的功能可让运行在独立进程中的其他组件彼此通信。
它同时负责维护文件、进程等系统级资源。
此外，微内核还提供接口，使其他组件能够访问其功能。

若某些核心功能在微内核中实现，会导致其体积或复杂度不必要地增加，则应将其分离到 *内部服务器 (internal servers)* 中。

*外部服务器* 基于底层微内核实现各自的视图。
为构建该视图，它们会通过微内核提供的接口，使用相应机制。
每个外部服务器都是独立进程，其本身即代表一个应用平台。
因此，微内核系统可被视为一个集成了其他应用平台的应用平台。

客户端通过微内核提供的通信机制与外部服务器进行交互。

## 结构 (Structure)
微内核模式定义了五种参与组件：

- 内部服务器
- 外部服务器
- 适配器
- 客户端
- 微内核

*微内核* 是该模式的核心组件，实现通信机制、资源管理等核心服务。
其他组件基于这些基础服务的全部或部分构建。
他们通过使用一个或多个接口间接地完成这一点，这些接口组成了微内核所公开的功能。

许多与特定系统相关的依赖关系都被封装在微内核内部。
例如，大多数与硬件相关的部分对其他组件而言是不可见的。
微内核的客户端只能看到底层应用领域和平台特性的特定视图。

微内核还负责维护进程、文件等系统资源，并控制和协调对这些资源的访问。

总之，微内核实现的是 *原子服务 (atomic services)* ，我们将这类服务称为 *机制 (mechanisms)* 。
这些机制构成了基础，更复杂的功能，称为 *策略 (policies)* ，便构建于此基础之上。

<img src="../../img/fig-2512.png" width="60%"/></br>

↘️在 Hydra 操作系统中，我们希望支持 UNIX System V、OS/2 Warp 以及其他多种操作系统。
在实现 Hydra 的进程模型时，我们遇到了一个问题。
在 UNIX 系统中，创建新子进程这类系统调用是通过克隆现有进程、复制整个地址空间来实现的。
而 OS/2 Warp 处理进程创建的方式完全不同，它不会复制父进程的地址空间。
换言之，OS/2 Warp 与 UNIX 提供了不同的进程管理策略。
因此，Hydra 在设计时提供了各类基础服务，既包含创建进程的机制，也包含克隆现有进程地址空间的机制。
通过对这些机制的多种组合方式，便可分别实现 UNIX System V 的进程模型与 OS/2 Warp 的进程模型。

*内部服务器* ，也称为 *子系统 (subsystem)* ，扩展了微内核提供的功能。
它是一个独立的组件，提供额外的功能。
微内核通过服务请求调用内部服务器的功能。
因此，内部服务器可以封装对底层硬件或软件系统的部分依赖关系。
例如，支持特定显卡的设备驱动程序就非常适合作为内部服务器。

<img src="../../img/fig-2513.png" width="60%"/></br>

设计目标之一应是尽可能缩小微内核体积，以降低内存需求。
另一个目标是提供执行高效的机制，以减少服务执行时间。
因此，额外的、更复杂的服务由内部服务器实现，微内核仅在需要时才激活或加载这些服务器。
你可以将内部服务器视为微内核的扩展。
请注意，内部服务器仅允许微内核组件访问。

外部服务器，也称为 *个性化 (personality)* ，是一种利用微内核，为底层应用领域实现其专属视图的组件。
如前所述，视图是指在微内核提供的原子服务之上构建的一层抽象。
不同的外部服务器为特定应用领域实现不同的策略。

外部服务器与微内核本身一样，通过导出接口来暴露其功能。
每个此类外部服务器都运行在独立的进程中。
它借助微内核提供的通信机制接收来自客户端应用程序的服务请求，对这些请求进行解析，执行相应的服务，并将结果返回给客户端。
服务的实现依赖于微内核的各种机制，因此外部服务器需要访问微内核的编程接口。

<img src="../../img/fig-2514.png" width="60%"/></br>

↘️在 Hydra 系统中，我们计划实现一个 OS/2 Warp 外部服务器和一个 UNIX System V 外部服务器。
这两类服务器均借助底层微内核的各类机制，来实现完整的 OS/2 Warp 与 UNIX System V 系统调用。

客户端是仅与唯一一个外部服务器关联的应用程序，它只访问该外部服务器提供的编程接口。

如果客户端需要直接访问其所属外部服务器的接口，就会出现问题。
每个客户端都必须使用现有的通信机制与外部服务器进行交互。
因此，客户端代码中必须硬编码与外部服务器的每一次通信。
然而，客户端与服务器之间如此紧密的耦合会带来诸多弊端：

- 此类系统对可变更性的支持不佳。
- 如果外部服务器模拟现有应用平台，那么为这些平台开发的客户端应用程序不经修改将无法运行。

因此，我们在客户端与其外部服务器之间引入接口，以避免客户端产生直接依赖。
*适配器 (Adapters)* ，也称为 *仿真器 (emulators)* 代表客户端与外部服务器之间的这类接口，使客户端能够以可移植的方式访问其外部服务器的服务。
适配器属于客户端地址空间的一部分。
如果外部服务器实现了某个现有应用平台，对应的适配器便会模拟该平台的编程接口。
如此一来，为被仿真平台编写的客户端无需修改即可编译并运行。
适配器同时也能屏蔽微内核的具体实现细节，对客户端起到保护作用。

每当客户端向外部服务器请求服务时，适配器负责将该调用转发至对应的服务器。
为此，适配器会使用微内核提供的通信服务。

<img src="../../img/fig-2515.png" width="100%"/></br>

↘️由于适配器的封装作用，与 OS/2 Warp 外部服务器关联的 Hydra 客户端无法知晓自身是运行在原生 OS/2 Warp 系统上，还是运行在提供了 OS/2 Warp 外部服务器的微内核系统上。
客户端只需像往常一样使用 OS/2 系统调用即可，底层的实际运行逻辑均由适配器进行屏蔽。

下面这张 OMT 图展示了微内核系统的静态结构。
其核心组件，微内核与外部服务器、内部服务器以及适配器相互协作。
每个客户端都与一个适配器相关联，该适配器充当客户端与其对应外部服务器之间的桥梁。
内部服务器仅允许微内核组件访问。

## 动态 (Dynamics)
微内核系统的动态行为取决于其为进程间通信提供的功能。
在以下场景中，我们假定支持远程过程调用。
第一种场景还假设外部服务器不访问微内核接口——后一种情况将在第二种场景中说明。

### 场景一
演示客户端调用其所属外部服务器服务时的行为：

- 在客户端控制流的某个时刻，客户端通过调用适配器向外部服务器请求服务。
- 适配器构造一个请求，并向微内核申请与该外部服务器建立通信链路。
- 微内核确定外部服务器的物理地址，并将其返回给适配器。
- 获取该信息后，适配器与外部服务器建立直接通信链路。
- 适配器通过远程过程调用将请求发送至外部服务器。
- 外部服务器接收请求，对消息进行解包，并将任务委派给自身的某个方法。
完成所请求的服务后，外部服务器将所有结果与状态信息返回给适配器。
- 适配器将结果返回给客户端，客户端继而继续执行其控制流程。

<img src="../../img/fig-2517.png" width="90%"/></br>

### 场景二
下图展示了在微内核架构中，当外部服务器请求由内部服务器提供服务时的系统行为。
在该场景中，我们假设内部服务器被实现为一个独立的进程。
它也可以被实现为一个与微内核动态链接的共享库。

- 外部服务器向微内核发送服务请求。
- 调用微内核编程接口中的一个程序来处理该服务请求。
在方法执行期间，微内核向内部服务器发送请求。
- 内部服务器接收到请求后，执行所请求的服务，并将所有结果返回给微内核。
- 微内核将结果返回给外部服务器。
- 最后，外部服务器接收结果并继续执行其控制流程。

## 实现 (Implementation)
实现微内核系统，请按以下步骤操作：

### 1. 分析应用领域
若你已明确外部服务器需提供的策略，或对即将开发的外部服务器有详尽了解，直接执行 [步骤 2](#2-分析外部服务器) 。
若无相关信息，则先开展领域分析，梳理搭建外部服务器所需的核心功能，再执行 [步骤 3](#3-对服务进行分类) 。
*「是不是步骤顺序写反了」*

### 2 分析外部服务器
分析外部服务器将要提供的策略。
随后你应能明确自身应用领域内所需的功能。

↘️针对 Hydra 系统，我们已明确需实现的外部服务器：UNIX System V、OS/2 Warp、微软 Windows 以及 NeXTSTEP 系统。
因此我们会分析其编程接口，确定它们所能提供的服务。
本次需求分析最终会整理出实现桌面操作系统所需的服务清单与服务类别。

### 3. 对服务进行分类
尽可能将所有功能归入语义独立的类别。

构建与应用领域无直接关联、但实现系统基础架构所必需的操作类别。
部分此类操作可考虑迁移至内部服务器。

↘️例如，在Hydra系统中，由操作系统领域预先定义的核心类别包括：内存管理、进程管理、底层输入输出服务与通信服务。

以下类别与应用领域的核心概念无直接关联：页处理 (page-handler) 进程、文件系统、软硬件驱动程序。
我们在实现 Hydra 系统时需要用到这些类别，但它们可迁移至内部服务器。

### 4. 拆分类别
将类别划分为应纳入微内核的服务，以及应以内部服务器形式提供的服务。
需制定划分标准。
例如，实时性 (time-critical)、高频调用或依赖硬件的操作，最好在微内核组件中实现。

↘️在 Hydra 系统中，微内核提供进程管理、内存管理、通信及底层输入输出等服务。
这类功能实时性要求高、被所有其他组件调用，且封装了系统依赖，因此应归入微内核。
缺页处理程序、驱动程序、文件系统等其余附加服务，均由内部服务器实现。

### 5. 寻找一套统一且完备的操作与抽象
为 [步骤 1](#1-分析应用领域) 中划分出的每一类，定义一套统一且完备的操作与抽象机制。
需牢记：微内核只提供底层机制，不定义策略。
外部服务器实现的所有策略，都必须依托微内核通过接口提供的服务来完成。

↘️UNIX、微软 Windows 等操作系统，在进程、线程创建这类操作上实现方式各不相同，Hydra 需要同时兼容二者。
因此系统会提供一套完整的进程与线程基础（管理）机制，例如包含以下服务：

- 创建与终止进程及线程。
- 暂停并重启它们。
- 读写进程地址空间。
- 捕获并处理异常。
- 管理进程或线程之间的关联关系。
- 实现线程的同步与协作调度。

### 6. 制定请求发送与获取的交互策略
明确微内核应为组件间通信提供的底层支撑能力。
可选用多种通信方案，例如异步通信或同步通信。
通信组件之间可采用一对一、多对一或多对多的交互关系。
集成的通信策略需贴合应用领域的实际需求。
多数场景下，消息传递、共享内存这类底层通信机制均可作为基础，在此之上可搭建更复杂的通信架构。
如需了解通信机制的落地实现，可参考转发器-接收器 (Forwarder-Receiver) （[307](#todo)）、客户端-调度器-服务端 (Client-Dispatcher-Server) （[323](#todo)）等设计模式。

↘️Hydra 提供两项基础通信机制：

- 同步远程过程调用（RPC）。RPC 可让客户端调用远端服务器服务，效果如同调用本地过程。
支撑 RPC 所需的底层机制，例如请求的打包解包、跨进程消息传输，对调用方与被调用服务端均完全透明。

- 异步信箱。信箱 (mailbox) 属于一类消息缓冲区。
一组组件可从中读取消息，另一组组件拥有写入权限；
单个组件也可同时具备读写权限。

### 7. 对微内核组件进行架构设计
如有可能，采用分层模式（[31](#todo)）设计微内核，将微内核中与系统相关的部分和与系统无关的部分分离开来。
把微内核向其他组件开放的服务置于最上层，通过下层隐藏上层所需依赖的系统相关内容。

↘️在我们的 Hydra 项目中，决定采用面向对象技术实现微内核：

- 最底层由底层对象构成，负责向微内核其他部分隐藏总线架构等硬件专属细节。

- 中间层通过系统对象提供核心基础服务，包括内存管理对象、进程管理对象等。

- 最顶层包含微内核对外公开的全部功能，是所有进程调用微内核服务的统一入口。

### 8. 定义编程接口 
要定义微内核的编程接口，需明确这些接口的外部访问方式。
显然必须考虑微内核是独立进程实现，还是作为可被其他组件物理共享的模块实现。
若为后者，可通过常规方法调用，直接调用微内核的接口方法。

若将微内核实现为独立进程，则需借助现有通信机制，把组件请求传递至微内核。
此种情况下需注意：内核属于独占资源，因此可能形成性能瓶颈。
为提升整体性能，可在微内核内部设置多条线程监听接入请求，并由这些线程（或其他线程）执行对应服务。
设计此类多线程系统时，务必保障内核内部数据的一致性。

↘️由于 Hydra 本身是一套操作系统，其微内核组件会内嵌至每个用户进程中。
因此各类服务可通过常规系统调用进行访问。
这些功能会按逻辑整合为应用程序编程接口（API），支持文件系统操作、进程管理等相关能力。

调用 Hydra 系统调用时会触发系统陷入 (system trap)。
软件异常由微内核内专用的陷入处理程序 (trap handler routine) 接管。
该处理程序会判定触发此次系统陷入的中断类型，并将任务委派给对应的内部服务对象执行。
服务执行完毕后，调度对象会选定下一个待执行的就绪线程，并为其分配处理器资源。

### 9. 微内核负责管理所有的系统资源
微内核负责管理内存块、设备，或是设备上下文（图形用户界面实现里输出区域的操作句柄）这类系统资源。
微内核会维护所有资源的相关信息，并以统一、有序的方式管控资源访问。
组件如需调用资源，只能通过唯一标识符（句柄）访问，禁止直接操作资源本体。
微内核的任务是创建这些句柄并提供句柄与资源之间的映射。
该映射常通过哈希表实现。
资源管理不止建立映射，微内核还需配套实现资源的共享、加锁、分配与释放策略。

↘️在 Hydra 系统中，句柄指代资源类实例化后的对象。
每个这类对象都提供统一接口，用于管控特定资源的访问权限。

我们的资源对象遵循 Windows NT 的设计思路，对外暴露如下接口：

```cpp
class Resource {
    String name;                  // Name of object
    void OpenHandle();            // open handle to object
    Handle IterateHandles();      // iterate over handles
    Body pointerToObject;         // pointer to real object
    ...                           // (much, much more ...)
}
```

### 10. 设计和实现内部服务器
将内部服务器设计并实现为独立进程或共享库。
本步骤可与 [第 7 步](#7-对微内核组件进行架构设计) 至 [第 9 步](#9-微内核负责管理所有的系统资源) 并行执行，因为部分微内核服务需要调用内部服务器。
建议区分主动 (active) 服务器与被动 (passive) 服务器：

- 主动服务器以进程形式实现
- 被动服务器以共享库形式实现

被动服务器始终通过直接调用其接口方法来触发，而主动服务器则需采用不同处理方式。
主动服务器进程会在事件循环中监听接入请求；
一旦通过现有通信机制收到请求，便会解析请求内容，并代为调用方执行对应服务。
需注意：内部服务器仅允许微内核进行访问，严禁其他组件调用其服务。

↘️在 Hydra 系统中，我们将设备驱动程序、身份验证服务器、缺页异常处理器等组件，均以内部服务器的形式实现。

显卡驱动程序被开发为共享库，因其仅为客户端提供代理服务。
与之相对，缺页异常处理器采用独立进程实现，这类进程必须常驻主内存，不可被置换到外部存储设备。

### 11. 实现外部服务器
外部服务器包含的所有策略，均依托微内核编程接口所提供的服务搭建。
外部服务器接收请求、解析请求、执行对应服务，并将结果回传给调用方。
外部服务器执行服务时，可调用微内核中的操作接口。

因此每台外部服务器都以独立进程实现，对外提供专属服务接口。
外部服务器的内部架构，由其承载的策略决定。

明确外部服务器如何将请求分发至其内部程序。
例如，可集成一个分发器组件，该组件运行主事件循环并监听接入请求。
当请求抵达时，分发器对其解包、解析，并通过回调机制调用对应程序。
若将外部服务器设计为应用框架，该方式尤为适用。
此事件驱动方案可参考反应器模式（Reactor）[Sch94](../../ref.md#sch94) 。

↘️我们需为 Hydra 开发以下外部服务器：

- 完整实现微软 Win32 与 Win16 应用程序接口，支持运行 Windows NT、Windows 3.11 及 Windows 95 应用程序；
- 完整适配 IBM OS/2 Warp 2.0 的全部功能；
- 实现 Openstep 接口规范；
- 兼容 X/Open 定义的全部 UNIX System V 相关接口。

### 12. 实现适配器
适配器的主要任务是向客户端提供可转发至外部服务器的操作。
每当客户端调用外部服务器的功能时，适配器会将所有相关信息封装为请求，并转发给对应的外部服务器。
随后适配器等待服务器响应，最终借助组件间通信机制，将执行控制权返还客户端。

你既可以将适配器设计为编译时与客户端静态链接的传统库，也可设计为按需动态链接至客户端的共享库。
适配器可视为仅代理单个外部服务器的代理组件，因此可采用代理模式 (Proxy)（[263](#todo)）实现适配器。
可对适配器做性能优化：部分 API 操作由适配器本地执行，无需转发请求至外部服务器；
也可将多条客户端请求缓存后再批量转发，常用请求的返回结果亦可存入缓存。
关于缓存的优劣与风险，可参考代理模式 (Proxy)（[263](#todo)）相关说明。

你必须决定是否一个适配器应该可供所有客户端使用，还是每个客户端都有自己的适配器。
前者可降低内存资源竞争，后者则能优化响应速度。

↘️倘若我们为 Hydra 系统设计微软 Windows 外部服务器，所有对接该服务器的客户端应用程序都会调用 Win16 或 Win32 应用程序接口。
在原生 Windows 3.11 系统中，全部接口都以一组共享库的形式提供。
但在 Hydra 系统里，Windows 客户端与 Windows 服务器是相互独立的进程。
由于我们要让 Windows 应用程序无需修改就能在 Hydra 上运行，就必须提供完全一致的运行环境。
因此我们搭建适配器，将其作为 Windows 客户端与 Windows 服务器之间的中转桥梁。
当客户端调用 Win16 或 Win32 接口函数时，该调用会由适配器接管，再由适配器将请求转发至 Windows 外部服务器。
如此一来，原生 Windows 客户端程序便可直接编译并在 Hydra 系统中运行。

### 13. 开发客户端应用
开发客户端应用程序，或直接选用现有可直接运行于微内核系统的应用。
为特定外部服务器新建客户端时，其架构仅受该外部服务器设定的约束限制。
也就是说，客户端依赖其对接的外部服务器所实现的策略。

↘️在 Hydra 系统中，可借助微软 Windows 适配器调用微软 Windows 外部服务器的服务，以此开发微软 Windows 应用程序。

## 示例解析 (Example Resolved)

Hydra 系统开发完成后不久，我们需要集成一个可模拟苹果 MacOS 操作系统的外部服务器。
若要在 Hydra 之上实现 MacOS 模拟功能，需完成以下工作：

基于 Hydra 微内核 *搭建外部服务器* ，完整实现 MacOS 提供的所有编程接口，同时适配 Macintosh 用户界面的各项策略。
该 MacOS 服务器会通过主循环监听接入请求，所有请求将存入专属分配给此 MacOS 服务器的消息端口。
服务器从消息端口提取请求、解析内容，并将其分发至内部程序执行。
这些内部程序会模拟 MacOS 环境特有的运行策略。

*提供适配器* 提供一个以库形式实现、可动态链接至客户端的适配器。
对于原生 MacOS 系统中的所有应用程序接口函数，该库都必须提供语法完全一致的子程序 (procedure) 。
每个子程序均负责将请求类型、参数以及收发方标识封装为消息，
随后调用微内核的 sendMessage ，由微内核将该消息存入 MacOS 服务器专属消息端口。

*实现内部服务器* 实现 MacOS 所需的内部服务器。
例如，某个内部服务器提供 AppleTalk 网络协议。
微内核需进行修改，以代表 MacOS 外部服务调用这些新增的内部服务器。

## 变体 (Variants)
*具备间接客户端-服务器连接的微内核系统* 。
在此变体中，客户端若需向外部服务器发送请求或消息，会先向微内核申请通信通道。
待所需通信链路建立后，客户端与服务器将以微内核作为消息主干 (backbone)，实现间接交互。
采用该架构时，所有请求都会经过微内核转发。
例如，当安全规范要求系统管控所有组件间通信时，便可使用此种设计。

*分布式微内核系统* 。
在该变体中，微内核还可充当消息主干 (backbone)，负责向远端设备收发消息。
分布式环境里的每台机器，都运行独立的微内核实例。
从用户视角来看，整套系统表现为单一微内核系统，分布式细节对用户完全透明。
分布式微内核系统支持将服务端与客户端跨机器、跨微处理器网络进行部署。
为此，分布式架构下的所有微内核，都必须包括用于节点间互通的扩展服务。
