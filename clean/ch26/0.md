在每个系统中，至少有一个组件负责创建、协调并监督其他组件。我称这个组件为 Main。

## 终极细节

Main 组件是终极细节 —— 最低层次的策略。
它是系统的初始入口点。
除了操作系统之外，没有任何东西依赖于它。
它的工作是创建所有的工厂（Factories）、策略（Strategies）以及其他全局设施，然后将控制权移交给系统中高层次的抽象部分。

正是在这个 Main 组件中，依赖项应当由依赖注入框架来注入。
一旦它们被注入到 Main 中，Main 就应该正常地分发这些依赖，而不再使用该框架。

可以把 Main 看作是所有脏组件中最脏的那个。

注意它是如何加载所有那些我们不希望代码主体部分知道的字符串的。

关键在于，Main 是一个位于整洁架构最外圈的低层模块。
它为高层系统加载一切所需资源，然后将控制权移交给它。

## 结论

<ins>可以把 Main 看作是应用程序的一个插件 —— 一个负责设置初始条件和配置、收集所有外部资源，然后将控制权移交给应用程序高层策略的插件。
由于它是一个插件，因此可以存在多个 Main 组件，每个对应应用程序的一种配置</ins>。

例如，可以为开发环境准备一个 Main 插件，为测试环境准备另一个，为生产环境准备再一个。
也可以为每个部署的国家、每个司法管辖区或每个客户分别准备一个 Main 插件。

当你把 Main 看作是一个插件组件，处于架构边界之后时，配置问题就变得容易解决得多了。
