# 15 什么是架构？

---
<center><img src="../imgs/fig15-0.png" width="40%"/></center><br/>

“架构” 这个词让人联想到权力和神秘感。
它使我们想到重大的决策和深厚的技术能力。
软件架构处于技术成就的顶峰。
当我们想到软件架构师时，我们会想到一个拥有权力、受人尊敬的人。
哪个有抱负的年轻软件开发者没有梦想过有一天成为一名软件架构师？

但是什么是软件架构？
软件架构师做什么？
他或她什么时候做这些工作？

首先，软件架构师是一名程序员，并且会继续是一名程序员。
永远不要相信那种说法，即软件架构师从代码中抽身出来去关注更高层次的问题。
他们没有！软件架构师是最优秀的程序员，他们继续承担编程任务，同时引导团队其他成员朝着最大化生产力的设计方向前进。
软件架构师可能不会像其他程序员那样写那么多代码，但他们仍会继续参与编程任务。
他们这样做是因为，如果他们不去亲身体验正在为其他程序员制造的问题，就无法恰当地完成自己的工作。

<ins>软件系统的架构是由构建它的人赋予该系统的形状。
该形状的形式体现为：将系统划分为组件、这些组件的排列方式，以及这些组件之间相互通信的方式</ins>。

<ins>该形状的目的是促进其中所包含的软件系统的开发、部署、运行和维护</ins>。

> <ins>这种促进背后的策略是：尽可能长时间地保留尽可能多的选项</ins>。

也许这句话让你感到惊讶。
也许你认为软件架构的目标是让系统正常工作。
当然，我们希望系统正常工作，并且系统的架构必须将其作为最高优先级之一来支持。

然而，系统的架构与系统是否能够正常工作关系甚微。
外面有很多架构糟糕的系统，它们运行得也很好。
它们的问题不在于运行，而在于其部署、维护和持续开发。

<ins>这并不是说架构在支持系统正确行为方面不起作用。
它当然起作用，而且这个作用是关键的。
但这个作用是被动且表面性的，而不是主动或本质性的。
系统的架构几乎没有留下什么行为选项（即使有也很少）</ins>。

<ins>架构的主要目的是支持系统的生命周期。
良好的架构使系统易于理解、易于开发、易于维护、易于部署。
其最终目标是最小化系统的生命周期成本，并最大化程序员的生产力</ins>。

## 开发

<ins>一个难以开发的软件系统不太可能拥有长久而健康的生命周期。
因此，系统的架构应该使该系统对于开发它的团队来说是易于开发的</ins>。

不同的团队结构意味着不同的架构决策。
一方面，一个由五名开发人员组成的小团队，可以非常有效地协同开发一个没有良好定义组件或接口的单体系统。
事实上，在开发初期，这样的团队可能会觉得架构的约束是一种障碍。
这很可能就是为什么那么多系统缺乏良好架构的原因：它们在开始时就没有架构，因为团队很小，不想要上层建筑的阻碍。

另一方面，一个由五个不同团队（每个团队有七名开发人员）开发的系统，除非将系统划分为具有可靠稳定接口的、定义良好的组件，否则将无法取得进展。
如果不考虑其他因素，该系统的架构很可能会演变成五个组件 —— 每个团队一个。

这种 “每个团队一个组件” 的架构不太可能是对系统部署、运行和维护的最佳架构。
然而，如果仅仅由开发进度驱动，一群团队自然会趋向于这种架构。

## 部署

为了有效运作，一个软件系统必须是可部署的。
部署成本越高，系统的有用性 (useful) 就越低。
<ins>因此，软件架构的一个目标应该是：使系统能够通过 *单一操作 (single action)* 轻松部署</ins>。

不幸的是，在初始开发过程中很少考虑部署策略。
这导致了架构可能使系统易于开发，但却使其非常难以部署。

例如，在系统的早期开发中，开发人员可能决定使用 “微服务架构”。
他们可能会发现这种方法使系统非常易于开发，因为组件边界非常清晰，接口相对稳定。
然而，当需要部署系统时，他们可能会发现微服务的数量变得令人生畏；
配置它们之间的连接以及它们的启动时机，也可能成为巨大的错误来源。

如果架构师早期就考虑部署问题，他们可能会决定使用更少的服务、服务和进程内组件的混合，以及更集成的连接管理方式。

## 运行

<ins>架构对系统运行的影响通常不像对开发、部署和维护的影响那么显著</ins>。
几乎任何运行上的困难都可以通过向系统投入更多硬件来解决，而无需大幅改变软件架构。

事实上，我们一次又一次地看到这种情况发生。
架构效率低下的软件系统，往往只需增加更多的存储和更多的服务器就能有效运行。
硬件便宜、人力昂贵这一事实意味着，阻碍运行的架构不像阻碍开发、部署和维护的架构那样成本高昂。

这并不是说，与系统运行良好匹配的架构是不可取的。
它是可取的！
只是成本的天平更倾向于开发、部署和维护。

<ins>话虽如此，架构在系统运行中还有另一个作用：良好的软件架构能够传达系统的运行需求</ins>。

也许更好的说法是：系统的架构使系统的运行对于开发人员来说一目了然。
架构应该揭示运行。
<ins>系统的架构应将系统的用例、特性和所需的行为提升为一等实体，使其成为开发人员可见的地标</ins>。
这简化了对系统的理解，从而极大地帮助开发和维护。

## 维护

在软件系统的所有方面中，维护的成本是最高的。
无休止的新功能需求、不可避免的缺陷和修复，消耗了大量的人力资源。

维护的主要成本在于 *探洞 (spelunking)* 和风险。
探洞是指在现有软件中挖掘，试图确定添加新功能或修复缺陷的最佳位置和最佳策略的成本。
在进行此类更改时，总是存在引入意外缺陷的可能性，从而增加了风险成本。

<ins>经过深思熟虑的架构可以极大地降低这些成本。
通过将系统划分为组件，并通过稳定的接口隔离这些组件，可以为未来的功能照亮路径，并大大降低意外破坏的风险</ins>。

## 保留选项

正如我们在前面章节中所描述的，软件有两种价值：行为价值和结构价值。
其中，结构价值是更大的，因为正是这种价值使软件变得 “软”。

软件之所以被发明，是因为我们需要一种能够快速、轻松地改变机器行为的方式。
但这种灵活性在很大程度上取决于系统的形状、其组件的排列方式以及这些组件相互连接的方式。

<ins>保持软件 “软” 的方法是：尽可能长时间地保留尽可能多的选项</ins>。
我们需要保留的选项是什么？
<ins>*它们是那些无关紧要的细节*</ins>。

<ins>所有软件系统都可以分解为两个主要元素：策略 (policy) 和细节。
策略元素体现了所有的业务规则和流程。
策略是系统真正价值所在</ins>。

<ins>细节是那些使人类、其他系统和程序员能够与策略通信所必需的东西，但丝毫不会影响策略的行为</ins>。
它们包括 IO 设备、数据库、Web 系统、服务器、框架、通信协议等等。

<ins>架构师的目标是为系统创建一个形状，将策略视为系统中最核心的元素，同时使细节与该策略无关。
这使得关于这些细节的决策可以被推迟和延后</ins>。

例如：

- 在开发早期不必选择数据库系统，因为高层策略不应该关心使用哪种类型的数据库。
实际上，如果架构师足够谨慎，高层策略甚至不会关心数据库是关系型的、分布式的、层次型的，还是纯平面文件。

- 在开发早期不必选择 Web 服务器，因为高层策略不应该知道它是通过 Web 来交付的。
如果高层策略对 HTML、AJAX、JSP、JSF 或其他任何 Web 开发的大杂烩一无所知，那么你就直到项目后期才需要决定使用哪种 Web 系统。
实际上，你甚至不必决定系统是否通过 Web 交付。

- 在开发早期不必采用 REST，因为高层策略应该对外部世界的接口一无所知。
同样，也没有必要采用微服务框架或 SOA 框架。
重申一下，高层策略不应该关心这些事情。

- 在开发早期没有必要采用依赖注入框架，因为高层策略不应该关心依赖关系是如何解决的。

我想你已经明白了要点。
<ins>如果你能够在不承诺其周边细节的情况下，开发高层策略，那么你就能将这些细节的决策推迟很长时间。
你等待做出这些决策的时间越长， *你就能掌握更多的信息来做出正确的决策* </ins>。

这也为你进行不同的实验留出了选项。
如果你有一部分高层策略能够正常工作，并且它与数据库无关，那么你可以尝试将其连接到几个不同的数据库，以检查适用性和性能。
对于 Web 系统、Web 框架、甚至 Web 本身也是如此。

<ins>你保留选项的时间越长，就能进行越多的实验，尝试越多的可能性，并且在那些决策再也无法推迟的时刻到来时，你将拥有越多的信息作为依据</ins>。

如果这些决策已经被其他人做出了呢？
如果你的公司已经承诺使用某个特定的数据库、某个特定的 Web 服务器或某个特定的框架呢？
<ins>*优秀的架构师会假装这个决策尚未做出* ，并塑造系统，使这些决策仍然可以尽可能长时间地被推迟或更改</ins>。

> 优秀的架构师最大化未做出决策的数量。

## 设备独立性

作为这种思维方式的一个例子，让我们回到 20 世纪 60 年代。
那时计算机正值青少年时期，大多数程序员是数学家或其他学科的工程师（其中三分之一或更多是女性）。

在那个年代，我们犯了很多错误。
当然，我们当时并不知道那些是错误。
我们怎么可能知道呢？

其中一个错误就是将我们的代码直接与 IO 设备绑定。
如果我们需要在打印机上打印一些东西，我们就编写使用 IO 指令的代码来控制打印机。
我们的代码是 *设备依赖* 的。

例如，当我编写 PDP-8 程序在电传打字机上打印时，我使用了一组如下所示的机器指令：

```
PRTCHR, 0
        TSF
        JMP .-1
        TLS
        JMP I PRTCHR
```

`PRTCHR` 是一个子程序，用于在电传打字机上打印一个字符。
开头的零用作返回地址的存储（不要问为什么）。
如果电传打字机准备好打印一个字符，`TSF` 指令会跳过下一条指令。
如果电传打字机忙，`TSF` 就会直接执行到下一条 `JMP .-1` 指令，该指令只是跳回 `TSF` 指令。
如果电传打字机已准备好，那么 `TSF` 会跳过到 `TLS` 指令，该指令将 `A` 寄存器中的字符发送到电传打字机。
然后 `JMP I PRTCHR` 指令返回到调用者。

起初，这个策略运行良好。
如果我们需要从读卡器读取卡片，我们就使用直接与读卡器对话的代码。
如果我们需要打孔卡片，我们就编写直接操作打孔机的代码。
程序运行得很完美。
我们怎么可能知道这是个错误呢？

但是大批量的穿孔卡片难以管理。
它们可能会丢失、损坏、卷曲、洗乱或掉落。
个别的卡片可能会丢失，也可能混入多余的卡片。
因此，数据完整性成了一个重要问题。

磁带是解决方案。
我们可以将卡片映像转移到磁带上。
如果你不慎掉了一盘磁带，记录不会被打乱。
你不可能仅仅因为拿取一盘磁带，就意外丢失一条记录或插入一条空白记录。
磁带安全得多。
它的读写速度也更快，而且制作备份副本非常容易。

不幸的是，我们所有的软件都是为操作读卡器和打孔机而编写的。
这些程序必须重写才能使用磁带。
那是一项艰巨的工作。

到了 20 世纪 60 年代末，我们吸取了教训 —— 并发明了 *设备独立性* 。
当时的操作系统将 IO 设备抽象为处理类似于卡片的单元记录的软件功能。
程序将调用操作系统中处理抽象单元记录设备的服务。
操作员可以告诉操作系统这些抽象服务应该连接到读卡器、磁带还是任何其他单元记录设备。

现在，同一个程序 *无需任何更改* 就可以读写卡片，或读写磁带。
开闭原则 (Open–Closed Principle) 诞生了（尽管当时尚未命名）。

## 垃圾邮件

在 20 世纪 60 年代末，我曾为一家为客户打印垃圾邮件的公司工作。
客户会寄来磁带，上面有包含其客户姓名和地址的单元记录，然后我们编写程序，打印出精美的个性化广告。

你知道那种广告：

> 你好，Martin 先生，
>
> 恭喜您！
>
> 我们从 Witchwood Lane 的所有居民中特别挑选了您，来参与我们全新、绝无仅有的优惠活动……

客户会寄来巨大的表格信件卷，除了姓名、地址以及他们希望我们打印的其他任何元素之外，所有文字都已印好。
我们编写程序从磁带中提取姓名、地址以及其他元素，并将这些元素精确地打印在表格上需要出现的位置。

这些表格信件卷每卷重 500 磅，包含成千上万封信件。
客户会寄给我们数百卷这样的卷。我们会逐一打印每封信。

起初，我们有一台 IBM 360，用它的唯一一台行式打印机进行打印。
每个班次我们能打印几千封信。
不幸的是，这使得一台非常昂贵的机器被长时间占用。
在那个年代，IBM 360 的月租金高达数万美元。

于是，我们告诉操作系统使用磁带而不是行式打印机。
我们的程序并不在意，因为它们是根据操作系统的 IO 抽象层编写的。

360 可以在大约 10 分钟内输出一盘满的磁带 —— 足以打印好几卷表格信件。
这些磁带被带到计算机房外，挂载到连接到离线打印机的磁带驱动器上。
我们有五台这样的打印机，每周七天、每天 24 小时运行，每周打印出数十万封垃圾邮件。

设备独立性的价值是巨大的！
我们可以在不知道或不关心将使用哪个设备的情况下编写程序。
我们可以使用连接到计算机的本地行式打印机来测试这些程序。
然后我们可以告诉操作系统 “打印” 到磁带上，并跑出数十万份表格。

我们的程序有一个形状。
该形状将策略与细节分离。
策略是姓名和地址记录的格式化。
细节是设备。
我们推迟了关于使用哪个设备的决策。

## 物理寻址

在 20 世纪 70 年代初，我为当地一个卡车司机工会开发了一个大型会计系统。
我们有一个 25MB 的磁盘驱动器，上面存储了代理人、雇主和会员的记录。
不同的记录有不同的大小，因此我们将磁盘的前几个柱面格式化，使每个扇区正好是一个代理人记录的大小。
接下来的几个柱面被格式化为扇区大小适合雇主记录。
最后几个柱面被格式化为适合会员记录。

我们编写的软件知道磁盘的详细结构。
它知道磁盘有 200 个柱面和 10 个磁头，每个柱面每个磁头有几十个扇区。
它知道哪些柱面存放代理人、雇主和会员。
所有这些都被硬编码在代码中。

我们在磁盘上维护了一个索引，以便查找每个代理人、雇主和会员。
这个索引位于磁盘上另一组特殊格式化的柱面中。
代理人索引由包含代理人 ID 以及该代理人记录的柱面号、磁头号和扇区号的记录组成。
雇主和会员有类似的索引。
会员还以双向链表的形式存储在磁盘上。
每个会员记录存放着下一个会员记录和前一个会员记录的柱面号、磁头号和扇区号。

如果我们需要升级到新的磁盘驱动器 ——比如磁头数更多、柱面数更多、或每个柱面的扇区数更多—— 会发生什么？
我们必须编写一个特殊的程序，从旧磁盘读取旧数据，然后将其写入新磁盘，同时转换所有的柱面/磁头/扇区号。
我们还必须更改代码中所有的硬编码 —— 而这些硬编码无处不在！
所有的业务规则都详细地知道柱面/磁头/扇区方案。

有一天，一位更有经验的程序员加入了我们的团队。
当他看到我们所做的事情时，脸上的血色瞬间褪去，他目瞪口呆地盯着我们，好像我们是某种外星人。
然后他温和地建议我们改变寻址方案，改用相对地址。

我们这位睿智的同事建议：将磁盘视为一个巨大的线性扇区数组，每个扇区按顺序整数寻址。
然后我们可以编写一个小的转换例程，该例程知道磁盘的物理结构，并能够动态地将相对地址转换为柱面/磁头/扇区号。

幸运的是，我们接受了他的建议。
我们改变了系统的高层策略，使其对磁盘的物理结构一无所知。
这使我们能够将关于磁盘驱动器结构的决策与应用程序解耦。

## 结论

本章中的两个故事是小范围内的示例，展示了架构师在大范围内运用的原则。
<ins>优秀的架构师谨慎地将细节与策略分离，然后将策略与细节彻底解耦，使得策略对细节一无所知，也绝不依赖任何细节。
优秀的架构师设计策略时，会将对细节的决策尽可能长时间地推迟和延后</ins>。
