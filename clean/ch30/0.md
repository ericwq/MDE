# 30 数据库是细节

---
<center><img src="../imgs/fig30-0.png" width="50%"/></center><br/>

从架构的角度来看，数据库是一个非实体 —— 它是一个不足以提升到架构元素层面的细节。
它与软件系统架构的关系，颇类似于门把手与你家房屋架构的关系。

我知道这些都是挑衅性的话。
相信我，我已经经历过争论。
所以让我明确一下：我不是在谈论数据模型。
你在应用程序中赋予数据的结构，对你的系统架构非常重要。
但数据库不是数据模型。
数据库是一个软件。
数据库是一个提供数据访问的工具。
从架构的角度来看，这个工具无关紧要，因为它是一个低层次的细节 —— 一种机制。
优秀的架构师不会让低层次机制污染系统架构。

## 关系数据库

Edgar Codd 在 1970 年定义了关系数据库的原则。
到 1980 年代中期，关系模型已发展成为数据存储的主要形式。
这种普及是有充分理由的：关系模型优雅、严谨且健壮。
它是一种优秀的数据存储和访问技术。

但无论一项技术多么出色、有用且在数学上多么严谨，它仍然只是一项技术。
这意味着它是一个细节。

虽然关系表可能对某些形式的数据访问很方便，但将数据安排在表中的行里，在架构上并无重要意义。
你的应用程序的用例既不应该知道，也不应该关心这些事情。
事实上，对数据表格结构的了解应被限制在架构最外圈的最低层工具函数中。

<ins>许多数据访问框架允许将数据库行和表作为对象在系统中传递。
允许这样做是一个架构错误。
它将用例、业务规则，在某些情况下甚至将 UI，与数据的关系结构耦合在一起</ins>。

## 为什么数据库系统如此普及？

为什么软件系统和软件企业被数据库系统所主导？
Oracle、MySQL 和 SQL Server 为何如此显赫？
一言以蔽之：磁盘。

旋转式磁盘在五十年间一直是数据存储的中流砥柱。
好几代程序员所知道的数据存储形式莫过于此。
磁盘技术已经从直径 48 英寸、重达数千磅、容量 20 兆字节的巨大盘片堆栈，发展到直径 3 英寸、仅重数克、容量达 1 TB 或更多的单片薄圆盘。
这是一段狂野的旅程。
而在这段旅程中，程序员一直被磁盘技术的一个致命特性所困扰：磁盘很慢。

在磁盘上，数据存储在圆形磁道中。
这些磁道被划分为扇区，每个扇区保存着方便数量的字节，通常为 4K。
每个盘片可能有数百条磁道，一盘可能有十几个盘片。
如果你想从磁盘上读取某个特定的字节，你必须将磁头移动到正确的磁道，等待磁盘旋转到正确的扇区，
将整个 4K 的扇区读入 RAM，然后在该 RAM 缓冲区中索引以获取你想要的字节。
而这一切都需要时间 —— 毫秒级的时间。

毫秒可能看起来不算多，但一毫秒比大多数处理器的周期时间长一百万倍。
如果数据不在磁盘上，它可以在纳秒级时间内被访问，而不是毫秒级。

为了减轻磁盘带来的时间延迟，你需要索引、缓存和优化的查询方案；
而且你需要某种规范的数据表示方式，以便这些索引、缓存和查询方案知道它们在处理什么。
简而言之，你需要一个数据访问和管理系统。
多年来，这些系统已经分化为两种不同的类型：文件系统和关系数据库管理系统（RDBMS）。

文件系统是基于文档的。
它们提供了一种自然且方便的方式来存储整个文档。
当你需要按名称保存和检索一组文档时，它们工作得很好，但在搜索这些文档的内容时，它们提供的帮助就不多了。
找到一个名为 `login.c` 的文件很容易，但要找到所有包含名为 `x` 的变量的 `.c` 文件，则既困难又缓慢。

数据库系统是基于内容的。
它们提供了一种自然且方便的方式来根据内容查找记录。
它们非常擅长基于某些共享的内容来关联多条记录。
但遗憾的是，它们在存储和检索不透明 (opaque) 文档方面相当弱。

<ins>这两种系统都在磁盘上组织数据，以便在满足各自特定访问需求的前提下，尽可能高效地进行存储和检索。
每种系统都有自己的索引和数据排列方案。
此外，它们最终都会将相关数据调入 RAM，以便在那里进行快速操作</ins>。

## 如果没有磁盘会怎样？

尽管磁盘曾经非常普及，但它们现在正逐渐消亡。
很快它们将步磁带机、软盘驱动器和 CD 的后尘。
它们正在被 RAM 所取代。

问你自己一个问题：当所有磁盘都消失，你所有的数据都存储在 RAM 中时，你将如何组织这些数据？
你会将它们组织成表，并用 SQL 访问吗？
你会将它们组织成文件，并通过目录访问吗？

当然不会。
你会将它们组织成链表、树、哈希表、栈、队列，或任何其他各种数据结构，并用指针或引用来访问它们 —— 因为这就是程序员所做的。

事实上，如果你仔细思考这个问题，你就会意识到这正是你已经在做的事情。
即使数据保存在数据库或文件系统中，你也会将其读入 RAM，
然后为了方便起见，将其重新组织成列表、集合、栈、队列、树，或任何你喜欢的其他数据结构。
你极不可能将数据保留在文件或表的形式中。

## 细节

正是这种现实使得我说数据库是一个细节。
<ins>它只是我们用来在磁盘表面和 RAM 之间来回移动数据的一种机制。
数据库实际上不过是一大桶的比特，我们在其中长期存储数据。
但我们很少以那种形式使用数据</ins>。

因此，从架构的角度来看，我们不应该关心数据在旋转磁盘表面时所采用的形式。
事实上，我们应该完全当作磁盘不存在。

## 那性能呢？

<ins>性能不是架构关注点吗？
当然是 —— 但就数据存储而言，它是一个可以完全封装并与业务规则分离的关注点</ins>。
是的，我们需要快速地将数据存入和取出数据存储，但那是一个低层次的关注点。
我们可以用低层次的数据访问机制来解决这个问题。
它与我们系统的整体架构毫无关系。

## 轶事

在 1980 年代末，我在一家初创公司领导一个软件工程师团队，试图构建并销售一种网络管理系统，用于测量 T1 电信线路的通信完整性。
该系统从这些线路端点的设备中检索数据，然后运行一系列预测算法来检测和报告问题。

我们使用的是 UNIX 平台，数据存储在简单的随机访问文件中。
我们不需要关系数据库，因为我们的数据几乎没有基于内容的关系。
将它们保存在这些随机访问文件中的树和链表里更为合适。
简而言之，我们将数据保存在最便于加载到 RAM 中进行操作的形式中。

我们为这家初创公司聘请了一位营销经理 —— 一个很不错且知识渊博的人。
但他立即告诉我，系统中必须有一个关系数据库。
这不是一个可选项，也不是一个工程问题 —— 这是一个营销问题。

这对我来说毫无意义。
我为什么要费尽心思把我的链表和树重新排列成通过 SQL 访问的一堆行和表？
当一个简单的随机访问文件系统已经绰绰有余时，我为什么要引入一个庞大 RDBMS 的所有开销和成本？
因此我极力反对他。

我们公司有一位硬件工程师，他也加入了 RDBMS 的合唱。
他确信我们的软件系统出于技术原因需要一个 RDBMS。
他在我背后与公司高管们开会，在白板上画着房子立在杆子上的简笔画，然后问高管们：“你们会把房子建在杆子上吗？”
他隐含的信息是，一个将表保存在随机访问文件中的 RDBMS，在某种程度上比我们正在使用的随机访问文件更可靠。

我反对他。
我反对那个营销人员。
面对惊人的无知，我坚持自己的工程原则。
我反对，反对，再反对。

最终，那位硬件开发者被提升到我之上，成为了软件经理。
最终，他们给那个可怜的系统加上了 RDBMS。
最终，他们完全正确，而我错了。

请注意，不是因为工程原因：我在这方面是对的。
我反对将 RDBMS 放入系统架构核心是对的。
我错的原因是我们的客户期望我们有一个关系数据库。
他们不知道他们会用它做什么。
他们没有任何现实的方法来使用我们系统中的关系数据。
但这没关系：我们的客户完全期望有一个 RDBMS。
它已经成了所有软件采购者清单上的一个复选框项。
没有工程上的理由 —— 理性与它无关。
这是一种非理性的、外部的、完全无根据的需求，但它同样是真实的。

这种需求从何而来？
它源于当时数据库供应商所采用的高效营销活动。
他们设法说服了高层管理人员，认为企业的 “数据资产” 需要保护，而他们提供的数据库系统是实现这种保护的理想手段。

今天我们也能看到同样的营销活动。“企业” 这个词和 “面向服务架构” 的概念，与营销的关系远比与现实的关系更大。

在很久以前的那个场景中，我本该怎么做？
我本应该在系统的侧面接上一个 RDBMS，并为它提供一条狭窄且安全的数据访问通道，同时在系统的核心中保留随机访问文件。
而我做了什么？
我辞职去当了顾问。

## 结论

<ins>数据的组织结构，即数据模型，在架构上是重要的。
而那些将数据在旋转磁盘表面来回移动的技术和系统则不重要</ins>。
那些强迫数据以表的形式组织并通过 SQL 访问的关系数据库系统，与后者（技术/系统）的关系远大于与前者（数据模型）的关系。
数据是重要的。
数据库是一个细节。
