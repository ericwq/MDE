# 16 独立性

---
<center><img src="../imgs/fig16-0.png" width="60%"/></center><br/>

如前所述，良好的架构必须支持：

- 系统的用例和运行
- 系统的维护
- 系统的开发
- 系统的部署

## 用例

第一点 ——用例—— 意味着系统的架构必须支持系统的意图。
如果系统是一个购物车应用程序，那么架构必须支持购物车用例。
实际上，这是架构师的首要关注点，也是架构的第一优先事项。
架构必须支持用例。

然而，正如我们之前讨论过的，架构对系统行为的影响并不大。
架构能够保留的行为选项非常之少。
但影响并不是全部。
<ins>良好的架构为支持行为所能做的最重要的事情，就是阐明并暴露该行为，使得系统的意图在架构层面清晰可见</ins>。

一个具有良好架构的购物车应用程序，看上去就像一个购物车应用程序。
该系统的用例在其结构内部应该是清晰可见的。
开发者不需要去寻找行为，因为这些行为是一等实体，在系统的顶层清晰可见。
这些实体将是类、函数或模块，它们在架构中占据突出位置，并且具有能够清楚描述其功能的名称。

[21 尖叫架构](../ch21/0.md) 将更清楚地阐明这一点。

## 运行

在支持系统运行方面，架构扮演着更为实质性而非表面性的角色。
如果系统必须每秒处理 100,000 个客户，那么架构必须为每个有此需求的用例支持相应的吞吐量和响应时间。
如果系统必须在毫秒内查询大数据立方体，那么架构必须被结构化为允许此类操作。

对于某些系统而言，这意味着将系统的处理元素排列成一系列小服务，这些小服务可以在许多不同的服务器上并行运行。
对于其他系统，这意味着大量的轻量级线程，在单个处理器的单个进程内共享地址空间。
还有一些系统只需要在隔离的地址空间中运行少数几个进程。
甚至有些系统可以作为在单个进程中运行的简单单体程序而存活。

<ins>尽管看起来可能有些奇怪，但这个决策正是优秀架构师应保留的选项之一</ins>。
一个作为单体编写的、且依赖于该单体结构的系统，在需要时无法轻易升级为多进程、多线程或微服务。
相比之下，一个保持了组件间适当隔离、并且不假设组件间通信方式的架构，将更容易随着系统运行需求随时间变化，在线程、进程和服务的谱系中过渡。

## 开发

架构在支持开发环境方面扮演着重要角色。
这就是康威定律发挥作用的地方。康威定律指出：

> 任何设计系统的组织，都会产生一个设计，其结构是该组织沟通结构的复制。

一个必须由拥有众多团队和众多关注点的组织来开发的系统，其架构必须促进这些团队的独立行动，以便团队在开发过程中不会相互干扰。
这是通过将系统恰当地划分为良好隔离的、可独立开发的组件来实现的。
然后这些组件可以分配给能够彼此独立工作的团队。

## 部署

架构在决定系统部署的难易程度方面也扮演着重要角色。
目标是 “即时部署 (immediate deployment)”。
良好的架构不依赖于几十个小配置脚本和属性文件的调整。
它不需要手动创建必须按特定方式排列的目录或文件。
良好的架构帮助系统在构建后能够立即部署。

同样，这是通过恰当划分和隔离系统组件（包括那些将整个系统结合在一起、并确保每个组件被正确启动、集成和监督的主控组件）来实现的。

## 保留选项

良好的架构通过一个组件结构来平衡所有这些关注点，使它们相互得到满足。
听起来很容易，对吧？
好吧，我写出来是很容易。

现实是，实现这种平衡相当困难。
问题在于，大多数时候我们不知道所有的用例是什么，也不知道运行约束、团队结构或部署需求。
更糟糕的是，即使我们知道了，它们也会随着系统在其生命周期中的演进而不可避免地发生变化。
简而言之，我们必须达到的目标是模糊且不稳定的。
欢迎来到现实世界。

<ins>但并非全无希望：有一些架构原则实现起来成本相对较低，并且即使在你不清楚必须达成什么目标的情况下，也能帮助平衡这些关注点。
这些原则帮助我们划分系统，形成良好隔离的组件，使我们能够尽可能长时间地保留尽可能多的选项</ins>。

良好的架构通过保留选项，使系统易于在其必须改变的所有方面进行改变。

## 解耦分层

考虑用例。
架构师希望系统的结构能够支持所有必要的用例，但他并不知道所有这些用例是什么。
然而，架构师确实知道系统的基本意图。
这是一个购物车系统，或者是一个物料清单系统，或者是一个订单处理系统。
因此，鉴于系统意图的上下文，架构师可以运用单一职责原则和共同闭包原则，分离那些因不同原因而变化的事物，并聚集那些因相同原因而变化的事物。

什么会因不同原因而变化？
有一些显而易见的事物。
用户界面的变化原因与业务规则毫无关系。
用例则同时包含两者的元素。
因此，优秀的架构师显然希望将用例的 UI 部分与业务规则部分分离开，使它们可以彼此独立地变化，同时保持这些用例清晰可见。

业务规则本身可能与应用程序紧密相关，也可能更为通用。
例如，输入字段的验证是一个与应用程序本身紧密相关的业务规则。
相比之下，账户利息的计算和库存的计数则是与领域更紧密相关的业务规则。
这两种不同类型的规则会以不同的速率、因不同的原因而变化 —— 因此它们应该被分离，以便能够独立地改变。

数据库、查询语言、甚至模式都是与业务规则或 UI 无关的技术细节。
它们变化的速率和原因独立于系统的其他方面。
因此，架构应该将它们与系统的其余部分分离开，以便能够独立地改变。

<ins>于是，我们发现系统被划分为解耦的水平层 —— UI、特定于应用程序的业务规则、独立于应用程序的业务规则以及数据库，仅举几例</ins>。

## 解耦用例

还有什么会因不同原因而变化？
用例本身！
在订单录入系统中，添加订单的用例几乎肯定会以与删除订单用例不同的速率、因不同的原因而变化。
用例是一种非常自然的系统划分方式。

<ins>同时，用例是窄的 *垂直切片 (vertical slices)*，贯穿系统的水平层</ins>。
每个用例都会使用一些 UI、一些特定于应用程序的业务规则、一些独立于应用程序的业务规则以及一些数据库功能。
因此，当我们将系统划分为水平层时，我们也在将系统划分为贯穿这些层的薄的垂直用例。

为了实现这种解耦，我们将添加订单用例的 UI 与删除订单用例的 UI 分离开。
对于业务规则和数据库，我们也同样处理。
我们沿着系统的垂直高度将用例保持分离。

<ins>你可以在这里看到模式。
如果你解耦了系统中因不同原因而变化的元素，那么你就可以继续添加新的用例而不会干扰旧的用例。
如果你还将 UI 和数据库分组以支持这些用例，使得每个用例使用 UI 和数据库的不同方面，那么添加新的用例就不太可能影响旧的用例</ins>。

## 解耦模式

现在思考所有这些解耦对第二点 ——运行—— 意味着什么。
如果用例的不同方面是分离的，那么那些必须以高吞吐量运行的部分很可能已经与那些必须以低吞吐量运行的部分分离开了。
如果 UI 和数据库已经与业务规则分离，那么它们就可以在不同的服务器上运行。
那些需要更高带宽的部分可以在多台服务器上进行复制。

<ins>简而言之，我们为用例所做的解耦也有助于运行</ins>。
然而，要利用运行方面的好处，解耦必须具有适当的模式。
为了在不同的服务器上运行，分离出的组件不能依赖于在同一个处理器的同一地址空间中共存。
它们必须是独立的服务，通过某种网络进行通信。

许多架构师根据某种模糊的代码行数概念，将此类组件称为 “服务” 或 “微服务”。
实际上，基于服务的架构通常被称为 *面向服务架构*。

如果这个术语集在你的脑海中敲响了某些警钟，别担心。我不会告诉你 SOA 是最好的架构，也不会说微服务是未来的浪潮。
<ins>这里要说明的是：有时我们必须将组件一直分离到服务层面</ins>。

<ins>请记住，良好的架构保留选项。
*解耦模式就是这些选项之一*</ins>。

在进一步探讨这个话题之前，让我们先看看另外两点。

## 独立的可开发性

第三点是开发。
显然，当组件被强解耦时，团队之间的干扰就会减少。
如果业务规则不知道 UI，那么专注于 UI 的团队就不会对专注于业务规则的团队产生太大影响。
如果用例本身相互解耦，那么专注于 `addOrder` 用例的团队就不太可能干扰专注于 `deleteOrder` 用例的团队。

只要层和用例是解耦的，系统的架构就会支持团队的组织方式，无论他们是按功能团队、组件团队、分层团队还是其他一些变体来组织的。

## 独立的可部署性

用例和层的解耦也提供了高度的部署灵活性。
实际上，如果解耦做得好，那么应该可以在正在运行的系统中热替换层和用例。
添加一个新用例可能就像向系统添加几个新的 jar 文件或服务一样简单，而其他部分则保持不变。

## 重复

架构师常常落入一个陷阱 —— 一个因恐惧重复而陷入的陷阱。

重复在软件中通常是一件坏事。
我们不喜欢重复的代码。
当代码确实是重复的时，作为专业人士，我们有义务减少并消除它。

<ins>但重复有不同的类型。
有真正的重复，其中对一个实例的每次更改都需要对该实例的每个副本进行相同的更改。
还有虚假的或偶然的重复</ins>。
如果两个看似重复的代码片段沿着不同的路径演化 ——如果它们以不同的速率、因不同的原因而变化—— 那么它们就不是真正的重复。
几年后再回来看，你会发现它们彼此之间差异很大。

现在想象两个具有非常相似屏幕结构的用例。
架构师很可能非常想共享该结构的代码。
但他们应该这样做吗？
这是真正的重复吗？
还是偶然的？

很可能是偶然的。
随着时间的推移，这两个屏幕很可能会分化，最终看起来非常不同。
因此，必须小心避免将它们统一在一起。
否则，以后再将它们分开将是一个挑战。

当你将用例彼此垂直分离时，你会遇到这个问题，并且你会因为用例具有相似的屏幕结构、相似的算法、或相似的数据库查询和/或模式而倾向于将它们耦合在一起。
要小心。
<ins>抵制住下意识消除重复的诱惑。
确保重复是真实的</ins>。

同样，当你水平分离层时，你可能会注意到某个特定数据库记录的数据结构与某个特定屏幕视图的数据结构非常相似。
你可能会倾向于简单地将数据库记录向上传递到 UI，而不是创建一个看起来相同的视图模型并逐个复制元素。
要小心：这种重复几乎肯定是偶然的。
创建单独的视图模型并不需要太多工作，而且它将帮助你保持层的适当解耦。

## 解耦模式（再次）

回到模式。
解耦分层和用例的方法有很多种。
它们可以在源代码级别、二进制代码（部署）级别以及执行单元（服务）级别进行解耦。

- **源代码级别**：
我们可以控制源代码模块之间的依赖关系，使得对一个模块的更改不会强制其他模块更改或重新编译（例如 Ruby Gems）。
  
  在这种解耦模式下，所有组件都在同一个地址空间中执行，并使用简单的函数调用相互通信。
  只有一个可执行文件被加载到计算机内存中。
  人们通常称之为 *单体结构 (monolithic structure)* 。

- **部署级别**：
我们可以控制可部署单元（如 jar 文件、DLL 或共享库）之间的依赖关系，使得对一个模块中源代码的更改不会强制其他模块被重新构建和重新部署。

  许多组件可能仍然存在于同一个地址空间中，并通过函数调用进行通信。
  其他组件可能存在于同一处理器的其他进程中，并通过进程间通信、socket 或共享内存进行通信。
  这里重要的是，解耦的组件被划分为独立可部署的单元，如 jar 文件、Gem 文件或 DLL。

- **服务级别**：
我们可以将依赖关系降低到数据结构级别，并仅通过网络包进行通信，使得每个执行单元完全独立于其他单元的源代码和二进制变化（例如服务或微服务）。

什么是最好的模式？

答案是：在项目的早期阶段，很难知道哪种模式是最好的。
实际上，随着项目的成熟，最佳模式可能会发生变化。

例如，不难想象一个现在在一台服务器上运行良好的系统，将来可能会增长到其某些组件应该在单独的服务器上运行的程度。
当系统在单个服务器上运行时，源代码级别的解耦可能就足够了。
然而，之后它可能需要解耦为可部署单元，甚至服务。

<ins>一个解决方案（目前似乎很流行）是简单地默认在服务级别进行解耦。
这种方法的一个问题是成本高昂，并鼓励粗粒度的解耦。
无论微服务有多么 “微”，解耦都不太可能足够细粒度</ins>。

服务级解耦的另一个问题是，无论是在开发时间还是系统资源方面，它都很昂贵。
处理本不需要的服务边界是对精力、内存和 CPU 周期的浪费。
而且，是的，我知道后两者很便宜 —— 但前者不便宜。

<ins>我的偏好是：将解耦推进到足以在必要时形成一个服务的程度；
但随后尽可能长时间地将组件保留在同一个地址空间中。
这保留了服务的选项</ins>。

采用这种方法，最初组件在源代码级别被分离。
这可能足以满足项目整个生命周期的需求。
然而，如果出现部署或开发问题，将部分解耦推进到部署级别可能就足够了 —— 至少在一段时间内是这样。

随着开发、部署和运行问题的增加，我会仔细选择将哪些可部署单元转变为服务，并逐渐将系统朝那个方向转变。

随着时间的推移，系统的运行需求可能会下降。
曾经需要在服务级别解耦的部分，现在可能只需要部署级别甚至源代码级别的解耦。

<ins>良好的架构将允许一个系统以单体诞生，部署在单个文件中，然后成长为一组独立可部署的单元，再一路成长为独立的服务和/或微服务。
随后，随着情况的变化，它应该允许逆转这一进程，一路回退成一个单体</ins>。

<ins>良好的架构保护了大部分源代码免受这些变化的影响。
它将解耦模式 (decoupling mode) 作为一个开放的选项，使得大型部署可以使用一种模式，而小型部署则可以使用另一种模式</ins>。

## 结论

是的，这很棘手。
我并不是说解耦模式的改变应该是一个微不足道的配置选项（尽管有时这样做是合适的）。
我是说：系统的解耦模式是那些可能随时间改变的事情之一，而优秀的架构师能够预见并恰当地促进这些改变。
