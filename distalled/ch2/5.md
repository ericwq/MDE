## 开发通用语言

那么，当你在实践中运用 DDD 提供的主要工具之一时，你到底如何在团队内部开发 *通用语言 (Ubiquitous Language)* 呢？
你的 *通用语言 (Ubiquitous Language)* 是由一组众所周知的名词构成的吗？
名词很重要，但软件开发人员往往过分强调领域模型中的名词，忘记了口头语言由远不止名词组成。
诚然，在之前的示例 *限界上下文 (Bounded Context)* 中，我们主要关注了名词，但那是出于我们对 DDD 另一方面的兴趣，即将 *核心域 (Core Domain)* 约束到基本的模型元素。

---
**加速你的发现**

你可能希望在处理你的场景时尝试几次 *事件风暴 (Event Storming)* 会议。
这些会议可以帮助你快速了解你应该处理哪些场景，以及它们应该如何被优先排序。
同样，开发具体的场景会让你更好地了解你在 *事件风暴 (Event Storming)* 会议中应该采取的方向。
它们是两个配合良好的工具。
我在 [第七章](../ch7/0.md) “加速与管理工具” 中解释了 *事件风暴 (Event Storming)* 的使用。

---

<ins>不要将你的 *核心域 (Core Domain)* 仅限于名词</ins>。
相反，考虑将你的 *核心域 (Core Domain)* 表达为一组关于领域模型应该做什么的具体场景。
<ins>当我说 “场景” 时，我不是指用例或用户故事，就像软件项目中常见的那样。
我字面上指的是关于领域模型应该如何工作的场景 —— 各个组件做什么。
只有通过 *领域专家 (Domain Experts)* 和开发人员组成的团队协作，才能以最彻底的方式完成这一点</ins>。

*「用户故事/用例问的是 “用户要做什么？”，而 DDD 场景问的是 “模型要如何表现（行为）才能支撑业务？”」*

这是一个符合 Scrum *通用语言 (Ubiquitous Language)* 的场景示例：

> 允许每个待办项被承诺到一个冲刺中。
只有当待办项已经被安排进一个发布时，它才能被承诺。
如果它已经被承诺到不同的冲刺中，则必须首先取消承诺。
当承诺完成时，通知相关方。

请注意，这不仅仅是一个关于人类如何在项目中使用 Scrum 的场景。
我们不是在谈论人类规程。
相反，<ins>这个场景描述了如何使用真实的软件模型组件来支持基于 Scrum 的项目的管理</ins>。

前面的场景并不是一个完美表述的场景，使用 DDD 的一个好处是我们不断寻找改进模型的方法。
然而，这是一个不错的开始。
<ins>我们听到名词被说出，但我们的场景并没有将我们限制在名词上。
我们还听到动词和副词，以及其他类型的语法。
你还听到存在约束 —— 在场景能够成功完成之前必须满足的条件。
最重要的好处和赋能特征是，你实际上可以进行关于领域模型如何工作 ——即其设计—— 的对话</ins>。

#### 关键区别：用户故事/用例 vs. DDD 场景

| 维度 | 用户故事 / 用例 | DDD 场景 |
|------|----------------|----------|
| **关注点** | 用户与系统的交互流程 | **领域模型内部的行为与协作** |
| **视角** | 外部视角：用户如何使用系统 | **内部视角**：模型组件如何工作 |
| **主语** | 通常是 “用户”、“角色” | 通常是 “实体”、“聚合”、“领域服务” |
| **目的** | 描述功能需求 | **揭示模型设计、约束和逻辑** |
| **产出** | 验收标准、UI 行为 | 模型概念、不变量、事件、职责分配 |


#### 为什么 DDD 场景不是用例？

1. **不涉及人类操作流程**：场景不描述 “用户点击按钮 → 弹出窗口” 这类交互。它描述的是模型在**内存中**如何变化。

2. **直接指导建模**：从场景中你可以直接识别出：
   - **实体**（如 `BacklogItem`、`Sprint`）
   - **不变量**（如“必须已安排发布”）
   - **行为方法**（如 `commitTo(sprint)`、`uncommit()`）
   - **领域事件**（如 `BacklogItemCommitted`）

3. **促进对话**：场景是团队（开发+领域专家）讨论模型设计的**起点**。例如：
   - “谁执行承诺？” → 引入 `ProductOwner` 角色
   - “谁被通知？” → 引出 `Sprint` 的更新逻辑

我们甚至可以绘制简单的图片和图表。
这一切都是为了在团队中更好地沟通。
这里有一个恰当的警告。
<ins>要注意在领域建模工作中花费的时间，尤其是在长期保持带有书面场景、图纸和图表的最新状态方面</ins>。
那些东西不是领域模型。
相反，它们只是帮助你开发领域模型的工具。
最终，<ins>代码就是模型，模型就是代码</ins>。
仪式是为隆重的庆典准备的，比如婚礼，而不是领域模型。
这并不意味着你放弃任何刷新场景的努力，但只应在有帮助而不是负担的情况下进行。

你会如何改进我们之前示例中 *通用语言 (Ubiquitous Language)* 的一部分？
想一想，就一分钟。
缺少了什么？
<ins>不久你可能希望了解 *谁* 将待办项承诺到冲刺中</ins>。
让我们添加谁，看看会发生什么：

> 产品负责人将每个待办项承诺到一个冲刺中……

<ins>你会在许多情况下发现，你应该为场景中涉及的每个角色命名，并为其他概念（如待办项和冲刺）赋予一些区分属性</ins>。
这将有助于使你的场景更加具体，而不像是一组关于验收标准的陈述。
尽管如此，在这种特定情况下，没有充分的理由命名产品负责人或进一步描述所涉及的待办项和冲刺。
在这种情况下，所有产品负责人、待办项和冲刺都将以相同的方式工作，无论它们是否有具体的角色或身份。
在场景中为概念命名或赋予其他区分身份有帮助的情况下，使用它们：

> 产品负责人 Isabel 将 “查看用户资料” 待办项承诺到 “交付用户资料” 冲刺中……

现在让我们暂停一下。
并不是产品负责人是决定待办项将被承诺到冲刺中的唯一责任人。
Scrum 团队不会喜欢那样，因为他们将被承诺在某个他们无权决定的时间范围内交付软件。
尽管如此，对于我们的软件模型来说，由一个人负责在模型上执行这个特定操作可能是最实际的。
因此，在这种情况下，我们声明是产品负责人角色执行此操作。
<ins>即便如此，Scrum 团队的性质迫使我们提出这个问题：“团队的其他成员是否需要做些什么来使产品负责人能够执行承诺？”</ins>

你看到发生了什么吗？
通过用 “谁” 的问题挑战当前模型，我们被引导到了一个更深入洞察模型的机会。
<ins>也许我们应该要求至少有一定程度的团队共识，即在实际允许产品负责人执行承诺操作之前，待办项可以被承诺</ins>。
这可能导致以下精炼的场景：

> 产品负责人将一个待办项承诺到一个冲刺中。只有当待办项已经被安排进发布，并且团队成员的法定人数已批准承诺时，它才能被承诺……

好的，现在我们有了一个精炼的 *通用语言 (Ubiquitous Language)*，因为我们已经确定了一个新的模型概念，称为 *法定人数 (quorum)*。
我们决定必须有一定数量的团队成员同意待办项应该被承诺，并且必须有一种方式让他们批准承诺。
这现在引入了一个新的建模概念，以及一些关于用户界面必须促进这些团队交互的想法。
你看到创新正在展开吗？

模型中还缺少另一个 “谁”。
哪一个？我们的初始场景以以下内容结束：

> 当承诺完成时，通知相关方。

<ins>谁或什么是 “相关方”？
这个问题和挑战进一步引导了建模的洞见</ins>。
谁需要知道待办项何时已被承诺到一个冲刺中？
实际上，一个重要的模型元素是冲刺本身。
冲刺需要跟踪总的冲刺承诺，以及交付冲刺所有任务所需的努力。
无论你决定如何设计冲刺来跟踪这些，现在重要的是当待办项被承诺到一个冲刺时，通知该冲刺：

> 如果它已经被承诺到不同的冲刺中，则必须首先取消承诺。
当承诺完成时，通知它被取消承诺的冲刺以及它现在被承诺到的冲刺。

现在我们有了一个相当不错的领域场景。
这个总结语句也让我们理解到，待办项和冲刺可能不一定会同时意识到承诺。
我们需要向业务方确认，但这听起来像是一个引入 *最终一致性 (eventual consistency)* 的好地方。
你将在 [第五章](ch5/0.md) “聚合的战术设计” 中看到为什么这很重要以及如何实现。

完整的精炼场景如下所示：

> 产品负责人将一个待办项承诺到一个冲刺中。
只有当待办项已经被安排发布，并且团队成员的法定人数已批准承诺时，它才能被承诺。
如果它已经被承诺到不同的冲刺中，则必须首先取消承诺。
当承诺完成时，通知它被取消承诺的冲刺以及它现在被承诺到的冲刺。

软件模型在实践中实际上会如何工作？
你可以很好地想象一个非常创新的用户界面支持这个软件模型。
当 Scrum 团队参与冲刺规划会议时，
团队成员使用他们的智能手机或其他移动设备对每个待办项添加他们的批准，因为它被讨论并同意在下一个冲刺中处理。
批准每个待办项的团队成员法定人数的共识，使产品负责人能够将所有已批准的待办项承诺到冲刺中。

### 将场景付诸实践

你可能想知道如何从书面场景转换为某种可用于根据团队规范验证领域模型的工件。
<ins>有一种名为 *实例化需求 (Specification by Example)* [Specification](../ref.md#specification) 的技术可以使用；
它也被称为 *行为驱动开发 (Behavior-Driven Development)* [BDD](../ref.md#bdd) 。
通过这种方法，你试图实现的是协作开发和精炼 *通用语言 (Ubiquitous Language)* ，以共同理解进行建模，并确定你的模型是否符合你的规范。
你将通过创建验收测试 (acceptance tests) 来做到这一点</ins>。
以下是我们如何将前面的场景重新表述为可执行规范：

```
场景：
产品负责人将一个待办项承诺到一个冲刺中

给定 一个已安排进入发布的待办项
并且 该待办项的产品负责人
并且 一个用于承诺的冲刺
并且 团队批准承诺的法定人数
当 产品负责人将该待办项承诺到冲刺中
那么 该待办项被承诺到冲刺中
并且 待办项已承诺事件被创建
```

<ins>以这种形式编写场景后，你可以创建一些支持代码并使用工具来执行此规范</ins>。
即使没有工具，你也可能会发现这种带有给定/当/那么 (given/when/then) 方法的场景编写形式，比之前的场景编写示例效果更好。
然而，将你的规范作为验证领域模型的一种手段来执行，可能令人难以抗拒。
我在 [第七章](../ch7/0.md) “加速与管理工具” 中对此做了进一步评论。

<ins>你不必使用这种形式的可执行规范来根据你的场景验证领域模型。
你可以使用单元测试框架来完成几乎相同的事情，创建验收测试（而非单元测试）来验证你的领域模型</ins>：

```csharp
/*
The product owner commits a backlog item to a sprint.
The backlog item may be committed only if it is already
scheduled for release, and if a quorum of team members
have approved commitment. When the commitment completes,
notify the sprint to which it is now committed.
*/

[Test]
public void ShouldCommitBacklogItemToSprint()
{
    // Given
    var backlogItem = BacklogItemScheduledForRelease();

    var productOwner = ProductOwnerOf(backlogItem);

    var sprint = SprintForCommitment();

    var quorum = QuorumOfTeamApproval(backlogItem, sprint);

    // When
    backlogItem.CommitTo(sprint, productOwner, quorum);

    // Then
    Assert.IsTrue(backlogItem.IsCommitted());

    var backlogItemCommitted =
        backlogItem.Events.OfType<BacklogItemCommitted>().SingleOrDefault();
    Assert.IsNotNull(backlogItemCommitted);
}
```

<ins>这种基于单元测试的验收测试方法达到了与可执行规范相同的目的</ins>。
这里的优势可能是能够更快速地编写这种场景验证，但代价是牺牲了一些可读性。
尽管如此，大多数 *领域专家 (Domain Experts)* 在开发人员的帮助下应该能够理解这段代码。
当使用这种方法时，最好在注释中维护与验证代码关联的场景文档形式，如本例所示。

无论你决定采用哪种方法，两者通常都会以红-绿（失败-通过）的方式使用，即你的规范在首次运行时将失败，因为尚未实现要验证的领域模型概念。
你通过一系列红色结果逐步精炼你的领域模型，直到你完全支持你的规范并且验证通过（你看到所有绿色）。
这些验收测试将直接与你的 *限界上下文 (Bounded Context)* 关联，并保存在其源代码仓库中。

### 长远来看呢？

现在你可能想知道，一旦创新停止、维护开始，我们应该如何支持 *通用语言 (Ubiquitous Language)* 。
实际上，一些最好的学习或知识获取是在很长一段时间内发生的，甚至是在某些人可能称之为 “维护” 的期间。
团队采取 “创新在维护开始时结束” 的观点是错误的。

可能发生的最糟糕的事情是将 “维护阶段” 的标签贴到 *核心域 (Core Domain)* 上。
持续的学习过程根本不是阶段。
早期开发的 *通用语言 (Ubiquitous Language)* 必须在岁月流逝中继续蓬勃发展。
诚然，它最终可能变得不那么重要，但很可能在相当长的一段时间内不会。
这都是你的组织对核心举措承诺的一部分。
如果无法做出这种长期承诺，那么你今天正在工作的这个模型真的是一个战略差异化因素，一个 *核心域 (Core Domain)* 吗？
