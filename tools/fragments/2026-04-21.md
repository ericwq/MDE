# 片段：4月21日
Martin Fowler：2026年4月21日

上周，Thoughtworks 发布了 [第 34 期《技术雷达》](https://www.thoughtworks.com/radar) 。
该技术雷达是我们每半年发布一次的行业技术洞察报告，
汇总了我们在技术领域的实践经验，重点呈现了我们已应用或密切关注的工具、技术、平台与编程语言。
本期报告共包含 118 项技术条目，每条目均简要阐述了我们对该项技术的评价与看法。

不出所料，本期雷达被人工智能相关主题占据。
其中一部分内容，是借助 LLM 的视角重新审视我们熟悉的领域：

> 人工智能给软件开发带来的一个有趣影响是，它不仅迫使我们展望未来，也推动我们重新审视行业的根基。
在编撰本期内容时，我们发现自己重新回顾了许多成熟技术，从结对编程到零信任架构，从变异测试到 DORA 指标。
我们也重新审视了软件工艺的核心原则，例如整洁代码、审慎设计、可测试性，以及将可访问性视为一等设计关注点。
这并非怀旧，而是为了平衡人工智能工具快速生成复杂代码所带来的问题，所必需的制衡。
我们还注意到命令行的再度兴起：在多年以易用性为名对其进行层层抽象之后，智能代理类工具正让开发者重新回到终端，将其作为主要交互界面。

我格外高兴地看到我的同事 Jim Gumbley 加入了撰写团队。
多年来，他一直是我获取安全相关信息的稳定来源，其中就包括参与本站 [威胁建模指南](https://martinfowler.com/articles/agile-threat-modelling.html) 的编写工作。
考虑到 LLM 在使用中存在严峻的安全问题，技术雷达团队中拥有强大的安全专业力量显得尤为重要。
本期雷达的主题之一，便是为 “权限饥渴型 (permission hungry)” 智能代理构建安全防护：

> “权限饥渴” 一词，描绘了当前智能代理领域的核心困境：值得打造的代理，恰恰是那些需要访问各类资源的代理。
OpenClaw 与 Claude Cowork 能够监管真实的工作任务；
Gas Town 可在整个代码库中协调智能代理集群。
这类代理需要广泛访问私有数据、进行外部通信并接入真实生产系统 —— 各方均认为其带来的收益足以 justify 这种权限需求。
>
> 然而，这就好比一名刚学会转弯的滑雪者，却信心满满地冲向难度最高的黑道雪道：安全防护措施尚未跟上这份野心。
对访问权限的渴求，与诸多悬而未决的问题产生了冲突。
提示注入问题意味着，LLM 仍无法可靠地区分可信指令与不可信输入。

鉴于以上种种，本期技术雷达中的多项内容都与管控工程 (Harness Engineering) 相关。
事实上，雷达研讨会也是 [Birgitta 那篇](../harness-engineering.md) 关于该主题的优秀文章的重要灵感来源。
本期雷达收录了多项技术建议，涉及构建完善管控所需的指导与监测手段。
我预计，六个月后下一期雷达发布时，这一清单还会继续扩充。

Mike Mason 探讨了 [当开发者不再阅读代码时会发生的问题](https://mikemason.ca/writing/ai-slop-code-april-2026/) 。

> Claude 生成的 Python 代码库大体上可以正常运行。
单元测试全部通过，并且经过数小时的真实场景测试，证实它能够成功管理我一套相当复杂的基础设施。
但在代码总量达到约 100KB 时，我注意到了一个问题：主文件膨胀到了约 50KB（2000 行），而 Claude Code 在需要编辑时，开始使用 sed 工具去查找并修改该文件内的代码。
当我看到这一幕时，立刻敲响了警钟。

除了这段 “如同朋友般” 的体验，他还对泄露后的 50 万行 Claude Code 代码进行了深入思考。

> 两种情况都真实存在：Claude Code 生成的代码里既有良好的架构设计，也有令人难以理解的混乱代码。
而这恰恰是问题的关键 —— 不亲自阅读代码，你根本无法分辨哪些是优质代码、哪些是杂乱实现。

他的结论形成了一个粗略的框架：一次性的分析脚本随便写写没问题；
但需要长期维护的工具和稳健代码，则必须定期由人工审查 —— 哪怕只是让人提示模型该如何评判优质代码，再让模型去评估代码也行。

> 一旦你说出：“这部分代码越写越大，我有点不放心，能不能优化一下？” 它就会做出正确的处理：合理拆分、新建类，有时甚至会为新模块编写单元测试。
它其实一直都懂，只是不会主动提出来。

他确实建议认真对待 CLAUDE.md 这份文档。
我不确定他是否尝试过 Rahul Garg 近期分享的诸多模式，那些模式正是用来 [打破他所遇到的这类无效循环困境](../reduce-friction-ai-main.md) 的。
