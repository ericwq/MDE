# 片段：2月13日
Martin Fowler：2026年2月13日

这周我一直在忙于出差，拜访了湾区的一些客户，并参加了 The Pragmatic Summit 。
因此，我没能像原本希望的那样，有充足时间分享更多在 [Thoughtworks 软件开发未来研讨会](https://martinfowler.com/bliki/FutureOfSoftwareDevelopment.html) 上的思考。
我仍在整理笔记并陆续发布片段内容，这里再补充一些：

❄ ❄

随着 LLM 日益普及，资深开发者将扮演怎样的角色？
在这场众多资深开发者参与的聚会上，我们一致认为，他们的未来依然光明 —— 将更多聚焦于架构层面的问题，而非繁琐的语法与编码细节。
部分过去十年里很少亲自写代码的人发现，借助 LLM ，他们可以重新回归编程；
而管理 LLM 智能体，在很多方面与管理初级开发者十分相似。

一位参会者分享道，尽管其团队中的资深开发者起初对使用 LLM 极为抵触，
但当这些资深开发者被迫参与一项实操练习、亲手使用 LLM 完成任务后，有三分之一的人立刻转变态度，成为 LLM 的坚定支持者。
这表明，要让资深人士获得可靠依据去判断其价值，实际体验至关重要，尤其是在过去几个月里模型能力已实现显著提升的情况下。
正如有人打趣说，某些对 LLM 能力的负面看法 “还停留在一月份的老黄历了” 。

❄ ❄

近几个月，网上充斥着对初级开发者职业前景的焦虑，人们担心他们会被不知疲倦的智能体所取代。
本次讨论小组对此则更为乐观，认为初级开发者仍有其不可替代的价值，至少他们对 LLM 持开放心态，且熟练掌握其用法。
真正面临最大挑战的是中级开发者。
他们的职业生涯是在没有 LLM 的环境中建立起来的，却又尚未像资深开发者那样，积累到足以高效驾驭这些模型的经验水平。

LLM 可以成为初级开发者的得力助手，如同一位随时在线的导师，帮助他们学习更优秀的编程方式。
当然，初级开发者应当对这位 AI 导师保持一定的怀疑态度，不过他们对真人导师也理应如此。
并非所有人都像自己想象中那样才华横溢。

❄ ❄

参会者 Margaret-Anne Storey 发表了一篇篇幅更长的文章，探讨 [认知债问题](https://margaretstorey.com/blog/2026/02/09/cognitive-debt/) 。

> 在我最近教授的一门创业课程中，我清晰地目睹了这种状况的发生。
各个学生团队在整个学期里开发软件产品，快速推进功能上线、达成各项里程碑目标。
但到了第 7、8 周左右，有一个团队彻底陷入了困境。
他们哪怕只做一点简单的修改，都会引发意料之外的故障。
我和他们沟通时，团队起初把问题归咎于技术债：代码混乱、架构糟糕、实现过程仓促。
但随着我们深入挖掘，真正的问题浮出水面：团队里没有人能说清当初为什么做出某些设计决策，也不清楚系统的各个模块本该如何协同工作。
代码或许确实杂乱，但更核心的问题在于，系统的原理、团队的共同认知已经支离破碎，甚至完全消失。
他们累积认知债的速度远超技术债，而这最终让项目陷入了停滞。

我认为这是一个值得思考的话题，但在我思索的同时，我看待它的方式与看待 [技术债](https://martinfowler.com/bliki/TechnicalDebt.html)颇为相似。
很多人把技术债简单理解为代码库中因草率开发而累积的不良产物 —— 比如模块边界混乱、命名糟糕等。
我把这类不良产物统称为 **糟粕 (cruft)** ，而我使用技术债这个比喻，是为了思考如何应对这些糟粕所带来的成本。
我们要么支付利息 —— 让后续对代码库的每一次修改都变得更加困难；
要么偿还本金 —— 通过明确的重构与重整，让代码更易于修改。

放到认知领域，这种糟粕与债务隐喻的区分又该如何对应？
我认为，糟粕的等价物就是无知 —— 既包括对代码本身的无知，也包括对代码所支撑的业务领域的无知。
此时，债务的比喻依然成立：要么我们为新增功能付出更高的代价，要么必须进行明确的知识投入来补齐认知。
债务这个比喻提醒我们，如何选择取决于两者之间的相对成本。
而在认知问题上，这些成本既会落在人身上，也会落在 AI 模型身上。

❄ ❄

我们当中许多人长期以来一直倡导通过改善开发者体验（DevEx）来 [提升软件开发团队的效能](https://martinfowler.com/articles/developer-effectiveness.html) 。
[Laura Tacho](https://lauratacho.com/) 如此评价道：

> 开发者体验与智能体体验的韦恩图 (Venn Diagram)，是一个同心圆。

我们为开发者所倡导的许多举措，同样也能让 LLM 更高效地工作。
流畅的工具链、清晰的开发环境说明，能帮助 LLM 快速且正确地编写代码。
尽管 AI 的 “超级大脑” 或许能够理解混乱的代码库，
但越来越多的证据表明，良好的模块化设计和表意清晰的命名，对 Transformer 模型和对人类这些 “柔软的神经网络” 同样有益。
这一点正逐渐被软件开发管理层所认可，进而推动他们为 LLM 铺平使用路径。
但正如 Laura 所指出的：令人唏嘘的是，这意味着管理者愿意为机器投入心力，却不愿为人类开发者做同样的事。

❄ ❄

IDE 依然有未来，但需要将 LLM 融入其工作流程。
一种方式是利用 LLM 来支持那些确定性方法无法实现的功能，例如从自然语言文档生成代码。
但也有大量任务并不适合使用 LLM —— 比如，用它来重命名函数就显得极其低效。

LLM 的另一个作用是帮助用户更高效地使用 IDE。
毕竟现代 IDE 是复杂的工具，很少有用户能充分发挥它们的全部潜力。
（作为一名长期使用 Emacs 的用户，我对此深有同感。）
IDE 可以辅助用户判断：何时针对任务使用 LLM ，何时使用 IDE 本身的确定性功能，以及何时将两者结合使用。

举个例子，我的领域模型中有一个 “person” 概念，我想把它改成 “contact” 。
这个词会出现在函数名、字段名、文档和测试用例中，简单的查找替换远远不够。
与其让 LLM 直接处理整个代码库，不如让它在识别到所有相关位置后，调用 IDE 自带的重构能力 —— 本质上是对 IDE 的各项功能进行统筹调度。
一位参会者提到，对 IDE 中重命名操作的分析显示，这类修改往往成批出现，因此这会是一项很实用的功能。

❄ ❄

双人披萨团队（小团队）会不会因为 LLM 不吃披萨而缩成单人披萨团队？
还是说我们会保留同样规模的团队，却能完成多得多的工作？
我更倾向于后者。
双人披萨团队的规模，在人类协作的收益与沟通协调的成本之间，确实达成了某种高效的平衡。

这也引出了一个关于结对编程形态的问题 —— 这个问题在我与 Gergely Orosz 以及 Kent Beck 在 The Pragmatic Summit 的圆桌讨论中也被提及。
目前一个普遍的看法是，最佳工作方式是一名程序员操控数个（甚至许多个）LLM 智能体。
但我在想，两个人共同操控一批智能体会不会效果更好？
既结合结对编程的优势，又能发挥 AI 强大的代码生成能力。

> - **two-pizza teams**：亚马逊著名的“两披萨团队”，指小到两张披萨就能喂饱的高效小团队
> - **one-pizza teams**：戏指单人团队


❄　❄　❄　❄　❄

Aruna Ranganathan 与 Xingqi Maggie Ye [在《哈佛商业评论》撰文](https://hbr.org/2026/02/ai-doesnt-reduce-work-it-intensifies-it) 写道：

> 我们针对一家约 200 名员工的美国科技企业，开展了为期八个月的研究，探讨生成式人工智能如何改变工作习惯。
结果发现，员工的工作节奏更快、承担的任务范围更广，且每日工作时长被进一步拉长，而这往往并非来自上级要求。

> ……

> 这一切听上去或许是管理者梦寐以求的状态，但对人工智能的热情投入所带来的这些改变，很可能难以持续，并在后续引发问题。
一旦尝鲜的兴奋感褪去，员工便会发现自己的工作量在无形中增加，疲于应付突然堆到面前的各项事务。
这种工作量隐性膨胀，进而会导致认知疲劳、职业倦怠与决策能力下降。
初期出现的生产力飙升，最终可能被工作质量下滑、人员流失等问题所取代。

❄　❄　❄　❄　❄

[Camille Fournier](https://bsky.app/profile/skamille.themanagerswrath.com/post/3mekddb6f4k27) ：

> 关于在 AI 时代 “人人都变成管理者” 这一点，我直到现在才真正意识到其中的问题 —— 多任务并行处理与频繁切换上下文带来的精神疲劳。
这无疑是做管理者最艰难的部分之一，而现在，所有人都要体验这份滋味了。

我们的行业正越来越明显地感受到一种转变：开发者将从亲自编写代码的程序员，转向管理一群智能体的监督型程序员。
我始终认为，无论是否承担监督角色，程序员依然要对自己监督下生成的代码负责。
目前尚无定论的是，日益频繁的上下文切换是否会削弱管理多个智能体的效率。
这会推动相关实践出现，目标是在充分利用智能体并行能力的同时，尽可能减少上下文切换。

无论我们走向哪条路径，预计未来几个月里，都会有大量探索涌现，研究如何为监督式编程构建高效的工作流程。
