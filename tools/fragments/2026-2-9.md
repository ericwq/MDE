# 片段：2月9日
Martin Fowler：2026年2月9日

上周围绕人工智能时代下软件开发未来的开放式空间研讨会上，我产生了一些进一步思考。
由于我们遵循 [查塔姆研究所规则 (Chatham House Rule)](https://www.chathamhouse.org/about-us/chatham-house-rule) ，文中未标注任何发言者来源；但若相关人士认出自己的观点并希望署名，可与我联系，我会编辑本文。

❄ ❄

研讨会开幕时，我表示，我天生对 LLM 的价值持怀疑态度。
毕竟，数十年来出现过许多号称要彻底改变软件开发本质的工具，其中绝大多数不过是狗皮膏药 (sname oil) 罢了。

但我是一个彻底、绝对的怀疑论者 —— 这意味着，我也必须怀疑自己的怀疑。

❄ ❄

我们其中一场讨论聚焦于认知债（cognitive debt）问题。
通常情况下，在构建软件系统时，开发者会逐步理解底层业务领域以及为支撑该领域所开发的软件本身。
但一旦大量工作交由 LLM 完成，这是否意味着团队学到的东西会变少？
如果真是这样，会带来哪些后果？
我们能否依赖 [精灵 (The Genie)](https://martinfowler.com/articles/who-is-llm.html) 记住所有事情，还是应该主动采取措施，确保团队更深刻地理解正在构建的内容及其背后的原因？

[TDD 流程](https://martinfowler.com/bliki/TestDrivenDevelopment.html) 包含一个关键（且常被忽视）的步骤：重构代码。
正是在这一步，开发者整合自身理解，并将其沉淀到代码库中。
我们是否需要类似的步骤，来确保自己明白 LLM 生成的内容？

当 LLM 写出复杂代码时，让它解释代码的运行原理。
或许可以用一种有趣的方式要求它解释，比如让它以童话故事的形式讲解代码行为。

❄ ❄

场外发言：

> LLM 就像毒贩，它们给我们提供东西，却完全不在乎最终形成的系统，也不在乎开发与使用它的人。

既然 LLM 每次交互都会刷新上下文，谁还会关心系统的长期健康状况？

❄ ❄

程序员对 LLM 持谨慎态度，不仅仅是因为担心工作受到威胁，更是因为我们害怕 LLM 会剥夺编程的大部分乐趣。
每当思考这点时，我都会反思自己究竟喜欢编程的什么。
一方面是交付实用的功能 —— 在我看来，随着 LLM 能力增强，这方面只会变得更好。

但对我而言，编程远不止于此。
我热爱的另一部分是建模（model building）。
我享受构建抽象概念的过程，这些抽象能帮助我理解代码所支撑的业务领域；
而我担心的是，LLM 会让我在这类建模工作上投入的注意力变少。
不过，建模反而有可能成为与 LLM 高效协作的重要环节 —— 这正是几个月前 [我与 Unmesh Joshi 探讨过的主题](../convo-llm-abstractions.md) 。

❄ ❄

在 LLM 时代，源代码这类事物还会存在吗？
如果存在，它会是什么样子？
提示词以及其他形式的自然语言上下文可以催生大量行为，并推动抽象层次提升，但 [同时也会转向非确定性](https://martinfowler.com/articles/2025-nature-abstraction.html) 。
在这一切中，是否仍然存在对非确定性行为的持久化描述的需求？

大约二十年前，我开始关注一类被称为 [语言工作台 (Language Workbenches)](https://martinfowler.com/articles/languageWorkbench.html) 的工具。
它们并未对软件开发产生重大影响，但或许 LLM 的兴起会重新引入其中的一些思想。
这类工具依赖一个 [语义模型(semantic model)](https://martinfowler.com/articles/languageWorkbench.html#workbench.gif) ，工具会将其持久化存储在某种介质中，而这种模型不一定是文本形式，也无法被人类直接理解。
相反，为了让人类理解，工具会提供投射式编辑器 (projectional editors)，从模型中生成人类可读的视图。

这种非人类可读、但具备确定性的表示形式，会不会成为未来的源代码？
一种以最少 token 实现最大表达能力为设计目标的形式？

❄ ❄

这门为类型理论 (type theory) 的高危实验而设计的语言，最终泄露到了普通开发者群体中。

❄　❄　❄　❄　❄

Matthias Kainer 撰写了一篇很酷的文章，[通过交互式示例讲解 Transformer 的工作原理](../so-whats-the-next-word-then.md) 。

> 上周二，我孩子放学回家，坐下就问：“ChatGPT 到底是怎么知道下一个词该说什么的？”  
我当时心想 —— 问得太好了。
就是时机太糟，因为晚饭马上就好，但问题本身真的很棒。

> 于是我试着解释。然后失败了。
不是因为难到无法理解，而是常规的解释要么是 “就是矩阵乘法而已”
（没错，但毫无用处），要么是 “它用了注意力机制 (attention mechanisms)”（名字很酷，信息量为零）。
这两种说法都帮不了一个 12 岁孩子。
说实话，也帮不了大多数成年人。

> 而且，我刚开口解释，耗时就比一条抖音还长，结果我还没说到 “矩阵乘法”，孩子就已经走神了。  
我需要更直观、更可交互、更有趣的东西。

> 所以这就是我希望当时晚饭时就能拿出来的版本：  
有图示，有可点击交互。
因为当一切都很抽象时，亲手摆弄真实数字往往能让人豁然开朗。

这份指南适合任何 12 岁孩子，也适合担心自己跟不上时代的 62 岁的人。
