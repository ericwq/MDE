- Unmesh

  我对如今全球范围内各种关于 LLM 在软件开发中作用的夸张说法，总感到有些反感。
为了梳理思路，我重新翻阅了敏捷早期的一些文章和论述。
我觉得 Fred Brooks 的经典之作 [没有银弹](https://www.cgl.ucsf.edu/Outreach/pc204/NoSilverBullet.html) 中的观点或许能带来启发。

- Martin

  我刚参加工作时，早期的导师就鼓励我去读 Fred Brooks 的 [人月神话](https://archive.org/details/MythicalManMonth/page/n9/mode/2up) 。
这本书的核心主题之一，就是概念完整性 (conceptual integrity) 的重要性。

  > 我认为大型编程项目会因分工而产生与小型项目截然不同的管理问题。
我认为最关键的需求，是维护产品本身的概念完整性。
  >
  > —— [Frederick P. Brooks, Jr](https://archive.org/details/MythicalManMonth/page/n9/mode/2up)

  不久之后，《没有银弹》一文发表，我们都认为这是迄今为止行业内最重要的文章之一。
<ins>他在文中对次要复杂度 (accidental complexity) 与本质复杂度 (essential complexity) 的区分，从那时起就一直萦绕在我脑海中</ins>。

  > 遵循亚里士多德 (Aristotle) 的思路，我将这些困难分为两类：本质困难 (nature) —— 软件自身特性中固有的难题；
次要困难 (accidental) —— 当下软件开发过程中伴随出现、但并非其固有属性的难题。
  >
  > 一个软件实体的本质，是由相互关联的概念构成的体系：数据集、数据项之间的关系、算法以及函数调用。
这种本质是抽象的，因为这一概念结构在诸多不同的实现形式中都是一致的。
尽管如此，它依然具备极高的精确性与丰富的细节。
  >
  > 我认为构建软件最困难的部分，是对这一概念体系进行规约、设计与测试，而非编写其实现代码、并验证实现是否准确的体力工作。
  >
  > —— [Frederick P. Brooks, Jr](https://www.cgl.ucsf.edu/Outreach/pc204/NoSilverBullet.html)

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  理解本质复杂度与次要复杂度的区别，能为我们在哪些场景运用 LLM 来简化问题提供清晰的指导。
</div></br>

- Unmesh

  我认为这种对次要复杂度与本质复杂度的区分非常实用。
  它能很好地帮助我们理解：系统本身的复杂度，与构建系统过程中的复杂度，二者并非同一回事。
  这种区分同样有助于我们思考 LLM 在软件开发中所扮演的角色。
  编程语言的语法、集成各类框架的繁琐问题、为了让软件正常运行所需的所有样板代码——这些都属于次要复杂度。
  <ins>在我看来，LLM 很擅长降低这类次要复杂度</ins>。
  比如需要一个对接 Kafka、并用 OpenTelemetry 记录日志的 Spring Boot 服务？
  LLM 几秒内就能生成可用的模板。
  但真正的软件开发工作，远不只是让代码编译通过那么简单。
  软件开发本身就是一个复杂的过程。
  <ins>尽管当下大家的注意力大多集中在利用 LLM 生成代码上，但我们更应该思考：“编写代码” 这一行为，其真正意义究竟是什么</ins>。

  编程绝不只是敲出计算机能理解并执行的语法代码，而是在构建解决方案。
  我们把问题拆解成聚焦的模块，将相关的数据与行为封装在一起，而至关重要的一步是 —— 选择能体现设计意图的命名。
  好的命名能穿透复杂度，让代码变成人人都能看懂的蓝图。
  <ins>最具创造性的工作，正是不断地精心命名，让这些名字清晰展现出解决方案的结构，并精准对应我们要解决的问题</ins>。

  然而，没有人在一开始就能看清完整的设计。
  进展源于一种往复迭代的节奏：稍作思考，编写一点代码，退后审视，再优化你所看到的内容。
  每一次迭代都会让代码更精炼，也让你对整体架构的理解更清晰，从而指引后续步骤。
  很多时候，正是 “编写代码” 这个过程，让设计决策得以最终成型。

  在软件领域中，“不可能性结果 (Impossibility Results)” 是让我们保持务实态度的一种方式。
  它们帮助我们在明确真实约束后聚焦于解决方案，而非假设约束不存在。
  如果我们能更深刻地理解这些 “不可能”，就能更好地把握构建软件系统的本质，以及通常所说的 “编码” 这项工作。

  我倾向于用一个我称之为 “前期规格不可能性”（Upfront Specification Impossibility, USI）的不可能性结果，来思考 “前期设计” 这一问题。

  USI 为软件设计划定了现实边界：最初的规约充其量只是一个假设。
  瀑布式开发试图对抗 USI，而敏捷开发则顺应它开展工作。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  编程不只是敲出计算机能够理解并执行的代码语法，更是在构建解决方案。
</div></br>

- Martin

  事实上，设计与架构是逐步演进的，这一理念正是敏捷软件开发方法的核心思想之一。
  而你所说的精心构建命名的过程，也不可避免地让我想到了 [DDD](https://martinfowler.com/bliki/DomainDrivenDesign.html) 中的 [通用语言 (Ubiquitous Language)](https://martinfowler.com/bliki/UbiquitousLanguage.html) 概念。
  本质上，你所表达的就是：我们与 LLM 的交互过程，正是在构建这种语言；
  而 Eric 也强调过，这一过程必须以迭代的方式完成。

- Unmesh

  或者，与其说我们在构建 (building) 一门语言，不如说我们在培育 (growing) 一门语言 —— 这正是借鉴了 Guy Steele 在 1998 年 OOPLSA 大会上主题演讲的标题。

  > 我的观点是：在这个时代，一名优秀的程序员绝不只是编写程序。
  优秀的程序员会打造一套实用的词汇体系。
  换句话说，优秀的程序员也在做语言设计工作 —— 尽管不是从零开始，而是在一门基础语言的框架之上进行构建。
  > 
  > —— [Guy Steele](https://www.youtube.com/watch?v=lw6TaiXzHAE)

- Martin

  对我而言，那是我听过最棒的技术演讲（我很幸运当时就在现场）。
  我强烈建议还没看过的人抽出一小时时间去观看。

  我听说 Lisp 开发者们常说，他们会把自己的开发环境塑造成一门专门用于表达业务领域的语言 —— 而这也正是我编程时一贯努力遵循的思路。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  编程，即是培育一门语言的过程。
</div></br>

- Unmesh

  看程序员们如何 “培育语言” 是件很有意思的事。
  这些抽象概念的词汇从何而来？
  一部分取自领域中已有的成熟概念，另一部分则是广为人知的设计模式，逐渐成为编程语言或开发范式社区中的常用表达。
  “我需要一个 Repository 来访问存储的数据”，“我需要一个 Controller 处理用户请求”，或是 “我需要用 Factory 创建对象”，
  这些都是编程社区中十分经典的抽象概念。
  值得关注的是，那些生命力持久的编程语言和开发范式，往往都具备打造活跃社区的巨大潜力。
  Ruby、Smalltalk、Lisp、Java，以及如今的 Python，都是绝佳的例子。

  根据我的经验，编程语言与编程范式能极大地引导和塑造程序员的思维过程。
  在构建数据管道时，函数式编程范式有助于我思考作用于数据流上的一系列转换逻辑链。
  而在开发 Web 应用或其他交互类应用时，面向对象编程范式则能帮助我以对象为单位进行思考，每个对象都封装了自身的状态与行为。

  编程还有另一个微妙却至关重要的方面：当我们编写代码、用代码的方式思考时，我们也在不断理解解决方案的结构。
  这个过程中会做出大量连贯的微观决策：这个逻辑应该独立成一个类吗？
  这个函数该如何命名才恰当？
  或许我该换一种方式来设计这个类的层次结构。
  <ins>所以整个过程包含两部分：做出决策，以及编写代码来落实这个决策。
  LLM 非常擅长针对非常明确的决策生成或修改代码</ins>。
  但要做出这些决策，却需要在代码中反复推敲、不断迭代。
  <ins>尽管开发者在这个过程中看似在 “浪费时间”，但这实际上是在特定编程语言中构建解决方案结构中的心智模型的关键环节。
  一旦这个心智模型清晰了，向 LLM 下达具体指令就会容易得多</ins>。
  正是这个过程让 “抽象” 得以固化 —— 而抽象，正是我们培育编程语言词汇体系的核心所在。

  很多时候，当我听到人们说 “一切都通过提示词来驱动” 时，我感觉这本质上是在做大量的前期设计。
  这很危险，因为如果不通过代码不断迭代，你就只能对设计结构做出大量猜测。
  更重要的是，你把构建代码抽象的决定权交给了 LLM 。
  当然，有人可能会反驳说，你可以审查每一处改动。
  但在相关决策已经被写成代码之后再做审查，效果往往不尽如人意。
  这也是我们更倾向于结对编程，而非被动审查的原因所在。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  在编写代码的同时，我们也在不断理解解决方案的结构。
</div></br>

- Martin

  你提出结对编程与代码评审的区别，这一点很有意思。
  尽管软件开发行业大多倾向于使用拉取请求和集成前代码评审，但我们属于少数派，更偏爱结对编程。
  我发现，如果你习惯了拉取请求模式，就会习惯于用这种 “这是我完成的工作 —— 请评审” 的方式与其他程序员协作。
  而结对编程的交互频率更高，角色切换也更灵活。
  此刻可能是你在评审我的代码，但十分钟后就换成我来评审你的。
  结对编程的核心是持续讨论，代码正是在这种交流中逐步成型的。

  因此在我看来，习惯拉取请求协作模式的程序员，与习惯结对编程的程序员，二者与 LLM 的交互方式自然会有所不同，这是合乎情理的。
  <ins>我们这些有结对编程经验的人，会很自然地选择与机器来回迭代、同步逐步完善思路，而非先敲定一个方案，再让 LLM 直接实现</ins>。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  编程包含两项深度交织的活动：发现抽象与应用抽象
</div></br>

- Unmesh

  我们对 LLM 使用方式的思考，同样取决于我们正在进行的编程工作类型。
  我认为编程包含两项紧密交织的工作：

  <ins>第一项是 发现并稳定抽象 (Discovering and Stabilizing Abstractions)。
  这本质上是一项创造性工作 —— 在这个环节中，我们探索多种方案、反复试验，并逐步收敛到稳定的抽象上。
  在此过程中，我们观察代码模式，提炼共性，识别未来可能发生变化的部分。
  随后选择合适的语言结构，对这些共性元素进行组织，使其能更轻松地应对未来的变化</ins>。

  <ins>一旦抽象趋于稳定——命名恰当、测试完备、易于理解 —— 我们就会应用这些稳定的抽象，来实现新的用例或变体</ins>。
  在 DDD 的术语中，这便是我们基于领域模型开展工作的阶段：领域模型是一组具名抽象，它们映射业务领域并承载精确的业务含义。

- Martin

  我们都称之为编程，但因为这是两种不同的活动，我们与 LLM 的协作方式也有所区别。
  一旦我们拥有了稳定的抽象，在其基础上生成代码就可以更偏向机械化的工作。
  我们已经定义好了相关 “语言”，LLM 也就知道该如何基于它来生成代码。

- Unmesh

  没错，因为此时模型有了具体的词汇体系可以参照，不必再去猜测术语的含义。
  像 “实现一个用于更新客户信息的 controller，以及一个针对客户的 repository” 这样的提示词，能为 LLM 提供清晰的上下文。
  最终生成的基本都是样板代码，可靠性很高。

  LLM 在开发其他偏机械化的工作环节中同样表现出色：

  - 搭建项目（初始迭代任务）
  - 创建构建流水线与 CI/CD 配置
  - 基于成熟编码模式进行微调并重复使用

  <ins>我们需要有选择地决定何时将工作交由 LLM 处理。
  在抽象发现阶段，开发者必须掌握主导权 —— 后续便能从中获益。
  一旦抽象趋于稳定，就可以在由抽象所确立的词汇体系指引下，把重复性、机械化的编码工作交给 LLM。
  这种方式能最大限度地发挥我们的创造力与生产力</ins>。

  即便如此，LLM 在抽象发现的过程中同样能提供极大帮助。

  <ins>开始编写代码时，快速查看多种方案往往有助于把握代码结构</ins>。
  LLM 能迅速将意图转化为代码。
  用通俗的英语描述你的需求，就能得到所选语言中简短且目标明确的代码片段。
  快速看到代码有助于你检验命名与边界、对比不同方案，同时完善抽象设计与整体解决方案。
  这种情况下我更倾向于以 “对话” 模式使用 LLM ，而非 “智能代理” 模式。
  这样我们可以尝试不同思路，感知代码结构，而不必担心破坏现有实现。
  我们可以有选择地决定修改代码的哪些部分，并进行非常细小的迭代调整。

  我经常和 LLM 一起使用的另一个非常实用的技巧是语言投影 (projections)。
  把同一段代码用另一种语言（Go、C++ 等）重写，是检验思路的绝佳方式。
  用不同语言呈现同一段逻辑，能帮我们快速验证多种实现方案。
  LLM 在这方面非常有用，你不必亲手敲写所有代码。
  更棒的是，即使是你并不熟悉的编程语言，也能直接得到对应代码。

  <ins>我发现这比费尽心思写提示词、只为得到精准想要的代码要有用得多。
  在寻找代码层抽象的过程中与 LLM 自然对话，不被任何特定规则束缚、不用刻意按某种格式写提示词，这种方式价值要大得多</ins>。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  编码时，快速查看多种替代方案很有帮助 —— LLM 非常擅长生成这些方案。
</div></br>

- Martin

  这让我想起上大学前我在 80 年代的一份工作，当时我因为在本地小型机上编译代码前没有仔细做三次检查而被训斥。
  他们告诉我，“正确的做法” 是让一页 Fortran 代码能 “一次编译通过”。
  可我发现，让编译器帮我跑几遍循环反而更快，因为它找错误比我自己找要高效得多。
  我多少能理解当时老板的想法 —— 在那个年代，交互式终端还算是新鲜事物，很多人都是从打孔卡时代过来的。
  但他们没意识到的是：一旦反馈回路变快，迭代就远比提交前反复检查更有效。

  那时候，我还因为 “过度依赖编译器” 而被批评。
  但随着经验不断增长，我越来越多地通过与计算机交互来探索抽象。
  编写代码时，我会留意自己已构建的抽象，是否能让我更轻松地表达所需的行为逻辑。
  如果发现某些逻辑难以用自己抽象所形成的 “语言” 清晰表述，那就意味着我需要进一步优化这些抽象。
  计算机在此扮演着至关重要的角色，让我保持客观清醒。
  如果不借助它运行编译、执行测试，我很容易自欺欺人，以为自己写的代码完全可行。
  计算机迫使我直面自己设计的抽象所存在的局限。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  如果我发现某些逻辑难以用我自己构建的抽象所形成的 “语言” 清晰表达，那就说明我需要进一步优化这些抽象。
</div></br>

- Unmesh

  <ins>关键在于，在这个过程中，我们无法预先完整确定这些抽象最终会是什么形态，也无法确定哪种语言特性或编程范式最适合表达它。
  抽象的形态只有在编写代码、逐步实践的过程中才会逐渐显现。
  开发者从具体代码中发现模式，并通过持续重构，慢慢收敛到一个稳定的结构</ins>。

- Martin

  <ins>LLM 似乎是人与计算机协作模式下很自然的进一步延伸，为我们对模糊想法快速获取反馈，提供了一种全新且高效的方式</ins>。

- Unmesh

  这个过程无法简化成给 LLM 的一段静态提示词。
  仅仅审查 LLM 生成的代码远远不够 —— 你会错失亲自编写代码时产生的深度思考。
  <ins>在这个阶段，LLM 最有价值的身份是头脑风暴伙伴：它们可以提供不同的设计思路或代码组织方式，但你应当避免让它们生成核心实现</ins>。
  这一阶段属于开发者本人。

  Richard Gabriel 将抽象 [定义为压缩](https://www.dreamsongs.com/WorseIsBetter.html) ——
  把已确立的完整含义编码成简洁形式，就像自然语言中的词语可以代表更宏大的概念。

  当我们在领域模型中谈及领域时，通常只想到业务领域（或称问题领域）—— 金融、零售、医疗等等。
  <ins>但在实际开发中，仅靠业务知识很少能构建出真正实用的领域抽象。
  始终存在另一个发挥作用的领域：解决方案领域</ins>。

  解决方案领域由我们选定的架构与技术方案所塑造：我们构建的是基于 Web 的系统还是事件驱动型系统；
  是依赖内存数据网格还是关系型数据库；
  集成模式采用的是同步 API 还是异步消息。
  <ins>这些选择并非次要细节 —— 它们直接决定了我们所创建的抽象形态</ins>。

  <ins>在大多数项目中，领域模型及其术语体系诞生于业务领域与解决方案领域的交汇之处</ins>。
  这也是为何仅专注于业务功能的领域专家无法独立开展工作。
  <ins>成功的团队会将具备深厚业务知识的人员与精通解决方案及技术的人员汇聚在一起，紧密协作，共同构建一套统一的语言</ins>。

  敏捷社区早已认识到这一点：良好的术语体系源于协作。
  我们构建解决方案、形成领域模型、创造术语体系的方式，并没有因为 LLM 的出现而改变 —— 这依旧需要刻意的投入与打磨。
  事实上，在使用 LLM 进行开发时，未能建立统一的领域术语体系所带来的代价会更为高昂。
  缺少这套体系，模型就没有具体的参照依据，只能猜测含义，生成的结果往往会偏离预期。
