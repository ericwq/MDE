大语言模型的基本工作原理，核心在于模式匹配与对下一个 token 的统计预测（即所谓“随机鹦鹉”）。
而从这一机制中涌现出的一个略显意外的能力是，它们在一定程度上也能对问题进行 “推理”。
不同模型的推理能力强弱各异，OpenAI 的 o1、o3 是两款代表性的推理模型，而 DeepSeek 的 R1 近期也引发了广泛关注。
但在我们将 AI 用于编程任务时，这种推理能力究竟扮演着怎样的角色？

剧透一下：我目前还没有确切答案！但我有一些疑问和思考。

我会先从两点讲起，在我看来，这两点是推理能力的局限，且与编程场景密切相关。
之后再分享我的看法：在哪些编程任务中推理能力能发挥作用，哪些则不能。

## 上下文至关重要，尤其对于推理而言
[苹果公司去年发表的一篇关于大语言模型推理局限性的论文](https://arxiv.org/pdf/2410.05229) 引发了广泛关注。
作者们提出了一个全新的基准测试，用于评估 LLM 在 "数学推理" 方面的能力水平。
该测试基于一个已有的、包含一系列小学数学题的基准数据集构建。
研究人员从中选取了 100 道题目，将它们改造成带有变量占位符的模板，再为每道题生成 50 种不同变式，最终得到了包含 5000 道题目的数据集。
第二步，他们还创建了另一组新数据集，在题目中加入了无关信息。

研究人员发现：

1. 人名和数字的变化会影响模型解题的表现。
因此，即使推理步骤完全相同
（比如从 “苏菲照看她的侄子” 变成 “安妮塔照看她的孙女”，或是毛绒玩具数量从 8 个换成 12 个），
模型的解题效果也并不稳定，甚至相比原始基准测试略有下降。

2. 当问题的难度和篇幅增加时，模型表现会进一步下滑。

3. 最后，研究人员发现，在题目中加入无关信息会对模型表现产生显著的负面影响。

首先，这是一个绝佳的例子，说明了我们应当对 LLM 的基准测试结果持保留态度。

而在编程领域，我认为最后一项发现尤其值得关注。
编码助手正越来越擅长为 LLM 梳理相关的代码上下文，让模型获得恰到好处的信息，从而为我们生成实用的代码。
<ins>这篇论文表明，要实现良好的推理效果，工具能否过滤掉无关代码片段尤为重要——这些内容会分散模型注意力，进而降低推理质量</ins>。

总而言之：<ins>数据质量至关重要，对推理任务而言，其重要性甚至超过其他类型的任务</ins>。
当我们遇到需要多步推理的问题时，尤其需要谨慎把控输入给模型的上下文内容。

## 推理阶段不执行函数调用

像 o1 和 R1 这类推理模型分为两个阶段工作：首先它们会对用户的提示进行 “推理” 或 “思考”，然后在第二阶段返回最终结果。
在推理阶段，模型会通过思维链逐步得出结论。
你能否完整看到该推理阶段的内容，取决于模型前端的用户界面。
例如，OpenAI 仅向用户展示每一步的摘要，而 DeepSeek 平台则会展示完整的推理链（当然，如果你自行运行 R1，也可以获取完整推理链）。
推理阶段结束时，聊天机器人界面通常会显示 “思考耗时 36 秒” 或 “推理耗时 6 秒” 之类的提示。
无论耗时长短、用户是否可见，后台都会持续生成 token ，因为 LLM 正是通过 token 生成完成思考过程。

据我观察，在这种推理模式下，模型无法调用自身可用的任何函数。
我不确定这是受 API 限制还是模型本身限制所致，
但推测原因是：推理阶段的核心目的是生成连贯不间断的 token 链，而函数调用会打断这一过程，违背该阶段的部分设计初衷。

## 推理能力在编码任务中何时适用？
许多推理基准测试采用小学数学题，因此当我在软件开发中寻找能借助思维链解决的类似问题时，这些数学题便是我的参考依据。
在我看来，这类问题需要多步骤推导才能得出答案，且每一步的执行都依赖上一步的输出结果。

根据这一定义，推理能力对于以下编码任务而言完全无关紧要，因为 LLM 无需分步推理即可完成：
- 行内代码补全建议
- 为新函数生成代码
- 解答具体操作类问题（例如 “如何删除所有本地 Docker 镜像？” ）
- 生成文档或内容摘要
- 将自然语言转换为形式化查询语句（如SQL）

这些正是专为编码优化微调的模型所擅长的领域，与其推理能力强弱无关。

推理模型的应用主要集中在以下两种场景：

### 调试
调试似乎是思维链的绝佳应用场景。
我主要困惑的是，调试过程中对推理能力的运用，会在多大程度上因无法调用函数而受到限制。

我们以一项调试任务为例：程序出现报错，我希望 AI 分析潜在原因。
此时我自己完全不清楚问题源头，也无法将整个代码库放入上下文窗口，因此我实际需要 AI 工具在推理过程中主动查询相关信息。

举个例子，假设日志中出现这样一条错误：“ERROR：duplicate key value violates unique constraint 'users_email_key' ” ，我让 AI 协助调试。
AI 首先可能提出的假设之一是：邮箱校验逻辑失效，这时它就需要去查看我的校验代码，再继续推理。
这是调试中非常典型的流程：先提出假设，再查询信息以验证或推翻该假设，随后要么推进下一步，要么彻底放弃该假设。
但据我对大语言模型推理阶段的理解，目前在推理过程中通过函数调用查询更多信息是无法实现的。

### 制定实施方案
我看到大量反馈显示，许多人会先用推理模型完成编码中的规划步骤，再用擅长代码生成的模型（如 Claude Sonnet）来执行方案落地。
热门开源编码助手 [Cline](https://github.com/cline/cline) 近期新增了一项功能，开发者可在 “Plan” 与 “Act” 模式间切换 —— 既能避免大语言模型直接进入代码实现环节，也方便在第一步使用具备推理能力的其他模型。

我也曾几次尝试用推理模型做方案规划，但说实话，目前并未感觉到它和优秀的代码专用模型有明显区别。
这让我不禁思考：编码任务的规划真的属于需要思维链的问题吗？
像 “我们需要一个 API 接口，并修改这个网页组件” 这类内容，究竟需要模型一步步推理，还是仅靠更基础的模式匹配能力就能完成？

目前我还没有进行足够多的使用，无法就此得出结论。
但我上文提到的那篇论文中的第二项研究结果，在此处或许同样适用：当作者们加大数学题的规模与难度时，
“……得出准确答案的概率会随着涉及的令牌数或步骤数呈指数级下降”，
“……难度越高，模型表现越差，结果的波动性也越大”。
在该论文中，“难度” 的定义是为原始问题增加更多变量与条件。

这些研究结果，是否为具备强推理能力的模型所能规划和解决的问题的规模与难度设定了上限？
我们或许会发现，o1 和 R1 在编码领域发挥的作用，并没有目前被宣传的那么重要。
举个例子，[Qodo 团队的研究](https://www.qodo.ai/blog/system-2-thinking-alphacodium-outperforms-direct-prompting-of-openai-o1/) 发现，分步式智能体流程的表现，能优于单独使用推理模型。
如果问题的规模与复杂度确实会成为这类模型的限制因素，那么想要让人工智能更擅长解决问题，我们就必须借助某种智能体将问题拆解，再把更小的子问题交给多个模型处理。
如此一来，我不禁思考，推理能力是否只对其中一小部分子问题而言是绝对必要的。

## 总结
我个人目前还不确定，推理模型对于软件开发任务而言，是否真如宣传那般具有颠覆性的变革意义。
这类模型除了速度慢、成本高之外，还存在上述潜在局限性。
而且在我多次将其用于编码或调试问题的尝试中，尚未出现过一种情况：在配备优秀 AI 上下文编排能力的 IDE 里，o1 模型给出的结果明显优于 Claude-Sonnet-3.5 。
