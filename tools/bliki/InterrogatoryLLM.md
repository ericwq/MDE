当我们需要让一个 LLM 执行复杂任务时，通常需要向它提供大量上下文。
为某个新功能进行设计，就需要描述我们希望该功能在用户面前如何呈现、实现的指导原则、需要查阅的外部系统信息等等。
所有这些内容可能是长达数页的 markdown。
最直接的做法是由人来编写这些上下文，但另一种方式是让 LLM 通过 “采访” 人类来撰写这些上下文。

具体做法是：提示 LLM 来询问我。
它应该问出所有为了生成这份合适上下文所需要的问题。
我可以提供它所需的大部分信息，并告诉它如果某些信息自己无法获取，该去查阅哪些其他来源。
完成后，它就可以生成一份上下文报告，供另一轮对话（可能是与另一个模型）执行下一步使用。

我最早是在 Harper Reed 的博客上看到对这种方法的详细描述。
他方法中一个引人注目的要素是坚持让 LLM 一次只问一个问题。
（我尝试时，发现需要频繁提醒它这一点。）

使用 “询问式LLM” 的另一种方式是：给它一份文档（例如一份记录某个领域知识的软件规格说明），然后让 LLM 采访人类专家，以判断该文档是否准确。
这可以替代让人类专家直接阅读文档进行评审的方式。
人们通常觉得评审工作很困难，因此与 LLM 进行对话可能会更有成效，尤其是在文档写得不够好的情况下。

当然，我们可以将这两种方式结合起来 —— 先使用一个询问式 LLM 来构建文档，然后再用其他询问式 LLM 与其他专家一起进行评审。

上述内容是利用 LLM 为 LLM 的某个特定用途创建或评估上下文。
但这项技术有着更广泛的应用场景。
我天生是一个写作者 —— 我发现写作过程是思考中必不可少的一部分。
要真正理解某件事，我需要把它写下来。
但人与人不同，许多人觉得写作很困难，有时甚至极其困难。
当我们需要把一个人头脑中的信息提取出来、转化成其他人可以阅读的形式时，这可能成为一个真正的难题。
也许这些人会发现，让 LLM 采访自己比亲自撰写文档更容易。
当然，结果会带有那种令我这样的人不寒而栗的 “AI写作” 的味道 —— 但这总比因为匆忙写作或根本不写而导致信息缺失要好。
