Vibe coding 是指通过向 LLM 发出提示来构建软件应用 ——告诉它要构建什么，试用一下，再提示它进行修改—— 但整个过程完全不看 LLM 生成的任何代码。
这种技术即使没有任何编程知识的人也可以使用。
然而，这样生成的软件通常在可维护性、正确性和安全性方面存在问题 —— 因此最适合用于面向有限受众的、一次性的软件。

该术语由经验丰富的程序员 Andrej Karpathy 于 2025 年 2 月在 X 平台上首次提出。

> 有一种我称之为 “氛围编程” 的新型编程方式，你会完全沉浸在氛围中，拥抱指数级的可能性，甚至忘记代码的存在。
之所以能做到这一点，是因为 LLM（例如 Cursor Composer + Sonnet）变得过于强大了。
而且我只用 SuperWhisper 跟 Composer 对话，几乎不怎么碰键盘。
我会提出一些最愚蠢的要求，比如 “把侧边栏的内边距减少一半”，因为我懒得自己去找到那个地方。
我总是 “全部接受”，不再看代码差异了。
当遇到错误消息时，我直接复制粘贴进去，不加任何评论，通常问题就解决了。
代码已经增长到超出我通常的理解范围，我得花上好一会儿才能真正读懂它。
有时候 LLM 无法修复某个错误，我就绕过它，或者要求随机改动直到错误消失。
对于一次性周末项目来说，这还不算太糟，而且相当有趣。
我确实在构建一个项目或 Web 应用，但这并不是真正的编程 —— 我只是看到东西、说点东西、运行东西、复制粘贴东西，然后它基本上就能跑起来。
>
> —— Andrej Karpathy

关于 vibe coding，关键在于 “忘记代码的存在”。
这正是它既有用又受限的原因所在。

自 [11 月的转折点](./NovemberInflection.md) 以来，许多程序员开始让 LLM 编写他们的所有代码，并表示自己可能再也不会直接写一行代码了。
然而，他们确实在意这些代码 —— 会进行审查，关注其内部结构。
在这种情况下，他们并没有忘记代码的存在，所以这实际上是另一种不同的东西，我称之为 [Agentic Programming](https://martinfowler.com/bliki/AgenticProgramming.html) 。
遗憾的是，“vibe coding” 这个术语确实流行起来了，所以很多人用它来指代 agentic programming。
但我认为，尽管 [语义已经快速扩散](https://martinfowler.com/bliki/SemanticDiffusion.html) ，仍然值得尝试将 vibe coding 和 agentic programming 这两个概念区分开来，因为它们在用法和结果上都是不同的。

由于氛围编程者不看代码，他们不需要编程技能，因此对于没有编程知识的人来说，这是构建自己所用应用程序的完美方式。
经验丰富的程序员也可能会发现它在快速开发一次性软件或原型时非常方便。

氛围编程仍然很新，我们还在探索它的局限性，而这些局限性会随着模型及其框架的成熟度变化而改变。
这些局限性确实引入了相当大的风险，特别是如果通过氛围编程生成的软件被广泛使用或能够访问敏感信息时。

也许最严重的风险是安全性问题。
LLM 本质上存在漏洞，因为它们为攻击者提供了巨大的攻击面。
通过氛围编程生成的应用程序常常可能暴露敏感信息，甚至更糟 —— 暴露凭证，从而让攻击者深入组织系统内部。
即使是非程序员也需要意识到 [致命三重威胁](https://martinfowler.com/articles/agentic-ai-security.html#lethal-trifecta) 。

由于很少关注代码，氛围编程生成的软件可以快速产生大量质量极低的代码。
这样的代码使得即使是 LLM 也难以在未来修改和增强该软件。
尽管不断增长的 LLM 能力可能会让它能够处理即使是最庞大的 “意大利面条式” 软件，但到目前为止，似乎结构良好的软件同样能让 LLM 的工作更轻松。

LLM 以习惯性地捏造错误事实并以极大的信心呈现这些事实而闻名。
这种习惯也会导致它们创建行为不正确的软件 —— 而这些错误可能对用户来说并不明显。
此外，LLM 的非确定性意味着，要求 LLM 增强某个软件很容易导致它引入错误，甚至是在那些由于新需求而不应该更改的代码部分。
因此，我们应该以怀疑的态度对待 LLM 生成的软件，它仍然可能有用，但我们需要意识到其中的风险。

总体而言，<ins>氛围编程生成的软件最适合用于一次性软件，仅供其作者或一小群了解并接受相关风险的协作者使用</ins>。
更复杂、使用更广泛、且风险后果更严重的代码，则不应当被遗忘。
