目前在 Thoughtworks 内部，使用最广泛的编码辅助方式是 IDE 中的行内代码生成：
开发者在 IDE 中编写代码时，IDE 插件会实时给出代码建议。

对于 “这有用吗？” 这个问题，简单的回答是：“有时候有用，有时候没用。” ¯\_(ツ)_/¯
你在网上能看到开发者们五花八门的观点，从 “这让我的效率大幅提升”，到 “我直接关掉了，毫无用处”，各种评价都有。
这是因为这类工具的实用性取决于具体场景，而对 “有用” 的判断，也取决于你的期望有多高。

## 我说的 “有用” 具体指什么？
在本文中，我对 “有用” 的定义是：生成的建议能帮我更快地解决问题，且质量与不使用该工具时相当。
这不仅包括代码编写环节，还涵盖对生成内容的审核、调整，以及后续若出现质量问题时所需的返工处理。

## 影响(代码)建议实用性的因素

注：本文主要基于使用 GitHub Copilot 的相关经验。

### 技术栈普及度
**更稳妥的场景** ：技术栈越普及，模型训练数据中包含的相关讨论和代码示例就越多。
这意味着，对于 Java、JavaScript 这类语言，生成的建议往往比 Lua 这类较新或讨论较少的语言更实用。

**然而** ：我的同事 Erik Doernenburg 曾记录过他使用 Rust [让 Copilot 处理复杂场景](https://erik.doernenburg.com/2023/06/taking-copilot-to-difficult-terrain/) 的经验。
他的结论是：“但总体而言，即便使用 Rust 这种不那么常见的编程语言，且代码库用到更复杂的数据结构，我依然觉得 Copilot 很有帮助。”

### 更简单、更常见的问题
**更稳妥的场景**：这一点有点难以明确定义。
“更简单” 指什么？“更常见” 又指什么？
我会用几个例子来加以说明。

- *常见问题* ：
在上一份备忘录中，我以生成中位数函数为例进行了讨论。
我认为这是一个非常常见的问题，因此很适合使用代码生成功能。

- *适配我们业务场景的通用解决方案模式*：
例如，我曾成功用它实现需要列表处理的需求，比如对列表进行一连串的映射、分组和排序操作。

- *样板代码*：
生成各类样板代码，如 ExpressJS 服务、React 组件，或是数据库连接与查询执行代码。

- *重复模式* ：
对于存在高度通用、重复模式的代码编写工作，它能显著提升效率，比如新建构造函数或数据结构，或是在测试套件中重复编写测试初始化代码。
这类工作我以往通常大量复制粘贴，而 Copilot 可以加快这一过程。

一位已经使用 Copilot 两个多月的同事，在和一位还没有订阅权限的人结对编程时，“觉得要手动写重复代码简直难以忍受”。
不过，对于那些本身就非常擅长使用 IDE 功能、快捷键以及多光标模式等技巧的开发者来说，
这种 “强化版自动补全” 的效果可能没那么显著。
同时也要注意：当编码辅助工具降低了重复代码带来的不适感时，我们进行重构的动力可能也会随之减弱。

**然而**，在着手处理更复杂的问题初期，你依然可以借助编码辅助工具来探索一些思路，即便之后并不采纳它给出的建议。

### 建议代码片段更短小
**更稳妥的场景**：
生成的代码片段越短小，所需的审核工作量就越少，开发者也越容易理解建议的内容。

代码片段越长，你花费在理解它上面的时间就越多，也越有可能需要修改以适配你的实际场景。
过长的代码片段还会诱使我们一次性大幅推进开发，这会增加遗漏测试覆盖范围、引入冗余内容的风险。

**然而**：我认为这个因素与其他因素存在很多相互影响。
当你已经有了解决问题的思路时，小步生成的方式尤其有帮助。
因此，若你因经验不足或问题较为复杂而尚未形成解决方案，那么更长的代码片段或许能帮你开启思路、制定计划。

### 开发者经验更丰富
更稳妥的场景：经验依然至关重要。开发者经验越丰富，就越能判断代码建议的质量，也越能高效地使用这些建议。正如 GitHub 官方所言：“它很擅长处理你已经遗忘的知识点。”甚至有研究发现：“在某些场景下，初级开发者使用这类工具完成任务的耗时，反而比不使用时多出 7% 到 10%。”

然而：我目前收集到的大多数观察结论，均来自经验较为丰富的开发者。因此在这一点上，我目前还无法完全确定其中的利弊权衡。我的假设是：若前文提到的其他因素都处于更稳妥的状态，这类工具就越不容易引导初级开发者走入误区，同时越有可能为他们提供助力。结对编程以及其他形式的代码评审，能进一步降低相关风险。

### 开发者经验更丰富
**更稳妥的场景**：经验依然至关重要。
开发者经验越丰富，就越能判断代码建议的质量，也越能高效地使用这些建议。
正如 GitHub 官方所言：“它很擅长处理你已经遗忘的知识点。”
甚至有研究发现：“在某些场景下，初级开发者使用这类工具完成任务的耗时，反而比不使用时多出 7% 到 10%。”

**然而**：我目前收集到的大多数观察结论，均来自经验较为丰富的开发者。
因此在这一点上，我目前还无法完全确定其中的利弊权衡。
我的假设是：若前文提到的其他因素都处于更稳妥的状态，这类工具就越不容易引导初级开发者走入误区，同时越有可能为他们提供助力。
结对编程以及其他形式的代码评审，能进一步降低相关风险。

### 更高的错误容错成本
我已经提到过判断代码建议质量与正确性的重要性。
正如广泛报道的那样，大语言模型可能会出现信息 “幻觉”，在代码场景中同样如此。
当你处理的问题或用例一旦出错会造成严重影响时，就需要格外谨慎地审核生成的建议。
例如，我最近在处理 Web 应用中的 Cooki e安全相关工作时，Copilot 为 HTTP 响应头`Content-Security-Policy`给出了一个参数值。
由于我在该领域经验不足，且这属于安全相关场景，我没有直接采纳 Copilot 的建议，而是查阅了可靠的线上资源进行研究。

## 总结
代码辅助工具有其更适用的场景，但从本次讨论中可以看出，有多种因素在共同作用、相互影响，决定了工具的实用性。
高效使用代码辅助工具是一项技能，无法简单通过培训课程或博客文章学会。
重要的是使用一段时间，在适用和不适用的场景中都进行尝试，逐渐摸索出这类工具对你而言何时有用、何时应该放下工具自行编写代码。
