什么是代码？
在宏观层面上，这个问题的答案似乎显而易见。
代码就是开发者编写的内容：用编程语言表达的、告诉机器该做什么的指令。
多年来，编写代码意味着逐字逐句地敲出来。
效率的衡量标准是代码能够被多快地编写、编译、测试和部署。
借助现代 LLM，我们不再需要输入每一个单词来生成代码。
大量可执行的代码现在可以从高层描述中生成出来。
这迫使我们思考一个更深层的问题：如果生产代码变得更廉价，那么代码的什么价值依然存在？

## 代码的两个方面
代码始终服务于两个不同但又相互交织的目的。

首先，代码是一组 **给机器的指令** 。
它指导计算、移动数据、与存储交互并协调执行。
在 LLM 时代，这部分正在被商品化。

其次，代码是 **问题领域的一个概念模型** 。
这是 “设计” 方面。
一个设计良好的代码库不仅包含给机器的指令，也包含供人类和工具进行推理的概念。

我们称之为 “编码” 的活动，正是这两个方面相遇的地方。
我们正在塑造概念、命名、边界和关系 —— 通过这些，系统被理解。

## 概念模型与词汇
将概念模型显式化是编码中更深层次的方面，它由领域以及系统所要解决的用例所驱动。
每个领域都有其既定的流程、实践，以及更重要的 —— 一个共享的词汇表。
正是这个词汇表，让概念模型变得可见。
*<ins>「译注：让我想到了 DDD 中的 Ubiquitous Language」</ins>*

词汇通常被理解为特定语言或学科中使用的词语集合。
我能够写这篇文章，是因为我懂英语的词汇。
读者能够阅读它，是因为他们与我共享这些词汇。

但要理解这篇文章，仅懂英语是不够的。
这篇文章是关于软件开发的。
软件开发是一个广阔而成熟的领域，拥有自己的技术词汇。
当我使用像 “abstraction” 这样的词时，我不仅仅是在使用一个英语单词。
我指的是一个特定的软件开发概念，它有自己的含义、历史背景和潜在影响。
一个不熟悉软件开发的读者，可能只是在表层理解这个词，却会错过它在当前语境下的深层含义。
这些拥有自身成熟词汇的领域，就被称为 “领域”。

所有严肃的领域都是如此。
沟通依赖于共享的词汇。
无论我们是在与一个人、一个框架还是一个 LLM 沟通，我们使用的词语必须映射到接收者能够理解并据此行动的概念上。

## 代码中的词汇
一个设计良好的代码库是特定词汇的一种表现形式。
代码中的这种词汇从何而来？
这正是软件开发独特本质的真正闪光之处。
软件开发工作于多个领域的交汇点上。
一端是诸如银行、金融、零售、库存、医疗、保险等领域。
另一端则是像 Web、基础设施、AI、数据工程等领域。

从事 Web 开发的人需要扎实掌握 Web 架构、Web 方法的语义、GET 的通用缓存潜力，以及这些语义所带来的影响。
不懂这些的人，无法很好地架构复杂的系统。
在其他领域也是如此。
<ins>词汇不仅仅是一组标签。
它承载着含义、约束以及设计上的后果</ins>。

以零售领域为例。
当我们为零售编写代码时，我们会谈论客户、产品、订单、发货、支付等概念。
当我们为零售领域进行 Web 开发时，代码中包含将零售领域映射到 Web 领域的概念。
例如，Catalog 是一个资源，我们可以使用 GET、POST、PUT、DELETE 等 HTTP 方法对其进行操作。
编写代码的人需要同时熟悉这两种词汇。

为某个领域编码，从根本上说是一种翻译行为。
开发者将领域词汇映射到技术领域的词汇上。
在这个过程中，还会利用编程语言提供的构造来构建一个新的词汇表。
其中有像日志、仓库、法定人数、事务这样的概念，也有像货币这样的特定概念。
<ins>概念成为类型，关系成为接口，规则成为不变量，工作流成为组合</ins>。

变量名的精确选择、方法的边界、类的层级，都是逐步发现的。
正确的抽象往往不是一开始就显而易见的；
它只有在您根据现实世界的约束不断塑造和重构代码时才会显现出来。
当运用得当时，代码会慢慢变得可读，成为领域本身的高度具体的表现形式。

对于技术领域，我们通常会找到提供基础实现模式的框架和库。
框架和库是已编码的词汇表。
它们捕获了最常见的用法模式。
这就是为什么会有像 Spring Framework 这样的生态系统，用于构建涉及 Web、集成及相关关注点的企业应用。
不同的编程语言带来各自的风格，以及其框架和库中所反映和编码的特定设计约束。

## 限界上下文与本地词汇表
当一个领域具有稳定、重复出现的结构且拥有广泛共享的语义时，框架能很好地工作。
但像 “在线零售” 或 “证券交易所” 这样的领域则不同。
它们不仅仅是技术栈。
之所以不存在通用的高层框架，主要原因在于这些领域词汇在所有实例中并不足够稳定。
试图寻找通用抽象，要么会变得过于泛化而失去用处，要么会变得过于固执己见而无法广泛适用。
你越是接近核心业务模型，就越必须在本地发现这些抽象。
这就是领域驱动设计中 [限界上下文 (bounded context)](https://martinfowler.com/bliki/BoundedContext.html) 概念如此重要的原因。
一个限界上下文标记了特定词汇和模型有效（valid）的边界。
相同的词语在不同的上下文中可能意味着不同的事物，每个上下文都需要自己的抽象、规则和语言。

我们如何构建这些本地抽象和词汇表？
这类词汇很大程度上是通过迭代式的会话构建的：我们编写代码，然后对其进行反思。
<ins>像 TDD 这样的技术，对于这种词汇的迭代式开发来说非常出色。
它们通过迫使模型与其行为之间产生持续的反馈，帮助我们找到正确的命名、正确的抽象和正确的边界</ins>。

编码不能孤立地进行。
领域专家、用户和开发者之间必须紧密协作。
这种协作对于构建这些本地抽象和词汇表是必不可少的。

这与敏捷软件开发的教训直接相关。
强调个体与互动、客户协作、可工作的软件以及响应变化，不仅仅是流程上的建议。
它是一种通过反馈来发现和完善词汇的方式。

领域驱动设计通过 [ubiquitous language](https://martinfowler.com/bliki/UbiquitousLanguage.html) 的概念，将这一点变得更加明确：
它是一种由开发者和领域专家共同开发、并持续通过可工作的软件进行测试的共享语言。

## 编程语言作为思考工具
通过代码构建词汇，需要积极参与编写和重塑代码，而不仅仅是被动地审查生成的代码。
深度思考代码的行为，往往只发生在我们积极编写代码的时候。
<ins>编程语言及其构造和约束本身，就成为了思考工具。
不同编程语言提供的设计约束，帮助塑造我们的思维</ins>。
Go 语言的通道和轻量级线程、Java 的面向对象模型、或是 Rust 的 ownership 模型，都以特定的方式促使我们去看待结构、边界和权衡。
<ins>从这个意义上说，编程语言不仅帮助我们表达一个设计，也帮助我们发现它</ins>。
我最近不得不为一个异步编程示例设计一个自定义的 Future 实现。
Future API 的一个重要方面是设计组合方式，以便能够表达一系列的操作。

```java
         var future1 = action1();
         future1.thenCompose(val1 -> action2(val1))
                .thenCompose(val2 -> action3(val2))
```

理解函数式编程的概念和词汇，对于良好地实现这个 API 至关重要。
不了解这些概念，就会导致别扭的实现和使用。

有时，编程语言的语法会变得过于冗长，掩盖了解决方案的底层结构。
例如，最近在为我的研讨会实现快照隔离（snapshot isolation）时，用简单的英语描述基本需求有点模糊，而用 Java 代码实现又太冗长。
像 TLA+ 这样约束性更强的形式化规约本来会有所帮助。
但即便如此，仅写了一页准形式化的伪规约，就已经带来了显著帮助。

```
         Begin(T, coord):
           R(T) := HLC(coord).now()
           writeSet(T) := {}


         Read(T, N, key):
             N.HLC.tick(R(T)) //HLC advanced. So any write or commit after this is guaranteed to be at a higher ts
             return latest committed version of key with ts <= R(T)


         Write(T, N, key, value):
             N.HLC.tick(R(T))
             if LatestCommittedVersion(key).ts > R(T):
                 abort T
             place provisional intent for (key, value)
             writeSet(T) := writeSet(T) union {key}
```

这份准形式化的规约帮助澄清了我的思路，并为进一步的讨论、实现以及通过测试进行验证提供了良好的基础。

## 与 LLM 协作
<ins>将 “编码视为词汇构建” 这一观点，对于 LLM 有着重要的启示</ins>。
LLM 是在大量文本和代码的词汇上训练出来的。
它们学习到名称、API、库、框架、惯用法、设计模式与实现结构之间反复出现的关联。
当它们看到像 Controller、Repository、Reducer、ConsensusModule 或 TransactionLog 这样的词时，这些名称不仅仅是标签。
它们携带着与已知代码结构和预期行为相关联的信息。

<ins>这就是为什么在与 LLM 协作时，词汇如此重要</ins>。
如果我们使用的提示词模糊或不一致，模型就不得不猜测我们意图的设计。
如果我们的代码库使用了不清晰的命名和不一致的概念，模型就几乎没有稳定的结构可循。
但是，当词汇精确、一致，并且体现在代码中时，LLM 就能更可靠地将我们的意图映射到有用的实现上。

## 认知债务
这也解释了 LLM 辅助编码的一个特定危险：[认知债务 (cognitive debt)](https://arxiv.org/abs/2603.22106) 。
当人们在使用词汇、抽象和结构时，并没有很好地理解它们的含义，认知债务就随之积累起来了。
LLM 会放大这种风险，因为它们能够极快地生成大量看似合理的代码。
生成的代码中可能包含 controllers、repositories、reducers、factories、transactions、schedulers 或其他看起来熟悉的结构。
代码可能能够编译通过，甚至可能通过基本的测试。
但是，如果团队不理解这些结构背后的概念模型，那么代码库就在没有共享理解的情况下，增加了词汇量。
问题不在于 LLM 生成了代码。
<ins>问题在于，代码引入词汇的速度，超过了开发者建立理解的速度。
这个差距，就是认知债务的主要来源之一</ins>。

## 代码作为共享的概念模型
正如我们在 [使用 LLM 设计抽象](./convo-llm-abstractions.md) 中讨论的那样，编写代码包含两个深度交织的活动：
发现抽象和应用抽象。
发现抽象的过程，就是我们发展词汇的过程。
<ins>一旦建立起强大的词汇表，它就代表了一个共享的概念模型。
这个模型存在之后，编码的大部分工作就变成了使用这个概念模型来构建用例</ins>。
这就是好的库和好的基础代码大放异彩的地方。
在这个部分，我们试图隐藏编程语言和环境的复杂性，并为我们所构建的词汇提供一个越来越接近英语的接口。
一个典型的实现方式是构建一个 DSL ，以便于使用这些词汇或抽象，使其接近自然语言。

LLM 在这方面表现出色。
它们为你的抽象词汇提供了一个自然语言接口。
最棒的是，如果你的抽象词汇背后有可执行的代码，那么这本身就能成为 LLM 修正自身错误的绝佳护栏。
良好的抽象、可执行的行为、测试、类型和不变量，都有助于约束模型，使其输出更有用。

我们可以利用这些词汇来构建一个外部 DSL。
LLM 作为外部 DSL 之上的自然语言接口，工作得非常好。
我一直在使用 LLM 配合 PlantUML 等工具，并取得了巨大成功。
这并不意外，因为 LLM 的设计初衷就是最擅长映射词汇。

从这个意义上说，在 LLM 时代，强大的基础代码变得甚至更加重要。
一旦一个系统的词汇表成型，编码就不再那么侧重于生成原始的语法，而更多地是使用一种发展完善的概念语言来构建可靠的软件。

## 代码本身作为工具链与上下文
许多关于上下文工程和工具链 (harness) 工程的讨论，都将代码视为一个黑盒，其生成正确性的责任被外部地管理：
通过在提示词中提供正确的上下文，或通过构建带有规约、测试、静态验证等的正确工具链，来确保代码结构合理并按预期工作。
然而，一个结构良好、其抽象构成了明确定义的词汇表的代码，其本身就能作为工具链和上下文中最重要的部分。
我屡次看到，设计良好的代码与 LLM 协作能取得很好的成功。
更重要的是，当代码建立在具有清晰语义的稳定抽象之上时，你就能获得一定的自由度去选择任何 LLM 模型，并且无需过度担心提示词的精确度问题。
代码结构和附带的测试本身就提供了上下文和工具链，使得 LLM 的输出变得可靠且有用。

## 结论
编码的角色并未消失，但它正在改变。

随着 LLM 使代码生成变得更廉价，编写指令这种机械行为变得不再那么核心。
更重要的转变为：使概念模型显式化、发现正确的词汇、并通过迭代、领域专业知识和反馈来精炼这些词汇。
这也是为什么编程语言仍然至关重要。
我们不应该成为生成代码的被动审查者。
编写代码的行为本身就是我们思考的一部分。

代码仍然是给机器的指令。
但它也是理解的模型。
在 LLM 时代，这第二个角色变得甚至更加重要。
编码的未来不仅仅是更快地编写更多代码，而是构建更好的概念模型、更好的词汇表、以及更好的基础 —— 让人类和 LLM 都能够在这些基础之上协同工作。
