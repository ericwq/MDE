## 在模型偏离正轨时将其停止——但要自动化
前几天我尝试为 Pi（一个编码智能体）开发一个扩展功能。
我希望在模型偏离正轨时能够将其停止。

我现在有一个很快的本地模型，可以调用工具（搜索、运行测试、编辑代码等）。
然而，它确实经常出现一些模型辅助编程的怪异行为：

- 将能正常工作的生产代码替换成抛出异常的版本
- 在测试中写入 if 语句
- 为不可能失败的事情添加回退逻辑
- 在能够正常工作的代码（通过了测试及其他检查、对用户来说也正常运作）中找出 “问题”

长期的解决方案可能在于小步前进。但这些步骤来自经验。

通过在问题发生时简单地匹配某些关键词来捕捉问题，可以作为一个起点：
扫描所有编辑内容中的关键词，并提示用户进行授权。当会话非交互式时，则直接中止。

看起来很简单，于是我让一个更强大但速度较慢的本地模型来弄清楚如何构建一个扩展。
每个 Pi 会话开始时都会出现这样一个邀请：

> Pi 可以解释其自身功能并查阅其文档。询问它如何使用或扩展 Pi。

经过几轮迭代，我们有了一个计划，Pi 也生成了一个看起来挺合理的扩展。

我在 Pi 中手动测试了一下，没有任何反应。于是回到原点重新开始。

我又进行了多次迭代，对比了示例代码，查阅了 Pi 的 API，但都没有成功。

最后我安装了那个示例扩展，它能够正常工作。
然后我删掉了自己扩展中的大部分代码，添加了一些日志 —— 这样我终于能看到一些信息了。

## 我对 Pi 及其扩展机制有了不少了解。

看起来，对于任何一个扩展点（例如工具调用或系统启动），只有最后一条 “UI notification” 会被显示出来。
我还不确定这是设计使然还是另有原因。

我从中得到的体会是，在那些不与智能体直接交互的部分，我同样希望采用测试优先的工作方式。
否则，反馈循环实在太慢了。

这也需要一些试验。对于一个仅仅是个想法的扩展，我并不想为了它单独搭建一个项目。
但我确实需要测试。
于是我再次向模型请教。
得到的建议是使用 Deno，因为它内置了测试功能。
随后又进行了一些调试：

- 让 [Deno](https://deno.com/) 在 [Nono](https://nono.sh/cli) 沙箱中正常工作
- 了解到 Pi 会自动加载 extensions 文件夹中的任何内容。
如果你把测试文件放在那里，Pi 会崩溃
（因为测试文件不包含定义扩展所需的方法 —— `.pi/agent/extensions` 目录下的所有文件都必须有该方法）。
- 了解到 “domain” 文件放在那里也行不通（我希望让扩展文件保持精简，把可测试的函数分开存放，这样测试就不会依赖 Pi 及其类型）。

## 最终我采用了这样的设计
<img src="../img/local-llm-handwriting.png" width="100%"/></br>

```
- .pi / test
      / core
      / extensions
```

core 包含功能核心，test 负责测试 core。
extensions 则是与 Pi 进行集成的薄层，它调用 core。
（Deno 项目位于此处）

这个设计足够清晰，以至于那个较慢的稠密模型能够基于这个方案，在经过几轮迭代规划后，相对轻松地构建出第二个扩展以及聊天中的性能指标。

不是出于什么原则，而是因为现在太晚了，而且我想在忘记之前先把这些尝试和错误的过程记录下来。
