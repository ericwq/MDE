# 小型开源 LLM 现已可用于开源智能体

2026/4/24 Willem van den Ende `AI` `privacy`

---
（整理我的思路并让我的工作对他人来说清晰可读的过程中没有 AI 参与，希望如此。）

我正在将我的 Claude Code 工作流 ——如果不是全部的话，至少是大部分—— 替换为 pi.dev（一个开源编码智能体）以及在我笔记本电脑上运行的本地 LLM。
如果你没有相应的硬件，较小的模型在 [Open Router](https://openrouter.ai/) 等托管服务上运行也更便宜（甚至便宜很多）。
随着前沿模型的价格持续上涨、订阅方案不断被稀释，开源权重和开源模型的能力与速度却在持续提升。
过去一个月出现了一个转折点，上周的几个发布
（Qwen 3.6 以及多个实现了性能改进的推理服务器 —— 更快的速度、更少的内存）
标志着用户体验的一次阶梯式跃升。

我在描述中写了 “调整中”，但也许 “挣扎中” 更能准确描述正在发生的事情。
我写下这篇更通用的说明，是因为我计划写一些操作指南，让更多人能够接近 “将编码智能体与推理托管解耦” 这件事。
我们都还在摸索之中，但一旦你搭建并运行起来，你就可以与本地智能体协作，以小步快跑的方式改进你的工具和工作方式。
我从 2024 年开始使用开源权重模型和开源编码工具，期间断断续续地与 Claude Code 一起使用。
2025 年下半年出现了一些强大的开源智能体（最值得一提的是 pi.dev 和 Open Code）， 而现在，与强大且平价的模型搭配，我可以在自己的笔记本电脑上舒适地进行严肃的开发工作了。

大语言模型正在让多种类型的知识和工作方式变得更加易于获取，并缩短了一些反馈循环。
我并不想依赖那些没有任何客户服务、且商业模式在财务上不可持续的前沿实验室。
我的一个朋友开通了账户使用 Claude Code，结果账户在没有任何解释的情况下被关停了。
这似乎是目前非常普遍的情况。
从种种迹象来看，他们的客户服务仅仅是一个表单，而且得不到任何回复。

## Claude Code 是什么？
在订阅 Claude Code 之前，我使用的是 Open Router 搭配本地编码智能体（Aider），半年大概花了 25 美元。
当然，那时候只是偶尔使用，做一些较小的任务。
Open Router 同时提供开源权重模型和前沿实验室模型，定价透明，因此你可以放心地进行尝试。
企业级的 AI 使用也可能会变得对价格更加敏感 —— 正如一些公司现在发现的那样，一个季度就能烧掉原本预算一年的 AI 开支。

我使用 Claude Code 已经有一年左右了。
上周我注意到，自己在谈论它时开始使用过去时态了。
我犹豫过要不要写一个标题党的题目：“Claude Code 已死？”。
去年十一月下旬，Claude Code 的工作效果发生了一次阶跃式的变化 —— Opus 4.5 模型的发布，再加上 Anthropic 那篇关于长时间运行智能体的论文，使得我可以迭代式地构思一个相当复杂的 Web 应用，然后相当轻松且可靠地把它构建出来。
这让我有时间去做探索性测试，并专注于用户体验。

这也让我有些焦虑。
万一我的账户因为某种原因被停用了怎么办？
从一开始按月订阅的模式（对我和许多其他人来说很明显，但显然不是所有人都意识到了）就表明这是不可持续的，总有一天他们不得不涨价。
现在 Anthropic 正把越来越多的功能放在按 token 计费的墙后，而且他们的模型价格很贵。

## 如果你必须精确地给出提示词，那还不如直接用一个小模型
与此同时，Opus 4.7 需要更明确的提示词 —— 大概是为了与 OpenAI 的模型竞争。
“创造力” 在做架构探索时是件好事，但当你在一个较大的代码库中尝试做些精确的工作时，就不那么理想了。
小型开源权重模型（至少直到上周也是如此）同样需要更精确的提示词。
既然无论如何都需要精确地写提示词，那你为什么还要为一个昂贵的前沿模型加闭源工具链付费呢？
你完全可以用本地模型来做同样的事情。

上个月我做了一个小实验，用了一个比我平时使用的小得多的模型（40 亿和 90 亿参数），并在提示词上运行了自动改进循环。
如果你知道预期的输出应该是什么，那么一个更大的模型可以迭代地生成一个更详细的、适用于小模型的提示词。
由于一切都运行在我自己的笔记本电脑上，而且晚上有便宜清洁的电力，我可以用这种方式来减少对前沿实验室的依赖，并帮助其他人。
需要注意的是，我只做过一次这样的实验，因为现在的更大模型相比一个月前所需的内存已经大大减少。
所以我可以让一个大模型在后台持续运行，每当我有问题或需要构建什么东西时，我只需从我的写作环境（编辑器）或编码智能体中直接向它提问即可。

## 大致时间线
开源权重模型已经在逐渐替代前沿模型。
2024 年底，我的一些 Stack Overflow 问题已经可以由本地模型来回答了。
去年我买了一台翻新的 MacBook（我尽量控制嵌入式碳排放）。
然后在去年下半年，主流语言中的越来越多代码开始能够正常工作，一些小众编程语言中的小任务也不成问题了。
一月份，我可以在本地进行事件风暴的迭代，并获得结构化的输出。
三月份，我能够对一个 50 万行代码的遗留 C# 代码库进行代码分析，而且不需要太多明确的提示词。
上周，我成功为一个 [本地编码智能体](https://pi.dev) 开发了扩展。
这周，我全面投入，把我最喜欢的 “技能 (skills)” 都迁移到了它上面。
其中部分迁移用到了 Claude Code，但相当大一部分是直接使用这个编码智能体本身加本地模型来完成的。

## 为什么是现在？
上周发生变化的是，一些开源权重模型现在在我笔记本电脑上的运行速度已经比 Claude Code 还要快，同时质量差距还在不断缩小。
速度本身就有其独特的价值。
“到下一次响应的时间” 在夜间跑任务时并不重要，但在你创建任务或对其进行迭代时却至关重要。
我喜欢快速的反馈，喜欢能够迅速地引导我的工作。
我认为很多那种 “我并行跑十个智能体会话” 的现象，其实只是推理被限流的一个副作用 —— 大规模模型由于其硬件占用，或多或少都会导致这种限制。

我去年就希望开源权重和开源模型能够有所改进。
这种动力是存在的：小模型训练成本更低（或者由许多小模型组成的大模型 —— 混合专家模型现在已经很普遍，并且越来越好），运行成本也更低。
更便宜，通常也更快。
你只要去 Open Router 看看托管的混合专家模型和稠密模型的 TPS（每秒 token 数）就能明白
（比如比较一下 Qwen 27B 和 35B 混合专家模型）。
更便宜、更快意味着更多的组织和个人能够参与进来。
有些开源权重模型的发布并不那么出色，但只要其中有几个是好的，那就没关系。
社区会弄清楚哪些模型适合哪些用例，以及哪些推理服务器在哪种硬件上运行良好。

## 从哪里开始？
你可以在浏览器中通过 Open Router 和 Hugging Face 试用模型，或者在手机上安装 “Google Edge” 或其他应用 —— 它们可以直接在你的手机上运行一个小模型。
你可以先感受一下它们是什么样的。
然后，你可以将一个 API（比如 Open Router，当然也可以是 Anthropic、Google 或 OpenAI 的前沿模型）连接到一个本地编码智能体。
OpenCode 开箱即用很容易配置，而且自带一个免费的云模型，所以你可以直接上手。
我不太喜欢这一点，因为当你在输入你想用的本地私有模型时，如果打错一个字，它就会默认回退到那个免费的云模型。
我目前使用的是 Pi.dev，它更加精简，而且 “快速失败”。

显然，我得先把 “我目前走到哪一步了” 这篇发出来。
