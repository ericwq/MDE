# 框架智能

Lattice 的基础框架和动态上下文层如何通过反馈循环、验证环节和 AI 合规技术来创造智能。

> **受众**：框架的使用者和贡献者，他们希望理解事物*为什么*这样设计，而不仅仅是*做什么*。

---

## 1. 两遍生成模型

**问题**：要求 AI 同时进行生成和验证是不可靠的 —— 就像要求一位作者在同一个环节中同时写作和校对一样。
创作任务（生成代码）和分析任务（检查合规性）会争夺注意力，总有一方会受损。

**解决方案**：先生成，后验证 —— 两个不同的认知任务按顺序执行。

**在 Lattice 中的工作方式**：code-forge 生成一个组件，然后在向用户呈现输出之前，根据原子技能清单对其输出运行检查。

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│  生成    │───→│  验证    │───→│  呈现    │
│  组件    │    │ 对比原子 │    │  给用户  │
└──────────┘    └──────────┘    └──────────┘
   创作任务        分析任务       仅当通过验证
                                 后才呈现
```

这种分离很重要，因为 AI 模型在每次只给出一个明确任务时，能更可靠地遵循指令。
“编写这个函数” 然后 “现在检查这 10 条规则” 比 “在遵循这 10 条规则的同时编写这个函数” 效果更好。

## 2. 验证层级

三个层级的验证，每个层级捕获不同类别的问题：

```
Level 1: 组件级      ──→ 逐组件原子技能清单
                               (clean-code, DDD, secure-coding)
                                 ↓ 所有组件完成
Level 2: 跨组件级    ──→ 架构一致性检查
                               + 经验教训检查
                                 ↓ 实现完成
Level 3: 审查级      ──→ 独立评估
                               + 见解捕获 + 健康日志
```

**第 1 层 —— 组件级**（code-forge 第 3 步）：“我这段代码写得好吗？” 针对组件中的每个函数和类运行原子技能自我验证清单。
捕获：单一职责原则违反、命名问题、缺失验证、基本类型偏执、安全漏洞。

**第 2 层 —— 跨组件级**（code-forge 第 4 步）：“所有部分是否能够正确地组合在一起？” 验证依赖方向、交互流程、安全边界以及对批准蓝图的遵循。
捕获：错误的依赖方向、抽象泄漏、缺失的集成点、在经验教训中标记过的重复模式。

**第 3 层 —— 审查级**（单独调用）：“独立的质量评估。” 以无生成偏见的全新视角审视增量变更。
捕获：生成器自身无法察觉的问题、横切关注点、只有审视全局时才会显现的模式。

为什么是三个层级：原因与人类团队为什么要有自审、同行评审和 QA 一样。
每个层级都能捕获前一层级遗漏的问题。
这些检查的成本很低 —— 只需几秒钟的 AI 思考时间。
而遗漏一个问题的成本，会随着在其之上构建的每个后续组件而不断累积。

## 3. 学习飞轮

飞轮是动态上下文层的核心机制 —— 它将审查发现转化为制度化的记忆，并将其反馈回生成过程。

**问题**：没有反馈循环，相同的错误会在不同功能中重复出现。
审查在支付功能中发现 “贫血领域模型”，然后在用户功能中再次发现，然后在订单功能中又再次发现。
每次审查都是从零开始。

**解决方案**：审查发现作为经验教训持久化存储。
code-forge 在会话开始时加载这些经验教训。
框架随着使用而变得更智能。

```
┌────────────────┐     ┌─────────────┐     ┌──────────────────────┐
│   code-forge   │────→│   review    │────→│ .lattice/learnings/  │
│  加载经验教训   │     │  发现问题    │     │ review-insights.md    │
└───────┬────────┘     └─────────────┘     └────────┬─────────────┘
        │                                           │
        └────────────── 反馈 ──────────────────────┘
```

**格式规范**：每条见解为一个条目 —— 日期、功能、模式、修复方法。
上限约 50 条。
足够简洁，AI 可在几秒内扫描完毕；足够具体，可据此采取行动。

```
- 2026-03-05 [Payment]: 领域服务承担了仓库层的工作 — 将数据访问推送到仓库接口后面
- 2026-03-05 [Payment]: 值对象构造函数中缺少验证 — 在构造函数中验证，而非调用方
- 2026-03-10 [User]: 贫血实体只有 getter — 将行为推入实体方法中
```

为什么这很重要：这是跨功能、跨会话存续的制度化记忆。
团队的编码标准不仅仅是写在风格指南中的内容 —— 它们是从过去审查中积累的经验教训。
飞轮为 AI 辅助开发捕获了这些经验。

## 4. 项目健康度可见性

**问题**：无法回答 “Lattice 真的有帮助吗？我们是否在随着时间进步？”

**解决方案**：review 分子技能在每次审查后，将结构化的摘要追加到 `.lattice/reviews/review-log.md` 中。

```
## 2026-03-05 — Payment 端点
- **范围**：8 个文件，3 个层（领域层、应用层、接口层）
- **原子技能**：clean-code, architecture, DDD, secure-coding
- **结果**：0 个严重，2 个警告，3 个建议
- **主要发现**：PaymentAmount 缺少输入验证；领域服务承担了仓库层的工作
- **优点**：清晰的聚合边界，正确的值对象
```

随时间推移，趋势可以揭示：
- 哪些原子技能捕获的问题最多（团队在哪些方面需要改进）
- 某些反模式是否反复出现（经验教训未被吸收）
- 代码质量是否在提升（每次审查的发现结果减少）
- 哪些层问题最多（培训应聚焦于哪里）

滚动窗口保持 15-20 条条目，防止臃肿。
较旧的条目会在 “历史” 部分中获得一行摘要。

## 5. AI 合规技术

挑战在于：原子技能是 markdown 格式 —— AI 可以阅读它们，但也可能忽略它们。
没有编译器、没有 linter、没有门禁。
合规性依赖于提示工程技巧，这些技巧使 AI 更难跳过步骤。

**技巧 1 —— 带有认知边界的命令式语言**

“停并验证以下**所有**内容” 优于 “在编写时应用这些检查”。
“停” 这个词创造了一个认知边界 —— 从生成模式到验证模式的清晰转换。

**技巧 2 —— 带编号、带标签的约束条件**

LLM 对编号列表的遵循比散文更可靠。
`1. 单一职责原则：` 比一段关于单一职责的段落更难被跳过。
标签既作为检查项名称，也作为思维锚点。

**技巧 3 —— 主动反模式扫描**

与其使用 “以下是要避免的坏模式”（被动引用），不如使用 “扫描你的输出，查找以下模式：[ ] 上帝函数 [ ] 基本类型偏执”（主动清单）。
复选框格式会触发完成行为 —— AI 希望勾选每个框。

**技巧 4 —— 展示你的工作**

要求 AI 报告它检查了什么，迫使其实际执行整个清单。
静默合规是不可靠的；可见合规是可问责的。
当检查通过时，一条简短说明就足够了。
当检查失败时，AI 必须展示它发现了什么以及如何修复的。

**技巧 5 —— 关注点分离**

创作任务（生成）和分析任务（验证）分属不同环节。
这正是第 1 节中所述的两遍模型在架构层面的体现。

## 6. `.lattice/` 文件夹作为制度化记忆

`.lattice/` 文件夹是动态上下文层的物理体现形式 —— 标准、上下文文档、经验教训和健康日志，每个都有各自不同的生命周期。
文件夹结构和生命周期详情请参见 [how-it-works: .lattice/ 文件夹](how-it-works.md#lattice-文件夹) 。

## 7. 端到端流程

流水线中的每个工具既消费也产生持久化产物 —— 这就是使 Lattice 随着使用而改进的累积循环。
design-blueprint 产生上下文文档；
code-forge 消费并丰富它；
review 产生经验教训和健康日志；
这些经验教训反馈回下一次 code-forge 会话。
基础框架在功能周期之间从不改变，但动态上下文层在每个周期中变得更加丰富 —— 标准变得更加精确，经验教训捕获更多模式，健康日志揭示更长期的趋势。

详细的分步流程请参见 [how-it-works: 流水线](how-it-works.md#设计到代码的流水线) 。
