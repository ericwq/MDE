# 协同判断：设计原理

为什么 Lattice 需要一个横切协议 (cross-cutting protocol)，教会 AI 何时提问、不问什么，以及如何高效提问。

## 问题：静默假设

AI 编程助手不断面临真正的歧义 —— 边界模糊的单一职责判断、有争议的层归属、可争辩的聚合边界。
在没有指导的情况下，AI 会默默解决这些歧义。
它选择其中一个选项，并将其呈现为唯一可能的方式。
用户永远不会知道有一个决定是替他们做的。

这种损害是微妙但累积的。
五个静默的微假设产出的代码让人觉得 “不对劲”，但用户无法说清楚原因。
撤销一个被编织进代码的假设，其成本远高于事先做出一个知情的选择。

这正是 collaborative-judgment 要填补的空白：
**让 AI 具备识别自身真正不确定或依据不足之处的智能，并将这种不确定性呈现给用户，而不是去猜测。**

## 该原子技能增加的智能

### 何时提问（以及何时不提问）

大多数决策并 **不是** 模糊的。
当规则明确时（一个 80 行的函数做了 5 件事，违反了单一职责原则 —— 修复它），
当项目已有文档化的偏好时（这不是模糊 —— 这是文档化的意图），
或者当风险较低时（变量命名、格式化 —— 随便选一个），AI 应该自主决定。

<ins>只有当以下三个条件都满足时，才暴露一个决策：存在多个有效的方案，没有项目上下文能解决它，并且该选择有有意义的后果</ins>。

信心测试：“我考虑了两种或多种方案，但根据该项目文档化的上下文，没有哪一种明显更好。”
如果为真 → 暴露。
如果为假 → 决策并继续。
一个什么都问的 AI 和一个什么都不问的 AI 一样没用。

但并非所有不确定性都属于判断性决策。
有时问题在于 **知识缺口**：AI 无法为项目特定的某个说法引用来源，
即将用通用的先验知识填补局部空白，发现存在冲突的有效来源，或者知道缺失某个事实就会导致决策无法成立。
在这些情况下，正确的做法不是编造选项 A 或选项 B，而是先审视现有证据，如果缺口仍然存在，再提出有针对性的澄清问题。

### 批处理

当 AI 确实需要提问时，它不应该为每一个问题都中断：

- **实现期间**：按每个组件进行批处理 —— 在呈现代码之前，将所有判断性决策一起暴露出来。
- **设计期间**：立即暴露 —— 每个设计层级都会约束下一个层级。
- **审查期间**：在行内注明不确定性，同时给出两种解释。
- **在缺失/冲突的依据情况下**：当下一步依赖于未解决的事实时，立即暴露。不要为了保持流程顺畅而将阻塞点进行批处理。

升级信号：如果单个组件产生了超过 3 个判断性决策，那么项目需要更清晰的标准，而不是更多的问题。
建议运行相关的精炼技能作为替代方案。

## 横切协议

Collaborative-judgment 与其他原子技能不同。
代码质量原子技能适用于特定的代码 —— clean-code 检查函数，DDD 检查领域实体，secure-coding 检查信任边界。
Collaborative-judgment 是一个 **横切协议 (cross-cutting protocol)**，加载到 AI 的上下文中，使得所有其他原子技能都可以通过其嵌入的信号回调到它。
它规定了任何歧义 ——无论是由哪个原子技能暴露的—— 如何被呈现、批处理和解决。

### 原子技能如何回调

每个 code-quality 原子技能都携带两个小的嵌入部分，用于连接到本协议：

1. **清单头部指令**：*“如果某项检查是一个存在多种有效方案的判断性决策（参见歧义信号），请标记它 —— 呈现你的选项和推理，而不是默默选择。”* 
这告诉 AI：当你在验证过程中遇到灰色地带时，不要默默解决它。

2. **歧义信号部分**：特定领域的灰色地带，在这些地方该原子技能的检查往往会产生判断性决策 —— clean-code 中边界模糊的单一职责、architecture 中的层归属、DDD 中的聚合边界、secure-coding 中的信任边界范围。

这两个部分构成了 **检测层 (detection layer )** —— 每个原子技能都知道自己的灰色地带，并告诉 AI 标记它们。
Collaborative-judgment 提供 **解决层 (resolution layer)** —— 如何呈现、何时批处理、如何解决。
分子技能将两者加载到同一个上下文中；AI 将它们作为一个整体应用。

## 运行时流程

当用户调用 `/code-forge` 时会发生什么：

```
第 1 步：AI 读取 code-forge → 将所有引用的原子技能加载到上下文中

第 2 步：AI 将所有指令保持在一个上下文中
        ┌─────────────────────────────────────────────┐
        │  code-forge 工作流                          │
        │  collaborative-judgment 协议                │
        │  clean-code 清单 + 歧义信号                 │
        │  architecture 清单 + 信号                   │
        │  DDD 清单 + 信号（如果是领域代码）          │
        └─────────────────────────────────────────────┘

第 3 步：生成组件（创作环节）


第 4 步：运行自我验证清单（验证环节）
        ├─ 明确失败         → 静默修复
        ├─ 明确通过         → 继续
        ├─ 判断性决策       → 从歧义信号中识别
        |                   → 清单头部指令说 “标记它”
        |                   → 收集用于批处理呈现
        └─ 缺失/冲突的依据  → 先审视证据
                            → 若仍未解决，提出有针对性的澄清问题

第 5 步：呈现
        ├─ 零标记          → 输出代码并附上合规说明
        ├─ 1+ 判断性决策   → “需要决策：[问题]
        |                     已检查：[来源]。缺失/冲突：[事实]
        |                     选项 A：... 选项 B：...
        |                     我倾向于 [X]，因为 [原因]。”
        └─ 知识缺口        → “需要澄清：[缺失的事实]
                             已检查：[来源]
                             需要你提供：[有针对性的问题]”

第 6 步：用户解决 → AI 应用 → 继续
```

## 示例

用户调用 `/code-forge` 来实现一个 Order 领域实体。

**没有 collaborative-judgment**：AI 生成了 `OrderValidationService`，将其放在应用层，然后继续。
它考虑过领域层，但默默选择了应用层，因为 “协调多个对象的服务” 听起来像是应用层的工作。
用户审查了 200 行代码，最终才意识到验证逻辑本应在领域层。

**有 collaborative-judgment**：AI 运行架构清单。
第 5 项检查（层归属）被触发 —— architecture 的歧义信号将 “既协调领域对象又包含业务规则的逻辑” 标记为判断性决策。
在组件检查点：

> **需要决策**：`OrderValidationService` 应该放在哪里？
>
> - **选项 A：领域服务** —— 将验证规则保留在领域层；风险是领域服务可能成为一个垃圾场
> - **选项 B：应用服务** —— 保持领域层精简；风险是业务规则泄漏到领域层之外
>
> 我倾向于**领域服务**，因为验证规则（订单金额限制、产品可用性）是领域不变量，而不是编排逻辑。
但这取决于你团队的约定。

用户说 “领域服务”。
AI 应用该选择，建议丰富上下文文档，然后继续。
总成本：10 秒。
价值：用户的架构，而不是 AI 的猜测。

## 飞轮连接

1. AI 遇到歧义 → 暴露给用户 → 用户决策
2. 决策被捕获到上下文锚定文档中（每个功能的持久化存储）
3. 下次该功能再次出现同样的问题 → 上下文文档已有答案 → 无需提问
4. 如果该模式在多个功能中重复出现 → 成为知识库或精炼技能产出标准的候选
5. 一旦纳入标准 → 原子技能清单有了明确答案 → 无歧义 → 无需提问

**该原子技能随着时间推移，会让自己变得不那么必要。**  
它教会项目变得更具体，这使 AI 更加自主，从而使人类更高效。
