# 审查精炼器模板

此模板定义 `.lattice/standards/review-standards.md` 输出文档的结构。它包含审查分子硬编码行为的所有默认值，穿插访谈指导注释。

生成输出时，删除所有 `<!-- INTERVIEW GUIDANCE: -->` 注释。最终文档是规范，而非对话日志。

**范围边界**：此模板配置*审查流程如何工作*——原子加载、严重程度级别、报告格式、范围规则、洞察捕获、健康日志。它**不**配置*原子检查什么*——那些质量标准属于原子特定的精炼器（architecture-refiner、clean-code-refiner、ddd-refiner）。

| 属于此处（流程编排） | 属于原子精炼器（质量标准） |
|---------------------------------------|----------------------------------------------|
| 哪些原子在何时加载 | 原子运行什么检查 |
| 严重程度级别定义 | 什么构成违规 |
| 报告格式和分组 | 检查清单项目和反模式 |
| 差异范围规则 | 层级定义、命名规则 |
| 洞察捕获偏好 | 领域建模规则 |
| 健康日志格式 | 安全检查阈值 |
| 自定义审查维度 | 原子特定的验证逻辑 |

## 1. 原子加载策略

<!-- INTERVIEW GUIDANCE:
询问："审查分子始终加载某些原子，其他原子则根据差异有条件地加载。以下是当前行为。您想要更改始终加载的原子、更改条件原子的条件、抑制任何原子或添加自定义触发条件吗？"

显示下面的表格。

探查问题：
- 是否有您希望始终加载的原子，无论差异如何？（例如，"每次审查都运行 secure-coding"）
- 是否有您想要完全抑制的原子？（例如，"我们不做 DDD，跳过那个原子"）
- 您有基于路径的规则吗？（例如，"src/api/ 中的任何文件都应触发 secure-coding"）
- 是否有任何当前条件原子应变为始终加载？

可自定义：
- 在始终/条件/抑制类别之间移动原子
- 为条件原子添加基于路径的触发规则
- 更改触发条件原子的条件

固定：
- clean-code 必须保持在"始终"类别（基本代码质量普遍适用）
- knowledge-priming 必须保持在"始终"类别（项目上下文总是需要的）
- 不能添加框架中不存在的原子

跨章节影响：
- §2 中的按原子严重程度覆盖必须引用实际可加载的原子
- §3 中的报告章节对应加载的原子
- §5 中的洞察类别可能引用原子名称
- §6 中的日志条目列出加载了哪些原子
-->

### 始终加载

无论更改什么，这些原子在每次审查中都会加载：

| 原子 | 原因 |
|------|------|
| `clean-code` | 代码质量标准（单一职责、命名、复杂度、错误处理）适用于所有代码 |
| `knowledge-priming` | 项目上下文（技术栈、架构、约定）指导所有验证 |

### 条件加载

当差异匹配特定标准时，这些原子会加载：

| 原子 | 触发条件 | 原因 |
|------|----------|------|
| `architecture` | 差异触及多个层级、添加新文件或更改文件位置 | 结构检查仅在结构变更时相关 |
| `domain-driven-design` | 差异包含配置领域文件夹中的文件或修改领域对象 | 领域建模规则仅适用于领域代码 |
| `secure-coding` | 差异触及信任边界（HTTP 处理器、认证、数据库查询、外部 API、密钥、配置） | 安全检查针对处理不受信任输入或敏感操作的代码 |
| `test-quality` | 差异包含测试文件 | 测试标准（AAA、隔离性、命名）仅适用于测试代码 |

### 抑制

默认情况下没有抑制的原子。被抑制的原子永远不会加载，无论差异如何。

### 自定义基于路径的触发器

默认情况下没有自定义路径触发器。路径触发器覆盖上述标准条件。

**自定义示例**（默认不激活）：

```yaml
# 示例：始终为 API 文件触发 secure-coding
- path: "src/api/**"
  loads: secure-coding

# 示例：始终为 domain/ 下的任何文件触发 DDD
- path: "src/domain/**"
  loads: domain-driven-design

# 示例：为配置文件和模块文件触发架构检查
- path: "src/modules/*/module.ts"
  loads: architecture
```

## 2. 严重程度分类

<!-- INTERVIEW GUIDANCE:
询问："审查使用三个严重程度级别对发现进行分类。以下是当前定义。您想要更改这些级别、添加新级别或设置按原子的严重程度覆盖吗？"

显示下面的严重程度表格和按原子覆盖部分。

探查问题：
- 您是否需要 critical 之上的级别？（例如，"blocker"用于 CI 门控或合并阻止问题）
- 是否有任何原子的发现应从最小严重程度开始？（例如，"安全发现从不低于 warning"）
- 是否有任何原子的发现应被限制？（例如，"在团队采用模式期间，DDD 发现限制为 warning"）
- 您想要重新定义每个级别的意义吗？

可自定义：
- 添加新的严重程度级别（例如，"blocker"、"nitpick"）
- 重新定义级别描述和指导
- 设置按原子的最小严重程度（底线）
- 设置按原子的最大严重程度（天花板）

固定：
- 必须至少有两个严重程度级别（一个"必须修复"和一个"可选"级别）
- 级别必须可从最严重到最不严重排序

跨章节影响：
- §3 中的报告排序遵循严重程度顺序
- §5 中的洞察捕获标准可能引用严重程度级别
- §6 中的健康日志计数按严重程度分组
- §7 中的自定义维度需要来自此列表的默认严重程度
-->

### 严重程度级别

发现按以下级别分类，从最严重到最不严重排序：

| 级别 | 含义 | 指导 |
|------|------|------|
| **critical（严重）** | 会导致 bug、安全漏洞或数据丢失 | 合并前必须修复 |
| **warning（警告）** | 违反原则并会导致维护痛苦 | 应该修复——保留会成为技术债务 |
| **suggestion（建议）** | 可以改进，但当前工作正常 | 考虑修复——不阻止 |

### 按原子的严重程度覆盖

默认情况下没有按原子的覆盖。所有原子使用标准严重程度分类。

**自定义示例**（默认不激活）：

```yaml
# 示例：安全发现从不低于 warning
- atom: secure-coding
  minimum_severity: warning

# 示例：在采用期间 DDD 发现限制为 warning
- atom: domain-driven-design
  maximum_severity: warning

# 示例：架构违规始终是至少 warning
- atom: architecture
  minimum_severity: warning
```

当按原子的覆盖激活时，来自该原子的发现被钳制到指定的范围。安全发现通常会是"suggestion"，使用最小覆盖会提升到"warning"。DDD 发现通常会是"critical"，使用最大覆盖会限制为"warning"。

## 3. 报告偏好

<!-- INTERVIEW GUIDANCE:
询问："审查报告默认使用摘要模式，按严重程度排序发现。您想要更改默认模式、调整发现格式或自定义报告中显示的内容吗？"

显示下面的默认值。

探查问题：
- 默认模式是否应该是完整而不是摘要？
- 您想要在摘要模式中限制发现数量吗？（例如，"仅显示前 10 个"）
- 您想要保留还是删除"做得好的地方"部分？
- 发现是否应该以不同方式分组？（按原子、按文件、按严重程度）
- 您是否想要在每份报告中添加自定义章节？（例如，"性能备注"、"迁移安全性"）

可自定义：
- 默认报告模式（摘要或完整）
- 摘要模式的发现上限
- "做得好的地方"开关（开/关）
- 分组策略（按严重程度、按原子、按文件）
- 发现格式模板
- 自定义报告章节

固定：
- 摘要和完整模式都必须保持可用（用户始终可以请求任一模式）
- 发现必须至少包括文件位置和描述
-->

### 默认模式

**摘要模式**是默认的。用户始终可以明确请求完整模式。

### 发现格式

每个发现遵循以下格式：

```
[严重程度] 文件：行号 -- 描述 (原子名称：检查名称)
```

### 摘要模式行为

- 按严重程度排序发现（critical → warning → suggestion）
- 显示最重要的发现——不要枚举每个小问题
- 发现没有硬性上限（包含所有重要的）
- 以**"做得好的地方"**句子结尾，突出差异中的积极方面

### 完整模式行为

- 按原子组织发现（每个加载的原子一个章节）
- 包含所有发现，每个原子章节内按严重程度排序
- 以**"做得好的地方"**（2-3 个积极观察）和可选的**"改进建议"**（1-2 个更广泛的模式）结尾

### 分组策略

默认分组：**摘要模式**按严重程度，**完整模式**按原子。

**替代分组选项**（默认不激活）：

- **按文件**：按文件路径对发现分组，每个文件内按严重程度排序。适用于触及许多文件的大差异。
- **按严重程度**：按严重程度级别对发现分组，所有原子混合。摘要模式的默认值。
- **按原子**：按原子对发现分组，每个原子内按严重程度排序。完整模式的默认值。

### "做得好的地方"

默认启用。每次审查都包括至少一个关于差异的积极观察——良好的命名、适当的错误处理、清晰的测试结构、正确的层级放置。

### 自定义报告章节

默认情况下没有自定义章节。

**自定义示例**（默认不激活）：

```yaml
# 示例：在每份报告末尾添加"性能备注"章节
- name: "性能备注"
  description: "关于差异中性能模式的观察"
  position: after-findings

# 示例：为数据库更改添加"迁移安全性"章节
- name: "迁移安全性"
  trigger: "差异包含迁移文件"
  position: after-findings
```

## 4. 范围规则

<!-- INTERVIEW GUIDANCE:
询问："审查专注于差异——仅更改的文件和行。当创建新违规时，它可以查看周围代码。您想要更改范围的工作方式吗？"

显示下面的默认值。

探查问题：
- 审查是否应扩展到"直接依赖项"——从更改文件导入的文件？
- 是否有目录应始终排除？（生成的代码、供应商依赖项、迁移）
- 是否有目录在其中的任何文件更改时应始终完全包含？（例如，"当任何领域文件更改时，始终审查完整的 domain/ 文件夹"）
- 周围代码应如何处理？（严格：仅差异，默认：差异 + 周围代码中的新违规，宽松：差异 + 更改代码周围的所有上下文）

可自定义：
- 将范围扩展到直接依赖项
- 目录排除（glob 模式）
- 目录包含（始终全量扫描目录）
- 周围代码策略（严格/默认/宽松）

固定：
- 审查必须始终包含差异本身（不能排除更改的文件）
- 范围扩展必须有边界——无界定的图遍历

跨章节影响：
- 扩展的范围可能触发 §1 中的更多条件原子
- §7 中的自定义维度遵循相同的范围规则
-->

### 差异焦点

审查专注于差异——更改的文件和行。这是变更集，而非整个代码库。

### 周围代码策略

**默认**：专注于差异。例外：当差异中的更改在现有代码中创建新违规时审查周围代码（例如，破坏现有文件依赖规则的新依赖）。当审查周围代码时，注明发现源于差异的影响，而非预先存在的问题。

**可用策略：**

| 策略 | 行为 |
|------|------|
| **严格** | 仅差异。从不查看周围代码。 |
| **默认** | 仅当差异在其中创建新违规时，才审查差异 + 周围代码。 |
| **宽松** | 差异 + 更改代码周围的更广泛上下文，用于整体审查。 |

### 依赖扩展

默认禁用。启用时，审查还检查直接从更改文件导入的文件（"直接依赖项"）。这捕获更改破坏消费者的情况。

### 目录排除

默认情况下没有排除。差异中的所有文件都会被审查。

**自定义示例**（默认不激活）：

```yaml
# 示例：排除生成的代码
- pattern: "src/generated/**"
  reason: "自动生成，非人工编写"

# 示例：排除供应商依赖项
- pattern: "vendor/**"
  reason: "第三方代码，非项目所有"

# 示例：排除数据库迁移
- pattern: "migrations/**"
  reason: "模式迁移单独审查"
```

### 目录包含（始终全量扫描）

默认情况下没有始终全量扫描的目录。

**自定义示例**（默认不激活）：

```yaml
# 示例：当任何领域文件更改时始终扫描完整的 domain/
- pattern: "src/domain/**"
  reason: "领域完整性需要完整上下文审查"

# 示例：当任何认证文件更改时始终扫描完整的 auth 模块
- pattern: "src/auth/**"
  reason: "安全敏感模块需要整体审查"
```

## 5. 洞察捕获偏好

<!-- INTERVIEW GUIDANCE:
询问："每次审查后，分子将重复模式捕获到代码生成使用的学习文件中以避免重犯。以下是当前默认值。您想要更改洞察的捕获方式吗？"

显示下面的默认值。

探查问题：
- 修剪阈值是否应该更高？（例如，100 项用于具有更多审查历史的大型团队）
- 您想要分类标签吗？（例如，[security]、[domain]、[performance]）
- 是否应始终捕获某些发现？（例如，"无论重复与否都始终捕获安全发现"）
- 您更喜欢分组格式（按类别）还是扁平格式（ chronological ）？

可自定义：
- 修剪阈值（在建议清理之前的条目数）
- 分类标签
- 捕获标准（什么触发洞察被保存）
- 格式（按类别分组或扁平 chronological ）
- 条目长度限制

固定：
- 操作学习由 learning-harvest 原子管理（用户确认，从不自动写入）
- 文件路径是 .lattice/learnings/operational-learnings.md（可通过 config.yaml 中的 paths.operational_learnings 配置）
- 此部分配置审查分子在收获期间传递给 learning-harvest 的额外指导

跨章节影响：
- 捕获标准可能引用 §2 中的严重程度级别
- 类别可能与 §1 中加载的原子对齐
-->

### 文件位置

由 `learning-harvest` 原子在 `.lattice/learnings/operational-learnings.md` 管理。审查分子在会话结束时调用 learning-harvest 的 Harvest 行为——此部分提供额外的捕获指导。

### 条目格式

```
- YYYY-MM-DD [上下文] 模式 — 可操作的收获
```

每个条目是一个项目符号，最多 2 行。在操作学习文档中按类别组织。

### 捕获标准

仅建议将有助于未来会话的模式——不是每个发现。一个偶然的拼写错误不是学习；"领域服务一直做仓储工作"是。要传递给 learning-harvest 的额外标准：

- 跨审查的重复模式
- 系统性违规（非一次性错误）
- 任何分子在未来会话中可以采取行动的图案

### 修剪阈值

如果文件超过 **~50 项**，建议修剪在最近审查中未重复的最旧条目。

### 分类

默认情况下没有分类标签。条目是扁平且 chronological 的。

**自定义示例**（默认不激活）：

```yaml
# 示例：为每个洞察添加类别标签
categories:
  - "[security]"
  - "[domain]"
  - "[architecture]"
  - "[performance]"
  - "[testing]"

# 带类别的格式：
# - YYYY-MM-DD [security] [功能]: 模式 — 收获
```

### 分组与扁平格式

**扁平**（默认）：所有洞察按 chronological 顺序，最新的在底部。

**分组**（替代方案）：洞察按类别标题组织。需要启用分类。

### 条目格式

## 7. 自定义审查维度

<!-- INTERVIEW GUIDANCE:
询问："除了标准原子检查之外，是否有您希望每次审查都检查的特定于项目的审查关注点？这些是不适合现有原子的检查。"

此部分**没有默认值**——完全是添加性的。仅当用户定义至少一个维度时才在输出中包含它。

显示下面的示例以激发想法。

探查问题：
- 是否有需要关注的性能模式？（N+1 查询、循环中的 DB 调用、无界结果集）
- 是否有 API 一致性规则？（命名约定、响应信封格式、版本控制）
- 是否应标记没有文档注释的公共 API？
- 是否有迁移安全性关注点？（模式更改无回滚、数据迁移无验证）
- 是否有向后兼容性规则？（公共 API 的破坏性更改、删除的字段）

可自定义：全部——这是一个空白画布。

固定：
- 每个维度必须具有名称、检查内容、触发条件和默认严重程度
- 维度遵循与原子相同的两遍审查模式（检查标准 → 记录发现）
- 维度使用 §2 中的严重程度级别

跨章节影响：
- 维度贡献需要 §2 中严重程度分类的发现
- 维度发现出现在报告中，遵循 §3 偏好
- 维度遵循 §4 中的范围规则

默认情况下没有自定义维度。此部分完全是添加性的——为现有原子未覆盖的特定于项目的审查关注点定义维度。

### 示例维度

这些是说明格式的示例。默认都不激活。

#### 性能模式

```yaml
name: "性能模式"
check: |
  - N+1 查询：循环中的 DB 调用或重复获取相关数据
  - 无界结果集：没有 LIMIT 或分页的查询
  - 热路径中的 DB 调用：频繁调用的工具函数中的数据库访问
  - 缺少索引：在非索引列上过滤的查询（当有模式时）
trigger: "差异包含数据库查询、仓储调用或数据访问代码"
default_severity: warning
```

#### API 一致性

```yaml
name: "API 一致性"
check: |
  - 端点命名遵循约定（例如，复数名词、kebab-case）
  - 响应信封格式一致（例如，{ data, error, meta }）
  - 错误响应包括错误代码
  - 分页遵循项目的标准模式
trigger: "差异包含控制器/处理器或 API 路由定义"
default_severity: suggestion
```

#### 文档覆盖率

```yaml
name: "文档覆盖率"
check: |
  - 公共 API 函数/方法有文档注释
  - 导出的类型/接口有描述
  - 复杂的业务逻辑有内联解释
trigger: "差异包含公共 API 定义或导出的接口"
default_severity: suggestion
```

#### 迁移安全性

```yaml
name: "迁移安全性"
check: |
  - 模式更改有回滚迁移
  - 数据迁移在转换前验证
  - 列删除前有弃用期
  - 新的 NOT NULL 列有默认值
trigger: "差异包含迁移文件或模式更改"
default_severity: critical
```

#### 向后兼容性

```yaml
name: "向后兼容性"
check: |
  - 删除或重命名的公共 API 端点已版本化
  - 更改的响应形状有弃用通知
  - 删除的字段在硬删除前使用软弃用
  - 破坏性更改记录在变更日志中
trigger: "差异修改公共 API 端点或响应类型"
default_severity: warning
```

### 维度格式

每个自定义维度必须包括：

| 字段 | 必需 | 描述 |
|------|------|------|
| `name` | 是 | 维度的可读名称 |
| `check` | 是 | 要查找的内容——模式或规则的清单 |
| `trigger` | 是 | 此维度适用的条件——基于差异分类 |
| `default_severity` | 是 | 来自此维度的发现的默认严重程度（必须是 §2 中的级别） |

维度遵循与原子相同的两遍模式：检查差异的标准，然后记录发现的严重程度、文件位置和建议修复。维度发现与原子发现在报告中合并。
