# Custom Architecture 执行规则

Custom architecture 模式的执行指令。当设置 `architecture_mode: custom` 时，指导 AI 读取/应用团队架构文档。无内置默认值——团队文档 = 唯一标准。

## 读取团队的文档

团队架构文档（在 `paths.architecture`）= 架构规则的唯一参考。在生成/审查代码之前完整读取。

在文档中查找关键部分：

1. **层定义**——层存在、每层属于什么、典型目录映射
2. **依赖规则**——哪些层依赖于哪些、依赖方向
3. **边界规则**——层如何通信、DI 模式、数据跨越格式
4. **每层规则**——每层允许/禁止的内容
5. **关键流程**——通过架构的代表性数据流（例如，写操作、读操作）
6. **验证清单**——生成代码后运行的编号检查
7. **反模式**——复选框模式扫描/修复
8. **模糊信号**（可选）——AI 呈现选项而非静默选择的灰色区域

如果文档有超出§8 的额外部分，读取/应用为额外的架构指导。

## 自我验证清单

生成每个组件后 STOP。从加载的架构文档中读取 **Validation Checklist** 部分（§6）。按顺序 walkthrough 每个编号项目，在继续之前验证所有。如果任何检查明显失败，在呈现之前修复代码。如果检查 = 判断调用（参见模糊信号），标记它——呈现选项/推理而非静默选择。

如果加载的文档没有 Validation Checklist 部分，浮现警告：

> "你的架构文档缺少验证清单部分。没有它，architecture atom 无法运行特定风格的生成后验证。考虑重新运行 `/architecture-refiner` 以添加验证清单。"

然后应用这些基线结构检查作为回退——部分执行优于无执行：

1. **层放置**：每个类/模块在加载文档定义的正确的层中？
2. **依赖方向**：所有源代码依赖遵循加载文档中的方向规则？
3. **边界数据**：跨越层的数据使用加载文档中描述的模式？
4. **单层**：每个类恰好属于一层？没有跨越层的多重关注点。

## 主动反模式扫描

在验证清单后，从加载的架构文档中读取 **Anti-Patterns** 部分（§7）。扫描输出以查找每个列出的反模式。如果找到任何，在呈现代码之前修复。

如果加载的文档没有 Anti-Patterns 部分，浮现警告：

> "你的架构文档缺少反模式部分。没有它，architecture atom 无法扫描特定风格的反模式。考虑重新运行 `/architecture-refiner` 以添加反模式部分。"

## 模糊信号

如果加载的文档有 **Ambiguity Signals** 部分（§8），在生成代码之前读取。当在生成期间遇到描述的场景时，使用 `framework:collaborative-judgment` 呈现选项/推理而非静默选择。

如果加载的文档没有 Ambiguity Signals 部分，使用判断——当组件可以合理地生活在文档规则的两个不同层中时，或流程可以遵循多个有效模式时，作为判断调用浮现。

## 应用架构

使用加载的文档定义强制执行结构规则：

- **层放置**：验证每个类/模块在文档 Layer Definitions 部分定义的正确的层中
- **依赖方向**：验证所有源代码依赖遵循文档 Dependency Rules 部分中的方向规则
- **边界规则**：验证跨越层边界的数据遵循文档 Boundary Rules 部分中的模式
- **每层规则**：验证每层允许/禁止的模式匹配文档 Per-Layer Rules 部分
- **流程验证**：当文档描述架构流程（Key Flows 部分）时，用作参考验证生成代码结构

应用规则时，将文档定义视为权威——代表团队架构决策，具有与 clean architecture 内置规则相同的执行权重。
