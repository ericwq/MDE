# 面向编码智能体用户的管控工程
*若要让编码智能体在更少监督下运行，我们需要建立对其输出结果的信心。
作为软件工程师，我们对 AI 生成代码天然存在信任壁垒 ——
LLM 具有非确定性，不了解我们的业务上下文，也并非真正理解代码，它们只是基于 token 进行推理。
本文探讨一种思维模型，整合来自上下文工程与管控工程的前沿理念，以此构建这种信任。*

术语 “harness”（管控）已经被用作一种简化说法，表示一个 AI 智能体中除了模型本身之外的所有部分 —— 即 智能体 = 模型 + Harness。
这是一个非常广泛的定义，因此有必要针对常见类型的智能体进行细化。
我想在此斗胆定义其在使用编码智能体的界限下文中的含义。
在编码智能体中，部分 harness 已经内置（例如，通过系统提示词、所选的代码检索机制，甚至是复杂的编排系统）。
但编码智能体同时也为用户提供了大量功能，让我们能够针对自身场景与系统，构建外部管控。

一套设计精良的外部管控框架（Outer Harness）有两大目标：
一是提升智能体首次输出即正确的概率；二是构建反馈闭环，在问题到达人工之前尽可能完成自我修正。
最终目标是减少审核工作量、提升系统质量，同时附带减少过程中 Token 浪费的额外收益。

## 前馈与反馈
要对代码智能体实施有效管控，我们既要预判不良输出并加以防范，也要部署相应感知机制，使智能体能够自我修正：

- **指南（前馈控制）**
预判智能体行为，并在其执行前进行引导。
Guides 可提升智能体首次输出即达到良好效果的概率。

- **感知器（反馈控制）**
在智能体执行后进行监测，并协助其自我修正。
若感知器能生成适合 LLM 读取的信号，效果会格外出色 ——
例如包含自我修正指令的自定义代码检查消息，这属于正向提示词注入。

若仅使用其中一种：
- 仅反馈 → 智能体会不断重复相同错误；
- 仅前馈 → 智能体虽编码规则，却无法验证规则是否真正有效。

## 计算型 vs 推理型
Guides 与感知器分为两种执行类型：

- **计算型** —— 确定性强、速度快，由 CPU 运行。
包括测试、代码检查工具、类型检查器、结构分析。
执行耗时毫秒至秒级，结果可靠。

- **推理型** —— 语义分析、AI 代码评审、“LLM 作为评判者”。
通常由 GPU 或 NPU 运行，**更慢、成本更高**，结果**不确定性更强**。

计算型 Guides 通过确定性工具提升优质结果的概率。
计算型感知器成本低、速度快，可在智能体每次修改时同步运行。
推理型控制固然成本更高、具有不确定性，但能够提供丰富的 guidance 并增加额外的语义判断。
尽管存在不确定性，若使用强大的模型（或更准确地说，适合当前任务的模型），推理型感知器能显著提升我们的可信度。

## 调控闭环 (steering loop)
在这一过程中，人工的职责是通过迭代优化管控来 **调控 (steer)** 智能体。
每当某个问题多次出现时，就应改进前馈与反馈控制机制，降低该问题未来发生的概率，甚至完全避免。

在调控闭环中，我们同样可以利用 AI 来优化管控。
如今代码智能体大幅降低了构建自定义控制与自定义静态分析的成本。
智能体可协助编写结构测试、从观测到的模式生成规则草案、搭建自定义代码检查工具，或通过代码库分析形成操作指南。

## 时机把控：质量左移
[持续集成](https://martinfowler.com/articles/continuousIntegration.html) 的团队始终面临一项挑战：根据成本、速度与关键程度，在开发全流程中合理排布测试、检查与人工评审。
若要实现 [持续交付](https://martinfowler.com/bliki/ContinuousDelivery.html)，理想状态是每一次提交都处于可部署状态。
应尽可能在通往生产环境的流程最左侧执行检查，因为问题发现得越早，修复成本越低。
包括新型推理型感知器在内的各类反馈感知器，也需按此原则在整个生命周期中合理分布。

### 变更生命周期中的前馈与反馈
- 哪些操作速度足够快，应当在代码集成之前，甚至在提交创建之前就执行？
（例如：代码检查工具、快速测试套件、基础代码评审智能体）

- 哪些操作成本更高，因此应当仅在集成后的流水线中执行，并同时重复运行快速检查控制？
（例如：变更测试 (mutation testing)、能够兼顾整体的更全面的代码评审）

## 监管类别
智能体管控的作用类似于 [控制论](https://en.wikipedia.org/wiki/Cybernetics) 调节器 (cybernetic governor)，
通过结合前馈与反馈，将代码库调控至预期状态。
区分该预期状态的多个维度十分有用 —— 可按照管控的调控对象进行分类。
区分这些类别具有实际意义，因为可管控性与复杂度在不同类别中存在差异；
对通用术语进行限定描述，能让表达更精准。

以下是目前我认为较为实用的三种类别：

### 可维护性管控
本文中我给出的几乎所有示例，都围绕内部代码质量与可维护性的调控展开。
这是目前最容易实现的一类管控类型，因为我们已有大量成熟工具可供使用。

为了衡量上述可维护性管控思路能在多大程度上提升我对智能体的信任度，我将此前整理的 [常见代码智能体失效模式](exploring-gen-ai-13.md) 与这套管控体系进行了对应分析。

计算型感知器能够可靠地捕获结构性问题：重复代码、圈复杂度、测试覆盖率缺失、架构偏移、编码规范违规。
这些感知器成本低、成熟可靠且具有确定性。

LLM 可以部分解决需要语义判断的问题 —— 语义重复代码、冗余测试、暴力修复、过度设计的方案 —— 但代价高昂且结果具有概率性，不适合在每次提交时运行。

两者都无法可靠地捕获部分影响更大的问题：问题误判、过度设计与冗余功能、指令理解偏差。
它们偶尔能发现这些问题，但可靠性不足以减少人工监督。
如果人类一开始没有清晰定义预期目标，正确性问题就不在任何感知器的处理范围内。

### 架构适配性管控
这个组合包括定义和检查应用程序架构特性的指南和传感器。本质上就是适配函数（Fitness Functions）。

示例：
- 前馈性能需求的 skills，以及向智能体反馈性能是提升还是劣化的性能测试。
- 描述编码规范以提升可观测性（如日志标准）的 skills，以及要求智能体反思其可用日志质量的调试指令。

### 行为管控
这是一个显而易见却被忽视的关键问题 —— 我们该如何指导 (guide) 并感知应用功能行为是否符合预期？
目前，我观察到大多数人赋予代码智能体高度自主权的从业者采用如下方式：

- **前馈**：功能规格说明（详细程度不一，从简短提示词到多文件描述）
- **反馈**：检查 AI 生成的测试套件是否通过、测试覆盖率是否达标，部分人甚至会通过变异测试 (mutation testing) 监控测试质量，
再结合人工测试。

这种方式过度依赖 AI 生成的测试，目前效果仍不够理想。
我的部分同事在使用 [认可测试夹具](https://lexler.github.io/augmented-coding-patterns/patterns/approved-fixtures/) 模式（approved fixtures pattern）时看到了不错的效果，但该模式仅适用于部分场景，无法全面解决测试质量问题，仅能在合适场景选择性使用。

因此总体而言，要构建完善的功能行为管控，使其足够可靠以减少监督与人工测试，我们仍有大量工作要做。

## 可管控性
并非所有代码库都适合同样程度的管控。
使用强类型语言编写的代码库天然具备类型检查器作为感知器；
清晰可定义的模块边界支持架构约束规则；
Spring 等框架封装了智能体无需关心的底层细节，从而隐性提升了智能体的成功概率。
若不具备这些属性，则无法构建对应的控制机制。

这一点在全新项目与遗留系统上表现迥异。
全新项目团队可以从第一天起就内置可管控性 —— 技术选型与架构设计直接决定代码库的可调控程度。
而遗留系统团队，尤其是存在大量技术债务的应用，面临更严峻的问题：最需要管控的地方，往往最难构建管控体系。

## 管控模板
大部分企业拥有若干种通用的服务架构模式，覆盖其 80% 的需求 —— 通过 API 暴露数据的业务服务、事件处理服务、数据仪表盘。
在许多成熟的工程组织中，这些模式已经被固化为服务模板。
未来这些模板可能演进为管控模板：一套将指南与感知器打包的组合，把代码智能体约束在既定架构模式的结构、规范与技术栈之内。
团队在选择技术栈与架构时，可能会根据已有管控的可得性 (available)。

我们当然会面临与服务模板类似的挑战。
一旦团队将其实例化，就会开始与上游的改进失去同步。
管控模板同样会面临版本管理与协作贡献问题；
而由于存在难以测试的非确定性指南和感知器，这类问题可能会变得更加严重。

<div style="background-color: darkblue; padding: 8px; border: 1px solid lightblue;">
  <b>艾什比定律 (Ashby's Law)</b>
  </br>
  </br>
  <a href="https://en.wikipedia.org/wiki/Variety_(cybernetics)#Law_of_requisite_variety">艾什比的必要多样性定律]</a>
  该定律指出：调节器 (regulator) 必须具备至少与其所管控系统同等的多样性，且只能对自身拥有模型的对象进行调控。
  基于 LLM 的代码智能体几乎可以生成任何内容，但选定一种架构模式 (topology) 会缩窄其空间，从而让完整的管控更易实现。
  定义架构模式 (topology) 本质上是一种降低多样性的举措。
</div></br>

## 人类的角色
作为人类开发者，我们将自身技能与经验作为隐性管控作用于每个代码库。
我们内化了各类规范与最佳实践，体会过复杂代码带来的认知负担，也清楚每次提交都关乎自身声誉。
我们还具备组织协同意识 —— 了解团队目标、因业务需求可容忍的技术债务，以及特定场景下 “优质代码” 的标准。
我们以人类的节奏循序渐进，这为触发并运用过往经验提供了思考空间。

代码智能体完全不具备这些特质：
没有社会责任约束，对长达 300 行的函数不会产生审美上的反感，没有 “我们这里不这么做” 的直觉，也没有组织记忆。
它不知道哪些规范是关键必要的、哪些只是习惯做法，也不清楚技术上正确的方案是否符合团队的实际目标。

<ins>管控的意义，在于将人类开发者经验所带来的隐性价值外化并明确化，但其作用终究有限</ins>。
构建一套包含指南、感知器与自我修正闭环的完整系统成本高昂，因此我们必须带着明确目标区分优先级：一套优秀的管控体系，未必旨在完全消除人工介入，而是将人类投入引导至最具价值的环节。

## 起点与待解决问题
我在此阐述的思维模型，描述了已在实践中应用的技术，并有助于梳理我们仍需探索的问题。
其目标是将讨论提升到功能层面之上 ——
从 skills 和 MCP 服务器 到我们如何战略性地设计一套控制系统，从而对智能体输出的结果产生真正的信心。

以下是当前讨论中与管控体系相关的若干示例：

- [OpenAI 一个团队记录了他们的管控体系](https://openai.com/index/harness-engineering/) ：
通过自定义代码检查工具与结构性测试来强制实施分层架构，并通过周期性 “垃圾回收” 扫描代码偏移，由智能体自动提出修复方案。
他们的结论是：“当前我们最棘手的挑战，集中在环境设计、反馈闭环与控制系统上。”

- [Stripe 在关于其 minions（智能助手）的文章](https://stripe.dev/blog/minions-stripes-one-shot-end-to-end-coding-agents) 中描述了相关实践：
例如基于启发式执行对应代码检查的 pre‑push 钩子；
他们强调 “反馈左移” 的对他们的重要性，而其 “蓝图（blueprints）” 展示了他们如何将反馈感知器集成到智能体工作流中。

- 变异测试 (Mutation) 与结构性测试是过去未被充分利用的计算型反馈感知器，如今正重新受到重视。

- 开发者群体中越来越多地讨论将语言服务协议（LSP）与代码智能能力集成到编码智能体中，这些正是计算型前馈 guides 的典型案例。

- 我从 Thoughtworks 的多个团队了解到，他们同时使用计算型与推理型感知器解决架构偏移问题：
例如通过智能体与自定义代码检查工具结合提升 API 质量，或利用 “清洁大军（janitor army）” 改善代码质量。

仍有大量问题有待解决，不仅仅是前文提到的行为管控。
随着管控体系不断扩展，我们如何保持其一致性，让指南与感知器保持同步、互不冲突？
当指令与反馈信号出现矛盾时，我们能在多大程度上信任智能体做出合理权衡？
如果感知器从未触发警报，这代表高质量，还是检测机制不足？
我们需要一种方法来评估管控覆盖度与质量，类似于测试中的代码覆盖率与变异测试。
前馈和反馈控制目前分散在各个交付环节中，实际上存在开发工具来帮助配置、同步和理解它们作为一个系统的潜力。
构建这种外部管控，正逐渐成为一项持续的工程实践，而非一次性配置。
