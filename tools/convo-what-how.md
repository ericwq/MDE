# 对话：LLM 与目标/实现闭环

*Unmesh, Rebecca 与 Martin 三人展开对话，探讨 LLM 如何帮助我们构建软件中的抽象设计。
我们面临的核心挑战是打造能够应对变化的系统，这就需要管控自身的认知负荷 (cognitive load)。
我们可以通过将软件所需实现的目标（what），转化为编程语言层面的实现方式（how）来达成这一点。
这种 “目标” 与 “实现” 会形成一个反馈闭环。
TDD 帮助我们将这一闭环落地实施，而 LLM 则让我们能以更随意、更灵活的方式探索这个闭环。*

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  软件开发的首要挑战，是构建能够应对变化的系统。
</div></br>

- Unmesh

  编程初学者，以及并非以编码为生、但会雇佣程序员的人，常常把编程看作是把需求线性翻译成编程语言语法的过程。
  这种观念会误导人，让人要么花费大量时间把需求或规格写得无比正确，要么埋头钻研编程语言语法本身。

  在使用 LLM 的场景下，这种观念体现在 “人在回路中 (Human in the loop)” 这类说法里。
  该术语意味着：将需求转化为代码的主要工作由 LLM 完成，人类只在模型出错时负责修正。

  但任何有经验的程序员都清楚，真正的挑战并非把需求转成代码，而是构建能够持续应对变化的系统。

  那么，是什么让系统在变化发生时更易于维护？

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  要让系统更易于变更，我们需要管控认知负荷 (cognitive load)。
</div></br>

- Martin

  让系统更易变更的关键之一，是为需要执行变更的人员管理认知负荷。
  我或许无法在脑海中装下一百万行代码，但如果系统被合理地拆分为多个模块，我可能只需要理解其中几百行 —— 就能推进工作。

  另一个关键在于，代码能映射出我已熟悉的事物。
  如果我在开发一个航运系统，而代码中包含船舶、港口、集装箱等元素，
  并且这些元素的行为符合我在该领域的认知预期，这就有助于我理解代码的运行逻辑。

- Rebecca

  管理认知负荷还需要在不同粒度级别上理解领域。
  这种分解让我们能够通过抽象来梳理属性，而不至于总是陷入细枝末节。
  和设计过程的其他方面一样，找到合适的抽象级别是一个迭代的过程。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  从本质上讲，编程行为就是将 “真实” 领域（目标）映射到计算模型（实现）。
</div></br>

- Unmesh

  即便你改用英文提示词或图表，认知负荷也未必会降低。
  从本质上讲，编程行为就是将 “真实” 领域（目标）映射到计算模型（实现）。
  我们必须在将领域逻辑转换为机器可执行形式的同时，保留其核心精髓。
  <ins>至关重要的是，这并非一个单向过程；而是一个持续的反馈闭环，“目标” 与 “实现” 能够彼此深度启发</ins>。

  <ins>二者结合，能够揭示系统中稳定不变的部分，以及未来可能发生变更的维度，从而让我们借助编程范式为这些变更预留扩展切入点</ins>。

- Martin

  我经常听到人们用 “目标” 和 “实现” 来区分需求分析与编程。
  需求讲的是 “目标”，编程讲的是 “实现”。
  <ins>但我一直不认同这种划分方式，它试图把 “目标” 和 “实现” 割裂成两个完全独立的部分，而我认为这种看待问题的方式并不高效</ins>。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  “目标” 与 “实现” 并非相互割裂，而是交织在一起的。
</div></br>

- Unmesh

  我同意。表面上看，这种区分似乎很清晰。
  我们从白板草图和高阶用户旅程开始，讨论系统目标，再将其映射到实现层面 —— 数据库、服务和用户界面。
  但 “目标” 的问题在每个层级都会持续存在：

  - 系统层级：用户想要达成什么目标？
  - 类层级：这个组件应该承担什么职责？
  - 函数层级：这段具体代码的作用是什么？

  其影响在于：对 “目标” 的回答决定了逻辑分组，更重要的是决定了命名。
  一旦意图被明确命名（并通过我们的实现经验不断完善），我们才会着手处理执行层面的具体技术细节。
  我们在此做出的决策，将塑造整个解决方案的结构。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  “实现” 的关键一环，是选择如何将真实领域抽象为计算机可执行的形式。
</div></br>

- Rebecca

  “实现” 的关键一环，是选择如何将真实领域表示为计算机可执行的形式。
  我们通常会把领域映射到熟悉的计算模型上来实现这一点 —— 比如状态机、数据表、流或日志。
  这种映射并非中立的：以状态机为例，将某个事物映射为状态，与将其映射为状态转移，我们的思考方式会截然不同。

  这种映射的结构，往往正是解决方案整体架构开始成型的地方。
  如果我们不使用具有内置语义的模型，就必须构建一套心智模型（抽象），再用代码将其表达出来。
  此时，编程语言范式的选择就至关重要，因为某些抽象在特定范式下可以轻松表达，而另一些则会困难得多。
  例如，无界但有限的列表，在函数式编程语言中就更容易实现。

- Martin

  这就是为什么我们在做领域建模时，不仅要对数据结构建模，还要对基于这些数据结构的计算逻辑进行建模。
  面向对象建模通过将行为与数据结构绑定，朝这个方向迈出了一步，但这仅仅是个开始。
  在处理更复杂的领域时，必须把计算逻辑设计进模型的底层结构之中。

- Unmesh

  <ins>这些决策并非为了实现某项需求使其运行，而是在于选择合适的架构</ins>。

  在我们于这两种模式之间来回切换时，会产生重要的变化：

  - **目标 → 实现**：在梳理明确业务意图的过程中，实现机制会自然显现。
  “原来这些请求是严格按照交付时间排序的，这里我可以使用优先队列。”

  - **实现 → 目标**：在实现具体逻辑的过程中，系统的本质会显露出来。
  “从我们持久化这些事件的方式来看，我们不只是在保存数据，实际上是在实现预写日志机制。”

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  从场景出发，并通过场景来构建连接 “实现” 与 “目标” 的抽象设计。
</div></br>

- Martin

  这就引出了一个问题：我们应如何更好地理解自身是如何认知和思考更高层面的 “目标” 的。
  我们无法给出宽泛、抽象的需求描述，因为如果不理解实现它的机制，我们就不知道该如何去表述这种抽象概念。
  “目标” 与 “实现” 的相互交织，使我们陷入了循环依赖之中。

  <ins>在我看来，行之有效的解决思路是：从人们使用系统的具体实例出发。
  以这些场景/用例为依据，进而提炼出支撑它们的抽象设计</ins>。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  TDD 是一种设计策略，它让 “目标” 与 “实现” 之间的反馈闭环得以落地执行。
</div></br>

- Unmesh

  而这些场景体现在两个层面：
  - 在系统层面（用例）：我们定义业务流程。“用户将商品加入购物车” 这一描述界定了系统边界。
  - 在模块层面（测试用例）：我们定义契约约束。单元测试本质就是一个微型场景（Given→When→Then），为单个函数明确 “目标”。

  我初次接触 [TDD](https://martinfowler.com/bliki/TestDrivenDevelopment.html) 时，就被它吸引，因为它看上去正是一种能让 “目标/实现” 反馈闭环真正运转起来的设计策略。

  TDD 之所以有效，是因为它让这个 “目标/实现” 循环变得清晰明确：

  - 锁定 “目标”：先编写测试，会迫使你在被实现细节干扰之前，先回答 “目标” 这一问题（命名、输入、输出）。
  你会暂时以自身代码的调用方视角思考。
  - 迭代 “实现”：测试编写完成（红灯）后，便可自由实现解决方案（变绿）。
  - 优化结构：随后进行重构。
  在这一阶段，“实现” 往往会暴露出你原先定义的 “目标”（接口设计或测试场景）不够合理或存在泄露，从而推动你做出更优设计。

- Rebecca  
  
  TDD 与用例之间的这种关联十分重要，因为二者都是聚焦于 “目标” 的方式。
  测试本应聚焦于这种 “目标”，实际却常常并非如此。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  编写测试能促使我们思考接口设计，而不将其与实现耦合在一起。
</div></br>

- Martin

  我经常发现，人们很难做好封装，原因在于他们很难在思考接口时不把它和实现绑定。
  测试对此很有帮助，因为在实现功能之前先编写测试，会促使我只关注 API 本身。
  这同时也是一次即时的可用性测试。
  编写测试时，我是以 API 使用者的角度去看待问题的，这种聚焦于底层 “目标” 的视角，塑造了我对 API 的思考方式。

  而到了重构阶段，我会结合让测试运行通过的经验，重新审视实现与接口。
  这有时会让我产生新的思路，去更好地表达 “目标” —— 既针对当前的 API，也用于完善我对整个模型更宏观的理解。
  这种情况并非每次测试都会出现，甚至不会出现在大多数测试中，但它发生的频率已足以带来显著的设计改进。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  仅靠提示词可以满足单个场景，却无法构建适配未来场景的解决方案架构
</div></br>

- Unmesh

  即使在使用 LLM 并编写提示词时，这一切依然至关重要。
  我最近参考 MinIO 代码库，尝试编写了一个微型对象存储。
  当我让 LLM 直接推导 MinIO 的实现逻辑时，它生成的代码过于过程化，也更难理解。
  而我自己一步步编写时，最终得到的抽象更少、更清晰，代码也更易读、更易于迭代演进。

  这与我反复观察到的规律一致。
  如果没有一套稳定的抽象术语体系，LLM 生成的代码往往偏向过程式。
  如果我让它 “重构”，它又常常走向另一个极端，创建过多的类和层级，让设计变得不必要地复杂。

  这也是我更倾向于把 LLM 当作我 “目标/实现” 循环中的翻译层来使用的原因。
  我用它快速草拟第一版代码，但依然依靠亲手编写和重构来塑造架构 —— 因为最终保留的，一定是我能够清晰解释、可靠测试并放心修改的代码。

  这同样也是为什么我认为，用 LLM 生成测试用例来提升 “测试覆盖率” 并没有太大价值。

  让测试通过，或是让代码在某个场景下正常运行，仅仅是最低要求。
  核心目标不只是满足当前场景，而是夯实并构建解决方案的架构，使其能够适配未来的各类场景。

  如果只追求 “让代码跑起来”，就会写出脆弱的代码。
  我们必须对方案进行合理组织，让 “实现” 能够持续演进，同时不破坏 “目标”。
  我们可以通过以下两点做到：
  - 内聚性：将承载相同 “目标” （业务意图）的部分归为一组。
  - 解耦：将变更原因不同的部分相互分离。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  我们将相似的行为提取为模块，构建出领域专属的抽象，以此帮助我们管理认知负荷。
</div></br>

- Martin

  这关乎认知负荷管理。
  我们可以把一个复杂程序写成一个超长函数，通过简单的 if/else 条件判断处理所有分支逻辑，但这对理解者来说负担过重。
  如果我们把重复代码片段抽成子函数，就能把行为上的共性明确体现在代码中，让程序更易理解。
  如果两段代码几乎相同，我们可以将其提取为函数，并通过参数体现差异 —— 同样能清晰地标示出变化发生的位置。
  随后，我们可以将这些子函数组织成更宏观的模式，进一步明确变化点所在。
  这些模式最终形成专属于我们特定领域的抽象。

- Rebecca

  完成 Martin 所描述的重构之后，我们需要确定如何通过这些不同模块的交互来实现预期行为。
  同样，把控好合适的粒度，既有助于对行为进行逻辑推演，也能帮助我们管理认知负荷。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  编程范式为如何组织这些抽象提供了规范 (conventions)。
</div></br>

- Unmesh

  编程范式为这类决策提供了规范，在有效表示稳定部分的同时，为变化预留扩展点。

  面向对象编程侧重于识别通用行为（接口）。
  变化体现为具体的类，这些类将数据与对应接口的实现逻辑封装在一起。
  当我们预期行为（接口）保持稳定、而类型会发生变化时，会采用这种范式。

  函数式编程通常将数据结构视为稳定部分，而将变化体现在操作逻辑（pipelines / filters）中。
  其前提是数据形态保持稳定，而转换逻辑可以变化。

- Martin

  没错……只是我认为，把面向对象和函数式当成两种割裂的范式并没有什么帮助。
  在我看来，对象（绑定数据与行为）、一等函数、多态、流水线等，都只是工具 (tools)。
  一个理想的编程环境，应该让我能够按需使用其中任何一种工具。
  函数流水线 (pipelines) 常被看作是函数式编程的特征，但我最早是在使用 Smalltalk 时接触到它的。

  关键在于，这些东西全都是用来构建抽象的工具。
  人们在谈论从汇编语言到第一代高级语言这一极其重要的跨越时，通常会说它提升了抽象层级。
  这话没错，但更重要的是，高级语言首次为我们提供了创建自定义抽象的工具。
  早期的工具往往比较简陋（比如 Fortran IV 中的子程序），但也有一些能力更强（例如 Lisp）。
  从那以后，我们一直在不断发现并完善更多这类工具。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  随着抽象趋于稳定，编程就变成了用成熟可靠的抽象来表达设计意图。
</div></br>

- Unmesh

  一旦我们构建的抽象趋于稳定，就可以用声明式编程的方式基于这些抽象进行开发。
  在一些已经形成标准抽象体系的领域，声明式语言尤其适用 —— 比如用于数据库查询的 SQL，或是用于构建编译的 makefile。
  <ins>随着抽象不断趋于稳定，编程会越来越像是运用成熟的抽象来清晰表达设计意图</ins>。

- Martin 

  我把这些看作不同的计算模型，这也是 [我和 Rebecca 在我们那本 DSL 书中](https://martinfowler.com/books/dsl.html) 探讨的核心内容。
  这个领域我仍希望做更深入的探索，而且我很好奇，使用 LLM 是否能在这方面为我们提供助力。

  > 顺带一提，我并不是 SQL 的忠实爱好者。
  我更倾向于通过函数流水线来查询表格数据，比如 [R 语言中的 dplyr](https://dplyr.tidyverse.org/) 包。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  社区为我们提供了一套用于组织抽象的人类词汇 (human vocabulary) 。
</div></br>

- Unmesh

  构建一套精准度堪比替代计算模型的体系难度极高，因此也极为罕见。
  但开发者社区会围绕共同的实现关注点，形成一套包含惯用写法与设计模式的通俗表达体系。
  编程终究是一项人类活动，而围绕编程语言与编程范式形成社区也是自然而然的事。
  正如 Bjarne Stroustrup 所言：“设计与开发是人类的活动，忘记这一点，一切皆为空谈。”

  我们能看到，当人们使用语言解决现实问题时，这类社区会自然形成。
  使用 Ruby 与 Ruby on Rails 进行 Web 开发、Python 用于数据科学与基础设施开发、
  Java 及基于 Java 的工具应用于数据工程领域、Go 用于系统开发，都是典型案例。
  这些惯用写法与设计模式不仅能降低实现决策的难度，还为沟通设计中的 “实现” 提供了统一词汇。

  类似 “用列表监听模式 (List-and-Watch) 使用 etcd” 或
  “在 Go 中通过 channel 结合 select 循环实现消息传递”
  这类表述，能让人轻松理解如何做出健壮的实现。
  这些惯用方式也让设计意图与实现逻辑之间的转换更加顺畅。

  基于这种认知去使用 LLM，效果会非常出色。
  它们可以充当翻译层，让我们能够用自然语言表达设计意图。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  由于 LLM 可以在较低的精度要求下工作，它们让我们能够更灵活地探索抽象概念。
</div></br>

- Martin

  这对 LLM 而言是一个机遇。
  构建抽象需要精确性，这也是其难度所在。
  而 LLM 能够在模式与惯用用法这类较低精度的层面运行，这使得我们可以借助它们更流畅地探索思路。
  我很期待这是否能帮助我们更轻松地达到替代计算模型所需的精准度。

- Rebecca

  表示方式的选择由领域本身及其与编程范式的映射关系决定。
  一旦我们建立了这种映射 ——即便它仍是非正式的—— 就可以借助 TDD 与 LLM，推导出适配该范式的实现方式。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  优秀的抽象能让设计意图轻松转化为实现。
</div></br>

- Unmesh

  当我们在 “目标” 与 “实现” 之间迭代时，只需表达出设计意图或描述场景，LLM 就能快速生成代码。
  需要牢记的是，生成的代码通常只是能让测试通过的初始版本。
  我们仍需借助编程范式和语言提供的工具，以及社区惯用的写法，来梳理解决方案的结构。

  当抽象趋于稳定，编程工作主要变为用声明式语法使用这些抽象时，LLM 在语言表达上的优势就显得尤为突出。
  几条提示词即可生成可运行的实现代码。
  由于抽象构成了提示词的核心词汇，生成的程序会更可预测，也更易于审查。

  像 “按交付时间戳在优先队列中对请求排序” 或
  “通过 repository 模式存储这条客户记录” 这类指令，往往能生成符合通用惯例的代码，更便于审核和维护。

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  LLM 依赖成熟的训练数据 —— 未来的新技术新范式将由谁来开创？
</div></br>

- Rebecca

  然而，这里存在一个隐性依赖：LLM 依赖成熟度。
  如果一门语言不够成熟，或者我们在探索一种全新的范式，LLM 就会因训练不足而无法胜任。
  在这些前沿领域，工具十分匮乏，我们实际上只能依靠自己。
  <ins>这一限制引出了一个关键问题：
  既然 LLM 只能在训练数据充足的领域发挥作用，那么在人工智能时代，新的编程语言与编程范式将从何而来？</ins>
  如果它们永远不再出现，我们还会在意吗？

<div style="background-color: grey; padding: 8px; border: 1px solid lightgrey;">
  LLM 可以提出随机、新颖的想法，再由人类筛选并加以完善
</div></br>

- Martin

  这让我想到 [作家 Will Wilkinson](https://bsky.app/profile/wlknsn.xyz/post/3m5tnpfqpgs2e) 曾说过的一句话：“最顶尖的写作会 ‘让熟悉的事物变得陌生’ —— 赋予它新鲜感，重新唤起它的魅力，诸如此类。但 LLM 的输出，从设计初衷来讲，只是对熟悉内容的平均化复述。”
  当我们面对一个常见的、大家都很熟悉的难题时，一个优秀的解决方案必然是跳出常规、让人感到陌生的。

  不过另一方面，我确实认为，无论是在写作还是编程领域，LLM 都能产出不落俗套的想法。
  究其本质，这源于它们生成内容的随机性，会 “幻想” 出一些非常规的事物。
  但从另一个角度来说，它们无法将这些新创意深化完善，因为缺少必要的训练数据支撑。
  人类可以借助 LLM “幻想” 出一些怪异却具备潜力的点子，然后在此基础上继续构建。
  这也是为什么让 LLM 生成三种不同方案往往很实用 —— 人类可以从中选出最优的一个，或是融合三者的亮点，打造出更好的成果。

- Unmesh

  这就是为什么在我们逐步深化认知、构建稳定架构时，不能简单地通过 “提示词” 让 LLM 代劳。
  我们当然会利用它们，但必须亲自通过迭代驱动代码，打造出稳定的结构。
  参考《领域专用语言》一书中的观点，这正是我们构建语义模型 (Semantic Model) 的阶段。
  在此阶段，LLM 应被视为强大的工具，但开发者必须始终作为模型的核心设计者。

- Martin

  就我个人而言，这种想法让我感到欣慰。
  编程中我最享受的事情之一就是构建模型。
  我希望 LLM 不会夺走这份乐趣 —— 事实上，它理应让这件事变得更轻松、更有成就感。
