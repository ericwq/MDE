我们先从工具链开始。
每当进入一个模式和技术仍在不断演进的新领域时，我都会试着构建一套事物之间如何相互关联的思维模型。
这有助于应对扑面而来的海量信息。
这个领域正在解决哪些类型的问题？
解决这些问题通常需要哪些通用的基础组件？
这些事物又是如何组合在一起的？

## 工具如何分类
以下是我当前对利用大语言模型（LLM）辅助编码的工具所构建的思维模型中的各个维度。

### 可辅助完成的任务
- 更快地在上下文内查找信息
- 生成代码
- 对代码进行 “推理”（解释代码或代码中存在的问题）
- 将代码转换为其他形式（例如文档文本或图表）

这些是我所见到的编码辅助场景中最常见的任务类型，若将范围扩展到软件交付生命周期中的其他任务，还会包含更多类型。

### 交互模式
我见过三种主要的交互模式：

- 聊天界面
- 行内辅助，即在代码编辑器中边输入边提示
- 命令行界面（CLI）

### 提示词构建
提示词的质量显然对工具的实用性影响极大，同时也与后端所用大语言模型（LLM）的适配性息息相关。
不过，提示词构建并非只能完全由用户手动完成，许多工具会在后端为你应用各类提示词技术。

- 用户从零构建提示词
- 工具根据用户输入及额外上下文（如已打开的文件、一组可复用上下文片段，或向用户发起的补充提问）自动构建提示词

### 模型属性
- 模型的训练数据来源
  - 是否专门针对代码及编码任务进行训练？支持哪些编程语言？
  - 何时被训练，即模型所包含信息的时效性
- 模型规模（尽管关于这一点仍然存在很大争议，以及在特定任务如编码中什么是 “合适” 的规模）
- <ins>模型支持的上下文窗口 (context window) 大小，本质即模型可接收的提示词词元 (token) 数量</ins>
- 模型或其部署的后端系统中应用了哪些过滤机制

### 来源与部署方式
- 商业产品，其 LLM API 由产品公司托管
- 开源工具，可对接 LLM API 服务
- 自研工具，可对接 LLM API 服务
- 自研工具，对接经过微调、自行部署的 LLM API

## 人们目前在使用哪些方式？未来又将如何发展？
如今，人们最普遍的做法是，结合使用，直接聊天交互（例如通过 ChatGPT 或 Copilot Chat）与代码编辑器内的编码辅助（例如通过 GitHub Copilot 或 Tabnine）。
相较于其他方式，在编辑器上下文里提供 *行内辅助* ，大概是当前利用大语言模型进行编码辅助最成熟、最有效的方式。
它以微小的步骤在开发者自然的工作流程中提供支持。
步骤细碎更便于开发者跟进并更认真地检查生成内容的质量，而在效果不佳时也可以轻松忽略、继续编写。

开源社区正在进行大量实验，研发能够通过提示词组合 (prompt composition) 生成较大规模代码的工具（例如 GPT Engineer、Aider）。
我也见过不少团队针对自身特定场景，定制使用这类小型提示词组合应用：
比如将可复用的架构与技术栈定义，结合用户故事来生成任务计划或测试代码，这与我的同事 Xu Hao 在 [此处](https://martinfowler.com/articles/2023-chatgpt-xu-hao.html) 描述的内容类似。
目前这类提示词组合应用最常搭配 OpenAI 模型使用，因为这些模型最易获取且能力相对强劲。
不过，随着人们希望对数据拥有更多掌控权，相关实验正越来越多地转向开源模型以及大型云服务商托管的模型。

作为更进一步的发展方向，除了更高级的提示词组合技术外，人们对未来的改进寄予厚望的是模型本身。
更大的模型，或是体积更小但针对编码专项训练的模型，究竟哪种更适合编码辅助？
拥有更大上下文窗口的模型，能否让我们输入更多代码，从而对代码库中更大规模模块的质量和架构进行分析推理？
针对企业自身代码进行微调模型，达到何种规模才能真正产生价值？
开源模型领域又将出现哪些新变化？这些问题将在后续的备忘录中继续探讨。
